## TL;DR

- この記事は [SAGA 定理の記事](https://zenn.dev/)の続編です。前回は「局所は正しいのに大域で壊れる」現象に Lean 4 で証明された定理を与えるところまでを書きました。今回はその定理を**現実世界に持ち出した報告**です。
- 誰も AAT を知らずに書いた実在のマイクロサービス(研究用ベンチマーク [train-ticket](https://github.com/FudanSELab/train-ticket)、41 万行超)に、SAGA 定理の前提を満たす構造 — **3つの金額規約が実呼び出しの三角形上で衝突し、三者を同時に照合するコードがどこにも存在しない** — がそのまま見つかりました。
- その構造の上で、計測ツール ArchSig の SAGA 診断階段がフルに発火しました。各サービスは自分の規約を守っている(measured_zero)。それでも大域では貼り合わない(非零 H^1 類)。修理案を与えると貼り合いが回復し、ゲートが BLOCKED から PASS に変わる — 定理が予言した通りの順序で、実データが階段を一周しました。
- 発見物そのものは、払い戻し計算の**1セント未満の丸め剰余が、どのサービスにも記帳されていない**という、嵐のように地味な代物です。しかし価値は発見物ではなく、**航路が実在すると確認できたこと**にあります。だからこの成果を「SAGA の喜望峰」と呼ぶことにしました。

## 1. 定理の残るリスクは、偽であることではない

SAGA 定理は Lean 4 で証明済みです。`sorry` はなく、依存公理は Lean 標準のものだけ。つまり定理が**偽である**リスクは、形式化の時点で消えています。

では何のリスクが残っていたのか。**空虚であるリスク**です。

形式化された定理は、前提を満たす対象が数学の中にしか存在しない、という形で静かに死ぬことがあります。SAGA の前提の束 — 各チャートが局所的には自分の法を満たし、チャート間には貼り合わせデータがあり、それを埋める「面」が欠けていて、ループ上の食い違いのパリティが奇である — は、悪くすると人工的な例でしか組み上がらない構造かもしれなかった。前回の記事で紹介した one-cent デモは、この構造を**私たちが設計して**作った合成例です。「この構造なら H^1 が立つ」の実演にはなりますが、「この構造は現実に生じる」の証拠にはなりません。

後者は、実データでしか確かめられません。そこで、実ユーザーが使うのと同じ手順で、外部の OSS を丸ごと計測してみることにしました。ドッグフーディングです。

大航海時代の喜望峰の発見が意味したのは「インドに着いた」ではなく、「**この航路は実在する**」でした。今回の実験が確かめようとしたのも同じことです。AAT という航路は、数学の中だけの海なのか。それとも現実のコードという海につながっているのか。

## 2. 実験の設計 — 外部 OSS を正規フローで丸ごと計測する

対象は [FudanSELab/train-ticket](https://github.com/FudanSELab/train-ticket)。マイクロサービス研究の標準ベンチマークとして広く使われている、鉄道チケット予約システムです。40 以上の Java マイクロサービスからなり、注文・決済・払い戻し・託送などの業務フローを持ちます。研究用の題材として意図的に「現実にありがちな」実装スタイルの混在を含む、こちらの都合を一切知らないコードベースです。

計測は2段階で行いました。

**第1段(フルビルド)**: 全 42 サービス+共有モジュールの Java ソース約 28.5 kLOC から、ArchMap(観測データ)を正規フローで構築しました。2 系統の独立した読み取りパスで抽出し、差分を全件調停し、監査を通す — 実ユーザー向けに定めた手順そのままです。結果は 2,118 個の Atom、43 の context。読み取り作業はすべて軽量モデル(Claude Sonnet)のサブエージェントで行い、供給コストは実測で **約 4.1 分/kLOC** でした。「観測データの供給は軽量モデルで回る」ことも同時に確かめています。

このフルビルドの時点で、金額表現の規約(お金をどう持ち運ぶか)を観測する cover には、**24 サービスに 8 流儀**が共存していることが写っていました。double のフィールドで持つ、文字列のまま素通しする、使う場所で BigDecimal に直す、文字列連結で組み立てる — どれも単体では動くコードです。

**第2段(SAGA)**: このフルビルドの成果物の上で、law を SAGA 診断のフルスタックに拡張し、階段を一周させました。この記事の主役はこちらです。

## 3. 発見 — one-cent 構造は野生に実在した

フルビルドの実呼び出しグラフを調べていて、次の構造が見つかりました。

**cancel(払い戻し)、inside-payment(内部決済)、order(注文)の3サービスが、実際の呼び出し関係で三角形をなし、その3辺すべてで金額規約が食い違っている。**

3サービスの規約は、実ソースでこう確認できます(commit `313886e9`)。

まず order。注文の価格は文字列のまま保管されます。

```java
// ts-order-service: Order.java
private String price;
```

次に inside-payment。決済残高の計算では、その文字列を **BigDecimal に直して正確に**演算します。

```java
// ts-inside-payment-service: InsidePaymentServiceImpl.java
totalExpand = totalExpand.add(new BigDecimal(order.getPrice()));
```

そして cancel。払い戻し額の計算では、同じ文字列を **double に直して 0.8 を掛け、小数第2位に丸めて**文字列に戻します。

```java
// ts-cancel-service: CancelServiceImpl.calculateRefund
double totalPrice = Double.parseDouble(order.getPrice());
double price = totalPrice * 0.8;
DecimalFormat priceFormat = new java.text.DecimalFormat("0.00");
String str = priceFormat.format(price);
```

つまり同じ「注文の価格」という値が、**素通しの文字列 / 正確な十進演算 / 浮動小数点+丸め**という3つの別世界を一周します。0.8 × 価格が2桁で割り切れないとき(たとえば 91.33 × 0.8 = 73.064)、丸めで生じる 1 セント未満の剰余は、**どのサービスの帳簿にも現れません**。

そして決定的なのはここです。**3サービスの金額を同時に突き合わせるコードが、リポジトリのどこにも存在しない。** ペアごとの受け渡しはあります(cancel は order から価格を読む、cancel は inside-payment に払い戻しを渡す、inside-payment は order の価格を読む)。しかし三者を一度に照合する場所はない。

前回の記事の言葉で言えば: 3辺の食い違い(奇パリティの 1-コサイクル)があり、それを埋める面(triple overlap)が欠けている。これは SAGA 定理の前提の束、そのものです。私たちが仕込んだのではありません。普通の開発者たちの、それぞれ合理的な実装判断の積み重ねが、この形をしていたのです。

## 4. 計測 — 診断階段が一周する

この構造の上に law を組み、ArchSig の SAGA 診断階段を流しました。結果を先に表で示します。

| 幕 | 結果 |
| --- | --- |
| head の計測 | `MEASURED_NONGLUING_RESIDUAL_CLASS` |
| ├ grounding(各チャートは自分の法を守っているか) | `measured_zero` — **守っている** |
| ├ descent(残差類は零か) | `measured_nonzero` — **貼り合わない** |
| └ comparison(H^1 の輸送は成立するか) | `established` |
| ゲート判定 | `BLOCKED_BY_GATE_POLICY` |
| 修理案を与えた状態の計測 | `REPAIR_GLUES_WITHIN_SELECTED_COMPLEX` |
| 修理前後の比較 | `MEASURED_OBSTRUCTION_NO_LONGER_RECORDED_AFTER_CHANGE` |
| 修理後のゲート判定 | `PASS_WITHIN_GATE_POLICY` |

順に読みます。

**grounding = measured_zero。** これが SAGA の罠の部分です。cancel も inside-payment も order も、自分が掲げる金額規約には**完全に従っています**。lint も静的解析も、各サービスのレビューも、ここでは何も見つけません。局所はすべて正しい。

**descent = measured_nonzero。** それでも、三角形を一周した貼り合わせの残差類は零ではない。しかもこの類は、**どれか1つのサービスを直しても消えません**(1辺だけの修正は、別の辺に食い違いを移すだけです)。「どのファイルにもバグはないのに壊れている」の、計測としての姿です。

**ゲート = BLOCKED。** CI に置いたとき、この PR は止まります。「全モジュールのテストは緑なのに、システムは 1 セントずれ得る」状態を、緑の海の中から止められる — これが前回の記事で「lint の先にある解析のクラス」と呼んだものの実走です。

**修理 → 貼り合い回復 → PASS。** 「3サービスの共有読みを scale-2 の BigDecimal に統一し、丸め剰余を明示的に記帳する」という修理案を仮説状態として与えると、残差類は零になり、比較は「障害はもう記録されない」を返し、ゲートは PASS に変わります。修理**案**の段階で、実装前に「この直し方なら貼り合う」を検証できるわけです。

one-cent デモで設計した5幕の物語 — 沈黙 → 非零類 → BLOCKED → 修理 → PASS — が、合成例ではなく実在のコードベースで、同じ順序で再生されました。

## 5. 何が確かめられたのか

3つあると考えています。

**第一に、前提の実在証明。** SAGA 定理は「証明済みだが、実例は合成のみ」から「証明済みで、**実例が野生に在る**」に移りました。定理の前提空間に現実のコードが住んでいる。空虚であるリスクは、この1例で消えました。

**第二に、忠実性契約の耐久試験。** 理論と計測器のあいだには「計測器は定理が語れる範囲だけを語る」という契約があります。今回、計測器は初めて敵対的な実データに触れましたが、含意の連鎖(局所合法 → 残差類非零 → 修理で貼り合う)は Lean 側の連鎖と同じ順序・同じ条件で発火し、契約は一度も破れませんでした。ツールの拒否もすべて正当でした — たとえばドリフトのある三角形を「面がある」と偽って申告すると、コサイクル条件との矛盾で正しく弾かれます。

**第三に、定理の教訓の最小実例。** 「全員が正しくても全体は間違い得る。そしてそれはどの一箇所のせいでもない」という SAGA の核心が、grounding = zero と残差類 ≠ zero の**同時成立**という計測結果として観測されました。理論が予言する障害の最小実例が、ちゃんと最小の姿(1セント未満)で見つかったことは、理論の解像度が現実に合っている証拠だと読んでいます。

## 6. LLM レビューでは見つからないのか?

当然の疑問だと思います。「コード全体を読める LLM に3ファイル見せれば、この不整合くらい指摘できるのでは?」

正直に答えます。**この個別事例を「見つける」こと自体は、LLM にもおそらくできます。** それどころか、今回のパイプラインでも各サービスの金額規約を読み取ったのは LLM(Claude Sonnet)のサブエージェントでした。LLM に見えない、という主張ではないのです。差は見つけた後の**出力の形式**にあります。

- **非局所性の認証。** LLM は「この3つは不整合です」と言えます。しかし今回の判定の核心は「**どれか1つのサービスをどう直しても消えない**」(1辺の修正は食い違いを別の辺に移すだけ)であり、これは cocycle が coboundary でないという計算結果です。語りには、この保証が付きません。
- **良性と構造的の選別。** フルビルドでは witness 6辺中5辺で規約の食い違いが実測されましたが、**類は零**でした(局所修正で貼り合うドリフト)。LLM レビューなら8流儀の混在をすべて「問題」として列挙するでしょう。計測は「その5辺は良性、この三角形だけが本物」と選別します。指摘の洪水と構造的な一点の差は、レビュー疲れの時代には実用価値そのものです。
- **零の言明と再現性。** LLM は「問題ありませんでした」を保証付きで言えず、2回走らせれば違う指摘が返ります。計測は選ばれた law と cover に相対的な measured_zero を証明書付きで返し、同じ入力から常に同じ類を返す。CI で PR を止める根拠には、段落ではなく再現可能な判定が要ります。
- **修理の事前検証。** 提案はどちらもできますが、「この修理案なら貼り合う」を実装前に判定できるのは計測だけです。

そしてコストの構造が違います。全読み LLM レビューは、レビューのたびにコードベース全体をコンテキストに載せます — 10 万行を超えたあたりから、トークンコストは莫大になり、長大なコンテキストでの読み落としも増えていきます。毎回 O(コードベース) を払い続ける構造です。一方 ArchMap の構築は**一度きり**で、実測 4.1 分/kLOC、しかも軽量モデルで足りました。以後は変更差分だけを更新すればよく(この差分更新を支える「独立供給が同じ鍵に収束する」再現性の工学が、いままさに進行中の戦線です)、貼り合わせ障害の計算自体は Rust 側で走るのでほぼ無料です。O(コードベース) を一度だけ払い、あとは O(差分) — コードベースが大きく長命であるほど、この差は開いていきます。

つまり本当の関係は競合ではなく**分業**です。LLM が局所観測を担い(ファイル単位の事実の読み取り — LLM が得意な形)、数学が大域化を担う(観測を貼り合わせてループ上の障害を計算する — 長大コンテキストの attention に頼ると崩れる形)。「LLM 全部読みレビュー」の限界は読解力ではなく、大域的な主張に保証を付ける出力形式を持たないことです。ArchSig は LLM の観測に数学の背骨を通す装置だ、というのが正確な位置づけです。

grounding = measured_zero という今回の計測結果は、この問いへの実証でもあります。diff 単位・サービス単位のレビュー(人間でも LLM でも)が見る範囲では、**全部合法だった**。障害はレビューの単位の外側、重なりの上にだけ存在したのです。

## 7. 正直な境界 — この報告が主張しないこと

シリーズの慣例に従って、言えないことを言わないための節です。

- **SAGA が新しい障害を「検出」したのではありません。** 規約の食い違い自体は、階段の最下段(Čech 計測)が既に写していました。SAGA 階段が加えたのは検出ではなく、**診断の構造**です — 局所は合法という罠の明示、残差類の認証、修理案の事前検証、ゲートへの接続。また、残差の配置は観測に基づいて私たちが供給したモデルであり、計測器が無から発見したものではありません。
- **修理は仮説状態です。** PASS が示すのは「この修理案なら貼り合う」という事前検証の機構であって、実装済みの修正ではありません。
- **実害の規模は測っていません。** 「丸め剰余がどこにも記帳されない」は静的に確実ですが、実運用でどの頻度・どの金額で非零になるかは runtime の実測が必要です。だから金額の重み付き解析(harmonic-debt)は今回**沈黙**させています。数字がないのに数字を語らない、も規律のうちです。
- **train-ticket は研究用ベンチマークです。** 本番システムの欠陥告発ではなく、「現実にありがちな実装スタイルの混在」が定理の前提をどう実現するかの観測として読んでください。むしろ、ベンチマークとして意図された「ありがちさ」の中にこの構造があったことが、野生での遍在を示唆します。
- **供給の敷居はまだ高い。** 今回、SAGA の供給契約(残差類の認証に必要なデータの形)を満たすために、私たちは評価器のソースコードを読みました。これは実ユーザーに要求できる手順ではありません。ここが製品としての次のボトルネックです(後述)。

## 8. 岬の先の海

喜望峰に到達した後、インドへの航路にはまだ 10 年かかりました。しかも最難関は岬の**先** — インド洋の横断には、沿岸航法とは別の技術が必要だったからです。

この研究も同じ位置にいます。航路が実在することは確認できた。ここから先の海は、理論ではなく**供給の UX** です。観測データと law 一式を、代数幾何を知らないエンジニアが組めるようにする抽象化(ArchMap 供給には既にあり、SAGA 供給にはまだない)。独立に供給した観測が同じ鍵に収束する再現性の工学。そして2つ目、3つ目の外部 OSS。

岬に旗は立ちました。定理は現実のコードに実在した。「局所は正しいのに大域で壊れる」は、実在の OSS で、計測として、確かに観測される — 前回の記事の最後に書いた「公理から生えた数学」は、現実の海につながっていました。
