# ArchView によるコード理解支援 構想ノート(AI時代仮説)

このノートは、ArchView の次段の方向性として立てた仮説
「AI 駆動開発の時代に、Atom を可視化することで人のコード理解を支えられる」を、
問い・仮説の強度評価・検証計画・反証条件込みで固定する構想メモである。
確定した PRD でも実装計画でもなく、ドッグフーディング実験の設計図として機能させる。

## 問い

**AI 時代において、人のコード理解を ArchView で支えられるか。**

この問いを採否の判定規律として使う。「支えられる」と判定するのは、
ドッグフーディング実験で次の3条件が観測されたときに限る。

1. **理解到達**: 初見のコードベースについてレビュワーが立てる問いのうち、
   意味のある割合が ArchView(+ArchMap)だけで答えられる。
2. **供給成立**: その ArchMap の抽出・レビューのコストが、
   コード本体を読むコストを下回る。
3. **欠損の有限性**: viewer で答えられなかった問いが、列挙可能な機能欠損
   (探索・投影の不足)として記録でき、構造的不可能(語彙で表現できない)に落ちない。

3条件のどれかが崩れた場合、崩れた条件がそのまま次の作業対象を指す
(条件2が崩れれば抽出 tooling、条件3が崩れれば vocabulary / contract 側の課題)。
「ArchView の機能を増やす」ことは、この問いに対する答えではなく手段である。

## 1. 仮説の分解と強度評価

出発点の仮説は次の4命題からなる。強度は均一ではない。

| # | 命題 | 評価 |
| --- | --- | --- |
| 1 | AI 駆動開発の進展により、人間レビュワーがコードを読むのが追いつかなくなっている | 真。外部の状況証拠も豊富で、時流として正しい |
| 2 | コードを読まなくても、可視化できれば構造を把握できるはず | 半分だけ真。本ノートの最弱命題(§1.1) |
| 3 | ArchView は Atom を読めるので、Atom の可視化でコードベース理解を支援できる | 真だが、正確には「ArchMap を読める」。Atom の供給が前提(§2) |
| 4 | Atom を元にしたコードベース理解・探索機能が重要 | 同意。ただし価値の源泉は Atom vocabulary の質と抽出コストにある |

### 1.1 命題2の立て直し: 可視化一般ではなく「選ばれた語彙」

コード可視化ツールには長い失敗史がある。依存グラフ、UML ブラウザ、
コールグラフ探索の類は繰り返し作られ、繰り返し使われなくなった。
共通の死因は「全部見せると人間はやはり溺れる」ことにある。
構造把握に効くのは可視化一般ではなく**選択と抽象化**である。

AAT はまさにここに答えを持っている。

- **Atom vocabulary と evidence contract が「何を観測するか」の選択**である。
  viewer が描く対象は、最初から選択済みの有限 artifact になっている。
- **law / lawful locus / obstruction が意味論的なランドマーク**になる。
  構文構造(import・call)しか描けない先行ツールと違い、semantic atom は
  責務・不変条件・補償・副作用規律を持ち込める。
- 貼り合わせの失敗(H^1 系の読み)は「どこで局所理解が大域に繋がらないか」を
  指す点であり、これはレビュワーが最も知りたい種類の情報である。

したがって命題2は「3D で描くから分かる」ではなく
「**選ばれた語彙で描くから分かる**」と立て直す。
この立て直しが実験で支持されるかどうかが、本仮説の中心的な検証対象である。

## 2. 律速は可視化ではなく供給

「コードを読まなくても構造を把握できる」を届けるには、
読まなくて済む ArchMap が**既に存在する**必要がある。
抽出が手作業なら、地図を作るためにコードを読むことになり仮説は自壊する。

AI 抽出に置き換えると、「コードのレビュー」が「ArchMap のレビュー」に置き換わる。
これは一見すり替えだが、実はここに本仮説の最も強い製品ストーリーがある。

**AI がコードと一緒に ArchMap を保守し、人間は地図をレビューする。**

- ArchMap = AI 時代のレビュー可能な中間表現(選ばれた語彙内の architecture evidence)
- ArchSig = 地図上の law 適合と obstruction の機械検査
- compare = 地図の時系列ドリフトの追跡
- ArchView = 人間向けの理解面

ツールチェーン全体が「AI 駆動開発のレビュースタック」として一つの物語になる。
本仮説は ArchView 単体の機能追加ではなく、この水準で立てる。

ただしこのストーリーの成立条件は供給側にある。実験では抽出コスト
(kLOC あたりの時間、AI 抽出物のレビュー時間)を必ず計測し、
条件2(供給成立)の判定材料にする。

## 3. 責務範囲の規律

ArchView が見せるのは「コードベース」ではなく、
**選ばれた vocabulary と evidence contract の中で記録された architecture evidence** である。

- 観測されていない領域は viewer 上で沈黙として扱う。レビュー用途では
  これは弱点ではなく**機能**である。「どこが観測済みでどこが沈黙か」こそ、
  レビュワーが判断の重みづけに使う情報だからである。
  実験の測定項目にも「沈黙領域の可視性」を含める(§5)。
- 公開文面・記事では「読まずに全部分かる」型の無制限 claim をしない。
  語れるのは「この ArchMap contract の下で語れること」までである。
- viewer は packet / ArchMap の実測フィールドからの投影だけを描く
  (捏造ゼロ)。この規律は現行の gluing geometry contract をそのまま引き継ぐ。

## 4. 優先順位の改訂記録

「計測が主・可視化が従」(2026-06-15 のユーザー優先順位)は、
計測が AAT AG 版に対して未完備だった時点の順序であり、
v0.5.1(LawPolicy Stage 2)+ v0.5.2(SAGA 完全対応)の完了と是正消化
(2026-07-15 検証)により、その前提が解消された。

本ノートはこの優先順位の**意識的な改訂**である:
計測完備後の可視化は従属変数のままではなく、
「AI 時代のコード理解支援」という独自のプロダクト仮説を持つ。
ただし M→V gating(計測に無いものを描かない)は改訂しない。
改訂されるのは投資判断の順序であって、忠実性契約ではない。

## 5. 検証計画: ドッグフーディングを仮説実験として設計する

もともと次工程だった外部 OSS ドッグフーディングは、
「初見のコードベースを理解する」というまさに本仮説のシナリオである。
これを仮説実験として設計し直す。

手順の骨子:

1. 対象 OSS を選ぶ(初見であること。規模は viewer の既知の実用限界を意識して選定)。
2. **コードを読む前に**、レビュワーが立てるべき問いのリストを先に固定する
   (例: 責務の分割はどうなっているか、どこで局所整合が大域に繋がらないか、
   変更が波及する面はどこか、観測されていない領域はどこか)。
3. AI 支援で ArchMap を抽出する。抽出時間と、抽出物のレビュー時間を記録する。
4. ArchView(+ArchSig packet)だけで問いリストに答える。
   コードに戻らされた場合は、その理由を1件ずつ記録する。
5. 答え合わせとしてコードを読み、viewer 由来の理解と実態の一致度を確認する。

測定項目:

| 項目 | 対応する判定条件 |
| --- | --- |
| viewer だけで答えられた問いの割合と、答えの正確さ | 条件1(理解到達) |
| ArchMap 抽出コスト(時間/kLOC)と抽出物レビュー時間 | 条件2(供給成立) |
| コードに戻らされた事由の分類(探索機能欠損 / 投影欠損 / 語彙欠損) | 条件3(欠損の有限性) |
| 沈黙領域(未観測)と観測済み領域の区別が viewer 上で即座につくか | §3 の規律 |

「コードに戻らされた事由」のログがそのまま探索機能のバックログになる。
旧 V-B1/V-B2(シーン群追加・v3 改名)の再定義は、この証拠で駆動する。
実験前に探索 UI を作り込むことはしない。

## 6. 既存計画との接続

- **V-C(SAGA ビュー)は従来どおり実験の前に先行する。**
  スコープは小さく(sagaDescent 区画投影+SAGA ビュー+golden fixture)、
  v0.5.2 の主成果を見せる面として記事にも要る。本仮説とは独立に正当化済み。
- Three.js vendoring(V-A 随伴の残件。archview.html が unpkg CDN 参照のまま)は、
  外部環境で使うドッグフーディングの前提整備として V-C と同波で拾う。
- 記事([ArchSig 記事計画]系列)は、実験の成果(SAGA 実戦+理解支援の実測)を
  載せられる形になり、本仮説が支持されれば「AI 時代のレビュースタック」という
  より強い主題を得る。
- 旧 V-B1/V-B2 は本ノートの実験結果を受けて再スコープする。
  それまでは着手しない。

## 7. 反証条件

次のいずれかが観測されたら、本仮説(または対応する部分)を棄却・縮小する。

1. **抽出自壊**: ArchMap を作るのにコードの精読が必要で、かつ AI 抽出物の
   レビューがコードレビューより高くつく。→ 理解支援ではなく抽出 tooling が
   先行課題になる(仮説は保留、順序が変わる)。
2. **語彙不足**: レビュワーの主要な問いの多くが Atom vocabulary で表現できない。
   → ArchView の課題ではなく AAT / evidence contract 側の拡張課題として別立てする。
3. **可視化無効**: ArchMap を直接(JSON / テキストとして)読む方が viewer より速い。
   → 「Atom ベースの理解」は生き残るが「3D 可視化が理解面」の部分を棄却し、
   別の閲覧形態を検討する。
4. **正確性不成立**: viewer 由来の理解が答え合わせで実態と食い違う。
   → 投影・contract の欠陥として個別に原因を特定する(最優先で是正)。

## 8. 未決事項

- **ペルソナの確定**: 初見レビュワー / 新規参画者 / アーキテクトのどれを
  一次ペルソナにするか。実験は「初見レビュワー」で行うが、製品判断は実験後。
- **規模限界**: 現行 layout の実用限界(数百 contexts 規模)を超える対象への
  対応(集約表示など)は、実験対象の規模選定でいったん回避し、証拠が出てから扱う。
- **ソース位置への接続**: 「viewer からコードへ降りる」導線(source anchor)を
  探索機能に含めるか。コードに戻らされた事由ログを見てから決める。
- **AI エージェント自身による消費**: 本ノートの問いは「人の理解」に限定した。
  AI が ArchMap を機械可読に消費する経路は ArchView の外(contract の価値)であり、
  別ノートの主題とする。
