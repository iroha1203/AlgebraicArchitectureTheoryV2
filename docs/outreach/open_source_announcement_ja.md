AAT / SFT の研究リポジトリをオープンソースにしました。

リポジトリはこちらです。

https://github.com/iroha1203/AlgebraicArchitectureTheoryV2

このリポジトリでは、ソフトウェアアーキテクチャを「設計図」や「ベストプラクティス集」としてだけでなく、変更・レビュー・CI・運用・AI agent の提案によって進化し続ける対象として扱うための理論とツールを作っています。

大きな目標は、ソフトウェアとコンピューターの進化を、あとから眺めるだけでなく、できるだけ測り、比べ、予測できる対象にすることです。

## 何を研究しているか

中心にあるのは、次の三つです。

```text
AAT makes architecture locally algebraic.
ArchSig makes architecture observable.
SFT makes software evolution computable.
```

**AAT**（Algebraic Architecture Theory / 代数的アーキテクチャ論）は、アーキテクチャを局所的な代数構造として扱います。

言い換えると、ある設計変更が「依存の向きは守れているか」「層の境界を越えていないか」「抽象に依存できているか」「障害が広がりやすくなっていないか」といった点を、できるだけはっきり扱おうとするものです。

**ArchSig** は、実際のコードベースや PR から、アーキテクチャ上のシグナルを取り出すためのツール群です。

単一の「良い / 悪い」スコアを出すのではなく、「循環依存がある」「境界を越えている」「抽象を飛び越えて具体実装に依存している」「まだ測れていない項目がある」というように、リスクの種類を分けて読むことを目指しています。

**SFT**（Software Field Theory / ソフトウェアの場の理論）は、ソフトウェアの進化を「変更の連なり」として扱い、それをできるだけ計算可能にするための理論です。

PRD、Issue、PR、review、CI、組織、AI agent は、いまの変更だけでなく、次に起こりやすい変更にも影響します。

たとえば、曖昧な PRD は場当たり的な実装を呼びやすく、境界がよく整理されたコードベースは次の変更もきれいに入りやすい。SFT では、そうした「この変更のあとにどんな未来が開きやすいか」を扱うための概念を育てています。

## なぜ作っているか

AI にコードを書いてもらうと、動くものはかなり速く出てきます。

一方で、速く出てきた差分が、将来の変更を簡単にするのか、逆に場当たり的な実装を増やすのかは、別の問題です。

知りたいのは、単に「テストが通るか」だけではありません。

- この変更で、守りたい設計上の性質は保たれているか。
- 保たれていないなら、どのコードや依存関係を見るとそれが分かるか。
- その問題は、循環依存なのか、境界違反なのか、抽象化の破れなのか。
- この PRD / Issue / PR は、次の変更を良い方向へ進めやすくするか。
- review や CI は、危険な変更の流れを早めに止められているか。

こうした問いを、できるだけ形式化・観測・ツール化できる形にしたい、というのがこのリポジトリの動機です。

## リポジトリに入っているもの

現在のリポジトリには、主に次のものがあります。

- Lean による AAT の形式化
- AAT / SFT の研究ノート
- ArchSig ツール群の Rust 実装
- 未証明の課題や Lean theorem index
- AAT / SFT / ArchSig を読むための静的 website

Lean 側では、証明できる範囲を小さく切り出しながら、少しずつ定理を積み上げています。

Rust のツール側では、コードベースや PR から設計上のシグナルを取り出し、レビューや CI で読めるレポートにする方向で開発しています。

website 側では、AAT、SFT、ArchSig を公開向けに読める研究ノートやマニュアルとして整備しています。

## まだ研究途中です

このリポジトリは、完成品というより研究と実装を同時に進める作業場です。

そのため、「数学的に証明できたこと」「まだ定義を置いただけのこと」「これから証明したいこと」「実際の開発データで確かめたいこと」を分けて管理しています。

たとえば、Lean で証明した定理があるからといって、そのまま現実のプロジェクトの未来を予測できるわけではありません。

ツールが「この依存関係が見えた」と言えること、Lean が「この前提ならこの性質が成り立つ」と証明できること、実際の開発データで「この傾向がありそうだ」と確かめたいことは、それぞれ別の種類の主張として扱います。
