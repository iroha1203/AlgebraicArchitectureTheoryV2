Architecture review for fast-moving code

Architecture analysis through LLM Skills.

When code changes fast, ask your LLM agent to run the ArchSig Skills. The agent can inspect what the change is doing to the design: which responsibilities are drifting, which boundaries are weakening, and where engineers should look next.

Three-step start

Ask your LLM agent the question you would ask a senior reviewer: what did this change do to the design, and where should we look first?

  1. Choose the scopeTell the LLM agent which repository, PR, branch, feature area, or architectural concern the Skills should inspect.
  2. Have the agent run the SkillsThe agent uses archmap-creater, law-policy-creater, and archsig-reader to turn selected source evidence into an architecture review surface.
  3. Act on the review queueUse the returned findings, repair candidates, code refs, and confirmation points to decide what to inspect or change next.

Three strengths

ArchSig is strongest where ordinary tools go quiet: the code compiles, tests pass, and the dependency graph has edges, but the design still feels like it is moving.

  • It reads design signals, not just edgesImports and calls show contact. ArchSig helps read whether that contact is crossing a boundary, moving responsibility, hiding a workflow assumption, or creating coupling that a graph alone does not explain.
  • It finds the pressure pointsArchSig can surface hotspots, repeated boundary stress, workflow risk, transfer paths, missing evidence, and architectural holes as one reviewable picture instead of scattered hints.
  • It turns diagnosis into review workThe LLM agent follows the Skill output back to source evidence, then returns comments, repair candidates, and confirmation checks that engineers can use in a PR or architecture review.

What you get back

The output is meant to be read by engineers, not decoded like a metrics dump. It tells you what looks structurally important, why it matters, and what evidence should be checked next.

  • Findings reviewers can inspectArchitecture-relevant issues with the code paths, evidence, and context reviewers should open first.
  • Architecture tendenciesWhere coupling concentrates, which boundaries keep getting crossed, which responsibilities keep moving, and what kind of drift is starting to appear.
  • Deeper structural readingsResponsibility pressure, semantic coupling, order sensitivity, path-like change patterns, repeated stress on the same support, and the evidence needed before repair decisions.
  • Repair candidatesConcrete moves such as restoring a port boundary, moving cache access behind an adapter, or making a workflow contract explicit.
  • Confirmation pointsChecks the team should perform before acting on a reading, including missing tests, docs, traces, ownership evidence, or policy decisions.
  • Traceable detailReferences back to the summary, packet detail, validation output, and coverage repair queue when someone needs to reproduce or challenge the reading.
