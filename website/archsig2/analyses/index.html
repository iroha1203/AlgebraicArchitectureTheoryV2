What ArchSig can inspect

Use ArchSig when local facts do not explain the design movement.

Linters, tests, and dependency graphs show important facts. ArchSig helps an LLM agent read what those facts mean for architecture: which boundaries are bending, which responsibilities are moving, and which evidence is still missing.

Analysis catalog

These are the questions ArchSig is designed to help an agent answer after it has source evidence and a selected LawPolicy.

  • Boundary movementWhich service, feature, adapter, provider, queue, cache, or ownership boundary is being crossed, weakened, or moved?
  • Responsibility driftWhich module is absorbing business rules, workflow assumptions, provider details, state decisions, or review pressure that used to belong elsewhere?
  • Semantic couplingWhere do files remain locally reasonable but share hidden assumptions about order, retries, rounding, lifecycle state, authorization, or contracts?
  • Feature extension pressureDoes a new feature extend the system through an intended interface, or does it reach into another feature's internals?
  • Operation order sensitivityDo two implementation paths preserve the same visible behavior, or does the order of operations change the architecture-relevant outcome?
  • Hotspots and recurrent pressureWhich supports keep attracting nonzero readings, repeated review focus, architectural holes, or missing evidence?
  • Evidence gapsWhich docs, tests, traces, policy decisions, ownership rules, or source refs are missing before the team should trust a reading?

AI-assisted code review

ArchSig is especially useful when generated code is locally correct but the design impact is unclear.

  • Working code with unclear architecture impactAsk the agent to inspect whether a generated change moved responsibility, bypassed a boundary, or introduced a shortcut that future changes may copy.
  • Many small PRs that feel harmlessUse the summary to see whether review pressure is accumulating around the same module, workflow, provider, or boundary.
  • Repair planning after a generated changeAsk for repair candidates and confirmation checks before turning a local cleanup into a broader refactor.

How it fits with other tools

ArchSig does not replace existing technical evidence. It gives an agent a way to connect that evidence to architecture review.

Tool familyWhat it mainly seesArchSig adds
LinterLocal rule violations.Architecture readings under selected laws.
Static analyzerSyntax, types, dataflow, and dependency facts.Source-backed architecture observations and witness readings.
Dependency graphEdges between modules.Boundary movement, semantic coupling, support, coverage, and path readings.
Metrics dashboardAggregate counts and trends.Evidence-backed review findings, repair candidates, and confirmation checks.
Generic AI reviewFree-form comments over code and diff context.ArchMap evidence, LawPolicy, validation, packet refs, and measurement families.

FieldSig handoff

ArchSig reads current architecture state. When the question becomes evolution, governance, operational feedback, or longitudinal monitoring, the bounded ArchSig packet can be handed to FieldSig instead of treating raw observations as forecast truth.
