Authoring the inputs

ArchSig measures what you declare. Here is how to declare it.

ArchSig does not scan your code by itself — it computes over declared, validated input artifacts. That is the biggest difference from a linter, and the biggest entry barrier. This page walks each input contract: who writes it, what it contains, and the validator that checks it before anything is measured.

The input surface

An analyze run takes ArchMap plus LawPolicy, law-equation-surface, and MeasurementProfile; the CI surface adds a gate policy and repair measurement adds a repair plan. Each has a single current schema version (…/v0.5.2) and a dedicated validator or consuming command.

ArchMap: the evidence file

The ArchMap records what is the case in your repository: components, relations, capabilities, declared section values, bounded contexts, restriction directions, and the selected cover — every entry grounded in source refs (path, symbol, line). It records observations only; evaluation vocabulary is a validation error (why).

Authoring is the archmap-creater Skill's job, and it is a real survey, not a one-shot generation:

  1. Scope manifest — you approve this stepThe agent runs archsig scope-manifest to build a deterministic worklist (repo-relative paths, content hashes, include/exclude globs) and presents exclusions for approval before reading any source.
  2. Two independent reading passesThe worklist is surveyed twice without sharing candidate atoms between passes. archsig extraction-diff then compares the passes by literal match keys.
  3. Human-rule adjudicationEvery candidate found by only one pass is re-read and explicitly adjudicated (adopted / merged / not-adopted). No similarity merging, no majority voting — the match rate is a record, not a pass/fail score.
  4. Coverage ledgerA row-per-source ledger records what was surveyed and what was not, with procedural reasons only. Unread files stay visible instead of silently disappearing.

What you supervise: the scope (is the agent reading the right code?), the exclusions (is anything important skipped?), and spot-checks of atoms against their cited sources. The ArchMap's claims are bounded to the recorded revision and approved scope — it never claims to have extracted everything extractable.

LawPolicy and MeasurementProfile: the selection

The LawPolicy selects what to measure: evaluator ids from the registry (see Analyses for the catalog), the laws they check, scope globs, severity, and basis refs into a basisLedger — the documents your team points to as "this is why this rule". The law surface declares the selected law universe and witness variables. The MeasurementProfile selects how: the cover, coefficient, predicates, verdict discipline, and finite bounds. A policy bundle fixes all three selection artifacts with canonical component fingerprints.

The division of labor is strict: the policy holds selection only, the law surface supplies the declared equations, and the profile supplies the finite measurement regime. Witness rules, distance profiles, operation costs, and repair recipes live in ArchSig's evaluator registry code — a policy that tries to embed calculation rules fails validation. The law-policy-creater Skill ships starter files for the three artifacts and bundle creation, and, when the evidence does not justify a selector, is instructed to ask you rather than pick a generic policy.

Gate policy: the CI mapping you own

The gate policy is the one input a human normally writes by hand, because it encodes an institutional decision: which measurement verdicts block a merge in this repository. An absolute rule must map all six verdict values explicitly to one of three actions — and the four non-terminal values cannot map to a plain pass, so silence can never slip through as approval.

An optional introduced-by-change rule activates only when a comparison report is supplied; without one it is recorded as not_applicable, never as pass. A conservative starter template ships in the repository at tools/archsig/tests/fixtures/ag_measurement/gate_policy_conservative.json.

Repair plan: the measured path out

When a gate blocks, the way back is not "delete the finding" — it is a repair plan: a declared assignment of repair variables over the finite complex, which ag.saga-descent then measures for whether it actually glues. The repair-plan-creater Skill authors these in complete-support mode, using only atoms, contexts, and overlap refs already present in the evidence.

The discipline mirrors the rest of the system: the plan is an input, so it may not contain conclusion tokens, may not claim faithfulness modes it cannot witness, and fails closed on unresolved refs or parity violations. If complete support cannot be read from the evidence, the Skill stops and reports the missing supply instead of weakening the plan.

Without --repair-plan, a policy that selects ag.saga-descent still runs — the evaluator just stays not_computed with a silence_by_design boundary. Silence, not error.

Validate everything first

Each input has a standalone validator subcommand, and analyze re-validates everything at the start of a run. Validation failures happen before measurement — the run writes validation reports and stops; it never emits a half-measured packet.

Two properties make the validators trustworthy inputs to CI. Acceptance is exact: only the current contract version <name>/v0.5.2 is accepted, and there is no compatibility window (see Reference). And rejection is loud: retired forms and unknown fields fail as ordinary validation errors instead of being silently coerced.
