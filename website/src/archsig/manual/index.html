Reading the output

Ten artifacts, one reading order.

An analyze run writes ten files into the output directory. You rarely need all of them: the summary tells you where to look, the insight report ranks what to inspect, and the packet holds the full evidence when you need to verify a claim. This page is the reading order, then the recipes.

One rule before reading anything: check that archsig-run-manifest.json exists and lists the artifact you are about to read. If input parsing fails before preflight, ArchSig leaves the output directory untouched — files from a previous run may still be sitting there, and the missing manifest is your signal not to read them.

Start: the summary and its readThisFirst

The summary opens with a block written for exactly this moment:

Around it, the summary carries the structural verdict counts (rowCount, measuredNonzeroCount, unmeasuredCount, nonTerminalCount), the input digests that pin exactly which evidence and policy produced this conclusion, the assumption summary, the translation-rule table that maps verdict rows to conclusion codes, and the non-conclusions — the standing list of things a summary never claims. Three reading habits:

  • Trust the conclusion's grammar…UNDER_PROFILE is not decoration: the claim is relative to the selected cover and coefficient. What that buys you is covered in Concepts.
  • Check the non-terminal count0 non-terminal rows means every selected evaluator actually measured. A nonzero count means some rows are unmeasured / unknown / not_computed — go see which, and why, before celebrating a zero.
  • Follow whereToLookFirstIt points into the insight report's top card, which carries the evidence refs for the headline finding.

The insight queue

archsig-insight-report.json turns the packet into ranked inspection cards: each card names a finding, its supporting evidence refs (context ids, atom ids, source paths), and what would confirm or discharge it. archsig-insight-brief.md is the same queue as short prose — the natural thing to paste into a PR comment. The queue ranks where to look; it does not add claims beyond the packet, and an agent following the archsig-reader Skill is required to open the cited source before turning a card into a recommendation.

The measurement packet

archsig-measurement-packet.json (archsig-measurement-packet/v0.5.0) is the artifact everything else summarizes, and the handoff FieldSig reads. Its load-bearing sections:

  • structuralVerdict[]One row per selected evaluator: the five-valued verdict, the certificate ref behind it, the law it checked, and dependsOnAssumptions — which assumption-ledger entries this row's validity rides on.
  • computedInvariants[]The concrete mathematical objects: ranks, class supports, spectra, forbidden supports, hitting sets. This is where a nonzero verdict names its evidence.
  • assumptions[] (the assumption ledger)Every premise the run leaned on, each with a status — checked (and by what) or assumed (and by which input). Theorem premises are discharged by finite checks where possible and honestly listed where not.
  • boundaryStatements[]The typed edge of the measurement: what was not observed, what stayed silent and why. Treat these as the exact limits of the claim, per Concepts.

Each support atom resolves back through the ArchMap's sources declarations to a path, symbol, and line — which is how ArchView's click-to-source landing works, and how a reviewer gets from "the loop doesn't close" to the three functions that disagree.

Gate and comparison reports

The CI surface adds two artifacts. The gate report records the decision and, rule by rule, which verdict rows mapped to which actions:

The comparison report grades comparability first (identical / verdict-row / not-comparable), then records row-level verdict transitions with refs into both runs. Its four conclusion codes are deliberately transport-free — MEASURED_OBSTRUCTION_NO_LONGER_RECORDED_AFTER_CHANGE says the record changed, not that your commit caused the fix. The reasoning behind that restraint is in Concepts.

Task recipes

CI gate

Run analyze on the head evidence, then gate against the packet. The exit code is the contract: 0 pass, 1 block, 2 invalid input. Store the run directory as a CI artifact so a blocked PR carries its own evidence.

PR review

The archsig-pr-reviewer Skill runs base and head measurement, compare, and gate, then reads the changed source before writing anything. What you should expect back: the gate decision with its blocking rules, the introduced / cleared / preexisting transitions, source evidence checked per finding, and concrete follow-up questions. What you should not accept: the gate decision presented as merge approval, or comparison output presented as proof the change caused anything.

Repair planning

After a block: read the packet's class support to see which contexts and atoms carry the mismatch, let repair-plan-creater author a plan over exactly that complex, validate it, and re-run analyze --repair-plan. REPAIR_GLUES_WITHIN_SELECTED_COMPLEX from ag.saga-descent is the measured version of "this fix actually closes the loop" — within the selected complex, which is precisely the scope the block was about. The full authoring contract is on the Contracts page.

Handing off to evolution analysis

FieldSig reads the measurement packet — not raw ArchMaps, not summaries — as its bounded input for software-evolution readings. If you are exploring that layer, start from SFT; the handoff contract stays out of this manual by design.
