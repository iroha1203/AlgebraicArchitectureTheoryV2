Commands

In a repository checkout, prefix each command with cargo run --manifest-path tools/archsig/Cargo.toml --; with the release bundle, call bin/archsig directly. archsig <command> --help prints the authoritative flag list.

The measurement workflow

CommandDoesKey flags
analyzeThe primary workflow: validates all inputs, normalizes the evidence, runs the selected evaluators, writes the eleven run artifacts.--archmap plus either --policy-bundle or the three component flags, --out-dir, optional --repair-plan
compareJuxtaposes two run directories at record level; emits archsig-comparison-report.json and archmap-diff.json.--base-run --head-run --out-dir
gateMaps a packet's verdicts to pass / pass_with_boundary / block under your policy. The CI decision surface.--packet --policy --out, optional --comparison

Input validators

CommandValidates
archmapAn archmap/v0.5.1 observation artifact; with --scope-manifest / --candidate-packets / --extraction-consistency / --coverage-ledger, also the authoring provenance (see Contracts).
law-policyA law-policy/v0.5.1 selector against its measurement profile.
law-surfaceA law-equation-surface/v0.5.1 declaration of the selected law universe and witness variables.
measurement-profileA standalone measurement-profile/v0.5.1 artifact, including finite bounds against registry hard caps.
policy-bundleCreates or validates an archsig-policy-bundle/v0.5.1 with canonical component fingerprints.
repair-planA standalone archsig-repair-plan/v0.5.1 against its ArchMap; fails closed on conclusion tokens, unresolved refs, parity or support inconsistencies.

Authoring support

CommandDoes
scope-manifestBuilds the deterministic authoring worklist (paths, hashes, approved globs) that ArchMap surveys start from.
extraction-diffCompares two survey passes' candidate packets by literal match keys; leaves adjudication to the integrator, never auto-adopts.
schema-catalogPrints the machine-readable catalog of every current artifact contract, with per-artifact boundaries and non-conclusions.

Exit codes

CodeMeaningCI should
0Success. For gate: no rule mapped to block.Proceed.
1The command ran and the answer is negative — gate blocked, or a validator found the artifact invalid.Fail the check and surface the report.
2Input error before measurement: unreadable file, JSON parse failure, schema mismatch.Fail as a configuration error; nothing was measured.
3Internal runtime error.Fail and report a bug.

The distinction between 1 and 2 is worth wiring into CI messages: exit 1 means "your architecture evidence produced a blocking verdict", exit 2 means "your setup is broken" — very different notifications.

Schema catalog

Every artifact carries a schema field with a single contract version. The current input contracts are archmap/v0.5.1, law-policy/v0.5.1, law-equation-surface/v0.5.1, measurement-profile/v0.5.1, archsig-policy-bundle/v0.5.1, archsig-gate-policy/v0.5.1, and archsig-repair-plan/v0.5.1; the output contracts are archsig-measurement-packet/v0.5.1, archsig-comparison-report/v0.5.1, archsig-gate-report/v0.5.1, archsig-atom-viewer-data/v0.5.1, archsig-run-manifest/v0.5.1, normalized-archmap/v0.5.1, and archmap-diff/v0.5.1.

Compatibility policy: there is none, on purpose

ArchSig is pre-1.0 and applies a deliberate no-compatibility rule. Validators accept exactly the current contract version; older or newer forms fail as ordinary validation errors (exit 1 / 2). There is no migration command, no acceptance window, no deprecated-but-tolerated fields. When the contract version moves, inputs are re-authored — for Skill-authored artifacts that is a re-run, not a hand-migration.

What this buys: every artifact in circulation means exactly one thing, and a reader never has to ask which era's semantics a field carries. Check the top of any artifact's schema string against the catalog and you know everything about its form.

Troubleshooting

  • Exit 2 with a schema messageYour artifact declares a retired or unknown schema version. There is no compatibility path — re-author against the current contract (see above).
  • The output directory has files but no fresh manifestIf input open / parse / decode fails before preflight, analyze leaves the out-dir untouched — what you see is a previous run. Never read run artifacts until you have confirmed this run's archsig-run-manifest.json. When a run does start, ArchSig first deletes its known generated artifacts from the out-dir, so a reused directory cannot serve stale results as current ones.
  • ag.saga-descent shows not_computedNot a failure: the policy selects it but no --repair-plan was supplied. The row carries a silence_by_design boundary. Supply a validated plan to compute it (Contracts).
  • compare says the runs are not comparableThe site changed between runs — contexts or covers differ. This is boundary data, not an error and not a degradation claim; the report still carries both runs' independent conclusions, and gate policies handle the transition through the other mapping key.
  • gate reports NOT_EVALUABLE for a ruleThe rule could not be applied to this packet (for example an introduced-by-change rule without a comparison). It is neither pass nor block; the decision comes from the rules that did evaluate.
  • An evaluator you selected is silentMost evaluators have explicit evidence preconditions — no restriction edges, no triple-overlap skeleton, no core/feature/boundary classification, no witness atom. The catalog lists each one's silence condition; the row's boundary statement names the missing supply.
  • ArchView opens as an empty shellIt fetches archsig-atom-viewer-data.json from its own directory — serve the run directory itself, or drag the file in. And it loads Three.js from a CDN, so it needs network access (details).
