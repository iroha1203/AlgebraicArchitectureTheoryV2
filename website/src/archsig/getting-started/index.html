Ask the agent to run the Skills

You do not need to begin with commands or schema names. Ask your LLM agent to use the ArchSig Skills, then state the architecture worry you want reviewed.

Three Skills

The ArchSig workflow is built around three Skills.

  • archmap-createrCreates the source-backed ArchMap v1: sources, atoms, and explicit molecule membership.
  • law-policy-createrCreates the LawPolicy v1 selector: policy pack or evaluator ids, basis refs, scope, and severity. If repository evidence does not justify a selector, the Skill should ask instead of silently choosing a generic policy.
  • archsig-readerRuns ArchSig, reads the summary and packet detail, checks the relevant source again, and returns review findings, repair candidates, and confirmation points.

How ArchSig works

Behind the Skill workflow, ArchSig has a simple shape: feed ArchMap v1 and LawPolicy v1 into the CLI, then read typed evaluator results, the analysis packet, and the compact summary.

  1. ArchMap records atomsAn ArchMap v1 is source-backed input: read sources, primitive atoms, and explicit molecule membership. It is evidence, not a verdict.
  2. LawPolicy selects evaluatorsA LawPolicy v1 says which policy pack or evaluator ids should run, with basis refs, scope, and severity. The evaluator registry owns witness and distance rules.
  3. The CLI runs typed analysisThe agent passes ArchMap v1 and LawPolicy v1 to ArchSig analyze. ArchSig normalizes atoms, runs typed evaluators, and keeps blocked / unknown / unmeasured support out of measured zero.
  4. The summary becomes the review surfacearchsig-analysis-summary.json exposes verdict, typed evaluator status counts, distance insights, distance diagnosis, dominant findings, action queue, detail refs, and measurement basis.

Tell it what to inspect

A good request gives the agent enough context to avoid a generic review. Name the scope, the design concern, and the evidence that should or should not count.

  • ScopeRepository, PR, branch, subsystem, package, service, feature area, or release window.
  • GoalBoundary review, architecture drift check, feature extension review, repair planning, evidence audit, or current-state inspection.
  • Risk focusResponsibility movement, hidden coupling, workflow ordering, provider contracts, cache access, authorization, retries, state transitions, or generated-code review.
  • Available evidenceCode, tests, docs, traces, PR description, architecture notes, ownership rules, or known missing evidence.
  • Excluded readingsAreas out of scope, private systems, stale docs, generic laws, or claims the team does not want inferred.

What the Skills build

The LLM agent uses the Skills to turn the request into bounded analysis artifacts. You normally read the result, not every intermediate file.

  • ArchMapA source-backed map of atoms and explicit molecule membership.
  • LawPolicyThe selected evaluator policy for this review: policy pack or evaluator ids, basis refs, scope, and severity.
  • ArchSig outputTyped evaluator results, analysis packet, compact summary, viewer data, and detail references.
  • Source comparisonThe agent reopens the relevant code paths before turning packet readings into human-facing review work.

What the agent returns

The useful result is not a raw score. It is a review queue that says what changed structurally, why it matters, and what should happen next.

  • FindingsArchitecture issues or pressure points with code refs and the reason they matter for the selected review.
  • Repair candidatesPossible moves such as restoring a boundary, separating a workflow contract, adding an adapter, or collecting missing evidence first.
  • Confirmation checksTests, docs, traces, ownership rules, policy decisions, or source refs needed before the team trusts the reading.
  • Trend summaryWhere pressure is concentrating, what responsibilities are drifting, and which parts of the codebase keep attracting review attention.

Improve the request

Ask sharper questions when the first reading is too broad. ArchSig works best when the review goal is explicit.

  • Broad inspectionFind architecture pressure across a subsystem and rank what reviewers should inspect first.
  • PR-local inspectionCheck whether one PR moved responsibility, crossed a boundary, or introduced hidden coupling.
  • Boundary-focused inspectionAsk whether a feature reaches into another feature, service, adapter, provider, or cache boundary.
  • Path-focused inspectionAsk whether two implementation paths preserve the same visible behavior or whether operation order matters.

When to open the manual

Open the manual when you need to reproduce a run, inspect packet detail, debug coverage gaps, compare schema versions, or understand exactly which measurement family produced a finding.
