Path A: supervise an agent

If your LLM agent has the ArchSig Skills installed (they ship in the release bundle), you do not run commands at all. Ask for the analysis, approve the scope the agent proposes, and read the findings.

The agent will build the evidence file (archmap-creater), select a policy (law-policy-creater), run the measurement, and come back with findings tied to source locations (archsig-reader). What you should check before trusting the reading is covered in the Manual's task recipes. The rest of this page is Path B: the same measurement driven by hand, so you can verify what the agent does.

Prerequisites

  • Rust toolchaincargo on your PATH (any recent stable). The demo builds ArchSig and a 3,000-line sample service from source.
  • Python 3Used only by the demo script to print conclusions, and later to serve ArchView locally.
  • The repositorygit clone https://github.com/iroha1203/AlgebraicArchitectureTheoryV2.git — the demo, its evidence files, and all fixtures live in the repo. Run everything below from the repo root.

One script, five acts

The shipped demo is a realistic Rust commerce service where a PR makes the customer's card get charged one cent more than the checkout page displays — while every unit test stays green in every build configuration. The checkout → payment → settlement flow is a saga, and the demo measures it as one: the script walks the full SAGA diagnostic staircase (grounding → descent → comparison → gate, with a harmonic-debt price on the side). One script runs the whole story:

The script walks five acts and prints one conclusion per step. This is the exact output to expect:

Two lines carry the whole argument. [grounding head] measured_zero is the trap stated as a measurement: every module satisfies its own displayed money law, chart by chart. [descent head] measured_nonzero is the defect: the reconciliation residual has odd drift parity around the money loop, so its class in Z1/B1 is nonzero — no assignment of per-module fixes closes the loop. The base act shows the other half of the discipline: the same LawPolicy asks the same SAGA questions, but with no RepairPlan supplied every stage stays typed silence (silence_by_design), not a conclusion.

Act 1 proves the point of the demo: all three build states (base, head, repaired) compile, pass cargo test, and run. The head state then prints the drift itself:

Three modules speak three money conventions — grand-total half-up in checkout, per-line half-to-even in the payment adapter, exact ten-thousandths in the new settlement ledger. Each is locally correct under its own spec. On the loop they cannot all be right, and ArchSig measures exactly that.

Note the two different comparison outcomes. Base→head is not comparable: the PR added a settlement context, so the measurement site itself changed, and ArchSig refuses to pretend the two runs measure the same thing — the gate blocks on the head run's own verdict instead. Head→repaired keeps the same site, so the comparison is row-comparable and records that the measured obstruction is no longer recorded. Why ArchSig is this strict is explained in Concepts.

The same acts by hand

Every step of the script is an ordinary CLI call over declared inputs: an evidence file (ArchMap), the policy components (LawPolicy, law-equation-surface, two MeasurementProfiles — the second carries the analytic cost model for harmonic debt), and a RepairPlan supplying the SAGA data. Here is the head measurement, the gate, and the repair, exactly as the script runs them.

The run directory now holds the full artifact set — packet, summary, insight report, viewer data, validation reports, manifest. Start with the summary's readThisFirst block:

The word class is earned, not decorative: it only appears because the RepairPlan supplied and passed validation for the triple overlap, the F₂ coefficient, the true-sheaf certificate, and the gluing data. Remove the --repair-plan flag and the same command still runs — but every SAGA row degrades to not_computed with a silence_by_design boundary, exactly like the base act.

The gate report's decision is BLOCKED_BY_GATE_POLICY, and the command exits with code 1 — that exit code is the CI contract. The policy file maps measured_nonzero → block; the mapping is yours to own, per repository.

The repair is what payment systems actually do: designate one money authority, keep submitting PSP line items per the processor spec, and book the rounding residue as an explicit adjustment line. In the evidence file this is one shared convention on every context, and in the RepairPlan the residual support is empty on every overlap — the residual lies in B1, the summary upgrades to REPAIR_GLUES_WITHIN_SELECTED_COMPLEX, compare records MEASURED_OBSTRUCTION_NO_LONGER_RECORDED_AFTER_CHANGE, and the gate exits 0 with PASS_WITHIN_GATE_POLICY. The harmonic-debt reading tells the last part of the story in cents: the head state's essential repair lower bound of 0.353553 (the quarter-cent no rounding scheme removes, under the declared cost model) drops to 0.0 once the residue is booked explicitly.

Your own repository

The demo ships its evidence files ready-made. On your own code, those inputs get authored — normally by your agent, with your approval at the scope step. That authoring path (ArchMap survey, policy selection, validation) is the subject of Contracts.
