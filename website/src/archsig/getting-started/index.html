Path A: supervise an agent

If your LLM agent has the ArchSig Skills installed (they ship in the release bundle), you do not run commands at all. Ask for the analysis, approve the scope the agent proposes, and read the findings.

The agent will build the evidence file (archmap-creater), select a policy (law-policy-creater), run the measurement, and come back with findings tied to source locations (archsig-reader). What you should check before trusting the reading is covered in the Manual's task recipes. The rest of this page is Path B: the same measurement driven by hand, so you can verify what the agent does.

Prerequisites

  • Rust toolchaincargo on your PATH (any recent stable). The demo builds ArchSig and a 3,000-line sample service from source.
  • Python 3Used only by the demo script to print conclusions, and later to serve ArchView locally.
  • The repositorygit clone https://github.com/iroha1203/AlgebraicArchitectureTheoryV2.git — the demo, its evidence files, and all fixtures live in the repo. Run everything below from the repo root.

One script, four acts

The shipped demo is a realistic Rust commerce service where a PR makes the customer's card get charged one cent more than the checkout page displays — while every unit test stays green in every build configuration. One script runs the whole story:

The script walks four acts and prints one conclusion per step. This is the exact output to expect:

Act 1 proves the point of the demo: all three build states (base, head, repaired) compile, pass cargo test, and run. The head state then prints the drift itself:

Three modules speak three money conventions — grand-total half-up in checkout, per-line half-to-even in the payment adapter, exact ten-thousandths in the new settlement ledger. Each is locally correct under its own spec. On the loop they cannot all be right, and ArchSig measures exactly that.

Note the two different comparison outcomes. Base→head is not comparable: the PR added a settlement context, so the measurement site itself changed, and ArchSig refuses to pretend the two runs measure the same thing — the gate blocks on the head run's own verdict instead. Head→repaired keeps the same site, so the comparison is row-comparable and records that the measured obstruction is no longer recorded. Why ArchSig is this strict is explained in Concepts.

The same acts by hand

Every step of the script is an ordinary CLI call over declared inputs: an evidence file (ArchMap) and a fixed policy bundle containing LawPolicy, law-equation-surface, and MeasurementProfile. Here is the head measurement, the gate, and the repair, exactly as the script runs them.

The run directory now holds ten artifacts — packet, summary, insight report, viewer data, validation reports, manifest. Start with the summary's readThisFirst block:

The gate report's decision is BLOCKED_BY_GATE_POLICY, and the command exits with code 1 — that exit code is the CI contract. The policy file maps measured_nonzero → block; the mapping is yours to own, per repository.

The repair is what payment systems actually do: designate one money authority, keep submitting PSP line items per the processor spec, and book the rounding residue as an explicit adjustment line. In the evidence file this is one shared convention on every context — the mismatch support is gone, compare records MEASURED_OBSTRUCTION_NO_LONGER_RECORDED_AFTER_CHANGE, and the gate exits 0 with PASS_WITHIN_GATE_POLICY.

Your own repository

The demo ships its evidence files ready-made. On your own code, those inputs get authored — normally by your agent, with your approval at the scope step. That authoring path (ArchMap survey, policy selection, validation) is the subject of Contracts.
