Part I

Atoms, Objects, and Laws

AAT begins with the Atom: a primitive typed architectural fact. Part I axiomatizes atoms, generates atom families, configurations, molecules, and architecture objects from them, introduces invariants, laws, and obstruction circuits over those objects, and proves that lawfulness and zero obstruction coincide under explicitly selected soundness, completeness, and aggregation assumptions. The chapter closes by assembling these ingredients into an object algebra: architecture as algebra.

How to read the claims

AAT states its claims relative to a fixed atom vocabulary, law universe, coverage topology, and selected reading. Three claim grades are kept distinct throughout the monograph.

Formal theorem
A mathematical proposition proved from the axioms, definitions, and stated assumptions.
Certified bounded inference
A relative inference that holds under explicitly stated coverage, exactness, or witness-completeness assumptions. It is never an unconditional absolute claim.
Analytic reading
A way of reading a constructed geometric object through graphs, signatures, periods, or metrics.

Theorem labels on this page follow the canonical text. A label carries a Lean status badge only when the AG Lean index has a corresponding ledger row; a label without a badge is a mathematical statement in the canonical text and is not read as a current Lean proved claim.

1. Primitive architectural fact

The minimal unit of AAT is the Atom. An Atom is a typed fact about software architecture that the theory treats as indivisible: components, relations, capabilities, state, effects, authority and trust, contracts, semantic facts, and runtime interactions all appear in atomic form. The set of atoms is written At, and every atom a : At carries five coordinates: 

kind(a)       : AtomKind
axis(a)       : Axis
subject(a)    : Subject
predicate(a)  : Predicate
payload(a)    : Payload

kind(a) names the atom's family, axis(a) names the structural axis the atom concerns, subject(a) names what the atom is about, predicate(a) states the fact that holds, and payload(a) carries content: a value, name, type, evidence handle, or weight.

Definition I.1.1 (Atom). An Atom is a typed fact a = (kind, axis, subject, predicate, payload) such that predicate gives exactly one atomic statement about subject.

An atom may be a proposition or a value. Inside AAT, atoms are generators: laws select atoms, operations map atoms, and signatures read the behavior of atoms, but the existence of atoms starts from the atom axiom system and nowhere else.

Example I.1.2 (Basic atom schemas). The current core atom families give the following schemas. This family is the current core, not a closed enumeration: if a primitive architectural fact stably resists expression as a molecule of existing families, a new atom family may be added. 

component(c)
relation_r(c, d)
capability(c, k)
state(c, x)
effect(e)
authority(actor, action, resource)
trust_relation(source, target)
contract(m, p)
semantic(t, s)
runtime_interaction(u, v, h)

Example I.1.3 (Reading the basic atoms)

The basic atoms are the minimal vocabulary for building architecture objects. Each atom is a generator connected to later laws, obstructions, operations, and signature axes, not a bare label.

Component atoms

component(c) asserts that an architecture unit exists: a module, service, class, package, process, table, queue, adapter, or port. A component atom generates a vertex in a graph representation, but the vertex is not the whole content: the same component becomes a richer architecture object by carrying state, capability, contract, effect, and semantic atoms. Component atoms answer which units enter an architecture object's carrier and which can serve as relation sources and targets. Lawfulness never follows from component atoms alone; components supply the objects that laws act on.

Relation atoms

relation_r(c, d) asserts that a relation holds between two or more atoms or subjects: dependency, calls, reads, writes, publishes, subscribes, owns, implements, substitutes, projects. A binary relation generates a graph edge with source = c, target = d, predicate = r. Relation atoms connect to cycle, reachability, layering, projection, substitution, and effect ordering laws, and they are the principal material of obstruction circuits.

Capability atoms

capability(c, k) asserts that a component or object provides an ability: an interface, port, query, command, handler, storage access, serialization, calculation. Capability sits between relation and contract: a relation says where something connects, a contract says what is promised, and a capability says what can be provided. Typical laws demand that a required capability is provided, that a provided capability satisfies a selected contract, and that capability use factors through a declared interface. A missing capability yields a missing-capability obstruction; a capability that exists but fails its contract yields a substitution or semantic obstruction.

State atoms

state(c, x) asserts that an architecture object holds a state quantity: a database column, aggregate field, cache entry, session value, event-log projection, or configuration value. State atoms generate state spaces and transition laws, and they raise the questions of which operations read or update state, which invariants are preserved by transitions, and which effects demand ordering around state updates. State atoms surface obstructions invisible to the static component graph: a dependency graph can be acyclic while state transitions fail to commute.

Effect atoms

effect(e) asserts that an architecture object acts on external or internal state: message send, email, payment capture, database write, cache invalidation, event publish, file write, remote call, retry, compensation. Effect atoms connect to ordering, idempotence, replay, compensation, authority, and handler-existence laws. Effect obstructions frequently do not appear in the dependency graph, so AAT keeps effects as independent atoms rather than collapsing them into component relations.

Authority and trust atoms

authority(actor, action, resource) asserts who holds the authority or trust to perform which action on which resource. Owner, visibility, permission, policy scope, access path, encapsulation, and allocation facts belong to this family, with schemas such as permission(subject, action, resource), trust_relation(source, target), owner(subject, owner), visibility(subject, v), policy_scope(policy, target), access_path(subject, resource), and encapsulates(subject, detail). Authority is not dependency and not effect: that a component can reach a resource and that it is authorized to act on it are distinct primitive facts. Typical laws demand that effects require authority, access paths respect permissions, hidden details remain encapsulated, and policies apply to their selected targets; their failures produce unauthorized-effect, leaked-detail, policy-mismatch, and unexpected-access-path obstructions.

Contract atoms

contract(m, p) asserts what an operation, capability, component, or effect must satisfy: preconditions, postconditions, return types, error behavior, event shapes, idempotency promises, latency promises, authorization rules. Contract atoms are the center of substitution laws: 

substitutes(Impl, Base)
contract(Base, C)
contract(Impl, C')
C' refines C

When C' fails to satisfy C, a substitution obstruction arises. Matching interface types do not give lawfulness when the contract atoms disagree.

Semantic atoms

semantic(t, s) asserts a semantic interpretation of an architecture object: domain invariant, business meaning, identity, ownership, money amount, authorization meaning, event meaning. The same types and relations under a different meaning give a different architecture reading. The uniqueness of semantic atoms is defined relative to rules of use rather than as an absolute uniqueness of meaning objects. This transfers the late-Wittgensteinian language-game position — meaning is use — to AAT's semantic atoms: the meaning of a program fragment or architecture subject is read from how it is used within a rule-governed practice. 

UseContext Γ
UseRuleSet R
ExtractorSemantics E
Normalization N

semantic_Γ(t, s)

Here \( \Gamma \) fixes which workflows, contracts, operations, state transitions, effects, authorization rules, tests, examples, and documented practices t participates in; R fixes identification, distinction, preservation, substitution, and failure within that use-context; E reads the source's syntax, types, language specification, and extraction rules; and N fixes identification and normalization. Together these form the extraction doctrine D = (V, Γ, R, ρ, E, N), and the uniqueness of the semantic atom family takes the form: if Atomize_D(S) = F and Atomize_D(S) = G, then F = G.

Semantic atoms do not demand single membership. The same subject may carry several semantic atoms at once — an overlap of semantic facts, not a non-uniqueness of meaning. If the same email-sending capability serves both two-factor authentication and newsletter delivery, the canonical atom family contains both semantic atoms: 

semantic(sendEmail, twoFactorAuthentication)
semantic(sendEmail, newsletterDelivery)

semantic_{twoFactorAuthGame}(sendEmail, verificationChannel)
semantic_{newsletterGame}(sendEmail, deliveryChannel)

An observation that collapses these into a coarse meaning such as notification is a coarse semantic observation. Semantic atoms separate static flatness from semantic flatness: a tidy dependency graph leaves an obstruction standing whenever a diagram demanded by a semantic atom fails to commute.

Runtime and interaction atoms

runtime_interaction(u, v, h) asserts a runtime call, message, event, retry, timeout, protection, synchronization, transaction, or schedule constraint, with schemas such as runtime_call(u, v), rpc(u, v), message(ch, payload), event_emission(u, event), event_handling(h, event), retry(op), timeout(op), circuit_breaker(op), bulkhead(op), lock(resource), transaction(scope), and schedule_constraint(subject, constraint). These are not obstructions: a timeout or circuit breaker is a primitive fact that runtime laws act on, not a failure. A flat static relation surface can still host a separate runtime obstruction, so runtime atoms are kept independent of relation and effect atoms.

Combining atoms

Basic atoms are meaningful alone, but AAT's objects appear as combinations. A family mixing component, relation, capability, state, contract, effect, authority, trust, semantic, and runtime atoms generates a molecule for one concern, and from it relation laws, state transition laws, effect ordering laws, authority laws, contract preservation laws, runtime interaction laws, and semantic consistency laws rise at once.

Example I.1.4 (Atom extraction from a source fragment)

Atoms are not the source code itself. To read a source fragment as an architecture object, an atom family is extracted from its declarations, references, calls, state updates, external effects, type annotations, and semantic annotations. Consider the abstract fragment:

Reading this source under the atom vocabulary determines, essentially, the following atom family. Semantic annotations enter the family as primitive facts appearing in the source.

Each atom does distinct work. The import and call relations feed dependency, reachability, layering, and cycle readings. The state and write atoms raise state transition laws. The effect and emit atoms raise effect ordering, retry, idempotence, and compensation laws. The authority atom raises access, permission, and encapsulation laws. The runtime interaction and timeout atoms raise runtime protection, retry, timeout, transaction, and synchronization laws. The contract and semantic atoms raise substitution, projection, and semantic consistency laws.

The atom family itself is read uniquely under the same source and atom vocabulary. Semantic reading and resolution are not existence conditions on atoms; they are downstream interpretation and projection or truncation of the canonical family: 

canonical:
  component(C), component(D), relation_imports(C,D),
  relation_calls(m,D.read), relation_reads(m,input.id),
  relation_writes(m,x), effect(e), relation_emits(m,e),
  authority(a,act,r), runtime_interaction(m,D.read,call),
  timeout(m,T), contract(m, P -> Q),
  semantic(q(y), denotes result-of-m)

coarse projection:
  component(C)
  relation_imports(C,D)
  contract(m, P -> Q)

AAT reads laws, obstructions, and signatures over the canonical atom family. Conclusions computed on a coarse projection are read only for the axes that projection preserves.

2. Atom axiom system

The atom axiom system is the starting point from which AAT rises. It fixes atoms as generators and fences off laws and observations from generating atom truth.

Axiom A0 (Primitive existence)
At is the type of primitive facts. AAT's architecture objects are generated from At.
Axiom A1 (Typing)
Every atom has a kind and an axis: for all a : At, kind(a) and axis(a) are defined.
Axiom A2 (Single fact)
One atom expresses one primitive fact. Composite claims are expressed as sets of atoms or configurations.
Axiom A3 (Predicate stability)
Atom identity is read through type, subject, predicate, and content: a = b iff the five coordinates kind, axis, subject, predicate, payload agree.
Axiom A4 (Composition)
A finite family of atoms generates a configuration: from F subset At, form Config(F).
Axiom A5 (Law non-generation)
Laws do not generate the holding of atoms. A law acts on an atom family as constraint, preservation condition, commutation condition, or absence condition.
Axiom A6 (Observation non-generation)
Observation, measurement, reporting, inspection, and recording do not generate the holding of atoms. They give readings of an atom family; they are not sources of the atoms themselves.
Axiom A7 (Operation preservation of atom-origin)
Architecture operations map atom families to atom families: op : Config(F) -> Config(G) with F, G subset At. Operations therefore close as algebraic transformations of architecture objects.
Axiom A8 (Essential uniqueness)
When a source and an extraction doctrine are fixed, the canonical atom family determined by that source is unique. The doctrine D = (V, Γ, R, ρ, E, N) covers not only the atom vocabulary but semantic reading, resolution, source semantics, and normalization.

Axiom A8 relativizes uniqueness rather than weakening it. The same source under the same atom vocabulary but a different semantic use-context, use-rule, resolution, source semantics, or normalization yields an atom family relativized to a different doctrine — a relativization of uniqueness, not a failure of uniqueness. Reading a fixed doctrine's canonical family coarsely is a projection pi : F -> F_coarse: not a rival candidate for atom truth, but a record of which atoms are visible, which are merged, and which relations are forgotten.

Proposition A9

Observation incompleteness and existence uniqueness

The existence of an atom and the observation of an atom are not the same. Incomplete observation does not make the existence of the atom family ambiguous: the canonical atom family exists uniquely while observation may be partial, coarse, or coordinate-forgetting.

Observation is read as a map from the canonical family into observed values, obs : F -> O. When obs is not injective, distinct atoms collapse to the same observed value — a != b with obs(a) = obs(b) — which shows the observation map forgetting information, not atoms losing uniqueness. Likewise an atom absent from the observed values need not be absent from F. 

Atom existence      : the unique fact fixed by the canonical source
Atom observation    : a map reading that fact
Atom projection     : a coarse reading at the observation's resolution
Atom reconstruction : an attempt to return from observed values to the canonical family

Correct reconstruction requires the observation map to preserve enough information, but a failed reconstruction never breaks the existence uniqueness of atoms.

3. Atom family

An atom family is a collection of atoms treated together.

Definition I.3.1 (Atom family). An atom family F is a subset of At: F subset At. A family may mix component, relation, state, contract, effect, and semantic atoms. AAT does not exclude the mixture; it asks which laws the mixed primitive facts satisfy and which obstructions they generate.

Definition I.3.2 (Support). support(F) = { subject(a) | a in F } is the set of subjects appearing in F. The support is not restricted to components: operations, states, effects, and contracts also appear as subjects.

Definition I.3.3 (Axis restriction). For an axis x, F|x = { a in F | axis(a) = x }. Reading the dependency, state, effect, contract, and semantic axes separately yields different faces of the same architecture object.

Definition I.3.4 (Compatibility). F is compatible when it never simultaneously contains contradictory payloads on the same subject and predicate: 

Compatible(F)
  iff not exists a b in F,
      sameSlot(a,b) and inconsistent(payload(a), payload(b))

Compatibility is not completeness: a compatible family can still lack atoms it needs.

Definition I.3.5 (Closure). For a family of inference rules R, cl_R(F) is the least G with F subset G and R(G) subset G. Closure extends the family, and atoms in the closure are treated as derived atoms justified by R; when primitive and derived atoms must be distinguished, an origin marker is attached.

4. Configuration and molecule

Atoms alone are not architecture objects. Architecture objects arise from the placement of atoms and their interrelation.

Definition I.4.1 (Atom configuration). An atom configuration is a triple C = (F, R, E), where F is an atom family, R is a relation among atoms — dependency, coherence, generation, reference, substitution, realization — and E is identification or equivalence data expressing identification and abstraction.

Definition I.4.2 (Molecule). A molecule is a finite atom arrangement inside a configuration that can be treated as one unit of meaning: M = (F_M, R_M, E_M) with F_M subset F and R_M, E_M restricted from C.

Example I.4.3 (Concern molecule). A finite arrangement carrying one architecture concern: 

component(c)
component(d)
relation_r(c, d)
capability(d, k)
contract(m, p)
state(c, x)
effect(e)
semantic(t, s)

The semantic atom gives the semantic facts inside the concern. A selected reading — which role, responsibility, or pattern name explains this molecule — is downstream interpretation and not an existence condition on the molecule.

Definition I.4.4 (Subconfiguration). C' <= C holds when the atoms, relations, and identifications of C' are contained in C: F' subset F, R' subset R, E' subset E. Subconfigurations express the substructures of an architecture object.

5. Architecture object

The architecture object is the central object of AAT, generated from atom configurations.

Definition I.5.1 (Architecture object). An architecture object is a triple A = (C, S, Q), where C is an atom configuration, S is a family of structure maps into graphs, categories, algebras, diagrams, and state transitions, and Q is a family of selected quantities — invariants, measures, signature axes, curvature values.

Definition I.5.2 (Generated object). F => A expresses that atom family F generates architecture object A: a configuration C is built from F, and A is obtained by equipping C with structure maps and selected quantities.

Proposition I.5.3

Atom-Origin

Lean: proved

Every architecture object is generated from an atom family:

\[ \forall A,\ \exists F,\ F \Rightarrow A. \]

This proposition is the generation principle of AAT: there is no architecture object whose origin lies outside the atoms.

Lean anchor
AAT.AG.ArchitectureObject.atom_origin, with ArchitectureObject.GeneratedObject and the generated_family_eq / generated_family_subset lemmas.

Definition I.5.4 (Object equivalence). Two architecture objects A and B are equivalent, A ~= B, when there exists a configuration equivalence preserving the selected structure maps and selected quantities. Objects generated from the same atom family may still be read as different objects when their structure maps are chosen differently.

6. Invariant

An invariant is a quantity or property preserved on architecture objects.

Definition I.6.1 (Invariant). An invariant on an architecture object is a function I : Obj -> Value or a predicate P : Obj -> Prop.

Definition I.6.2 (Invariant family). An invariant family is a collection of invariants to be preserved together: Inv(A) = { I_i | i in J }.

Definition I.6.3 (Preservation). An operation op : A -> B preserves invariant I when \( I(A) = I(B) \), or, for ordered quantities, when \( I(B) \le I(A) \).

Example I.6.4 (Preserved quantities). Typical invariants — each corresponding to a distinct law and obstruction: 

number of forbidden dependency atoms
cycle count
projection violation count
contract mismatch count
effect replay failure count
semantic inconsistency count
Proposition I.6.5

Layers of design principles

A design principle is not a single slogan; it is a reading that ties together several invariant families and operation families. The representative classification:

Design principle / pattern Principal invariants Layer
SRP / OCP / LSP / ISP / DIP Local contracts, abstraction, substitutability, interface separation Local contract layer
Layered Architecture Ranking, dependency direction, acyclicity, decomposability Global structure layer
Clean Architecture Inside/outside separation, inward dependency, abstraction coherence Global structure layer
Event Sourcing / Saga / CRUD Replay, projection, compensation, history-versus-overwrite relations State transition algebra layer
Circuit Breaker Runtime protection, failure locality Runtime dependency layer
Replicated Log Ordering, quorum, failure model, conditional convergence Distributed convergence layer

Local-contract-layer invariants do not automatically imply the invariants of the global structure, state transition algebra, runtime dependency, or distributed convergence layers. Proposition I.6.5 is a mathematical statement in the canonical text.

7. Law

A law expresses commutation, preservation, absence, and coherence on architecture objects.

Definition I.7.1 (Law). A law L is a predicate on architecture objects, L : Obj -> Prop. Structurally, laws take the forms: 

L(A) iff diagram(A) commutes
L(A) iff forbidden(A) is empty
L(A) iff projection(A) is sound
L(A) iff substitution(A) preserves contract
L(A) iff operation(A) preserves invariant

Definition I.7.2 (Law universe). A law universe is the family of laws considered together, Law(A) = { L_i | i in K }, further read through its components: 

required laws
optional laws
derived laws
witness family
selected reading
coverage assumptions
exactness assumptions

At the definition stage, lawfulness is not identified with obstruction absence. AAT first places semantic lawfulness and connects it through finite witness families and signature axes: 

SemanticLawful(A,U)
  = A is semantically lawful for the selected law universe U

NoRequiredObstruction(A,W)
  = the selected witness family W contains no required bad witness

RequiredSignatureAxesZero(A,S)
  = the selected signature axes S are zero

Definition I.7.3 (Lawfulness). An architecture object A is lawful for a law universe U when every law in U holds:

\[ \mathrm{Lawful}_U(A) \iff \forall L \in U,\ L(A). \]

Example I.7.4 (Representative laws). NoCycle, ProjectionSound, SubstitutionCompatible, LayerRespecting, EffectReplayLawful, StateTransitionCommutative, SemanticInterpretationConsistent. These laws are not natural-language design principles themselves; they are mathematical conditions on atom configurations.

8. Obstruction circuit

When a law fails, the failure is read as an obstruction.

Definition I.8.1 (Obstruction). An obstruction for a law L, written Ob_L(A), is a structure witnessing that L(A) fails.

Definition I.8.2 (Obstruction circuit). An obstruction circuit is a finite arrangement of atoms and relations that triggers a law failure: O = (F_O, R_O, L) with F_O subset At, R_O a relation on F_O, and not L(A).

Example I.8.3 (Cycle obstruction). An obstruction circuit for NoCycle: 

depends(A, B)
depends(B, C)
depends(C, A)

Example I.8.4 (Substitution obstruction). An obstruction circuit for substitution compatibility: 

contract(Base, returns NonNull)
contract(Impl, returns Nullable)
substitutes(Impl, Base)

Definition I.8.5 (Obstruction valuation). An obstruction valuation assigns a quantity of obstruction per law, omega_L(A) : Value — typically a count, a boolean, a weight, a rank, or an energy. To read zero obstruction and lawfulness as equivalent, the value domain and the aggregation are not arbitrary. The zero theorem is stated only when the selected value domain Value_U carries: 

0 in Value_U
positive predicate (> 0)
zero/positive dichotomy for selected readings
no cancellation at zero

and the aggregation over the law universe, Agg_U : (forall L in U, Value_U) -> Value_U, is assumed zero-reflecting:

\[ \mathrm{Agg}_U\big((v_L)_{L \in U}\big) = 0 \iff \forall L \in U,\ v_L = 0. \]

This assumption excludes signed sums and cancellation-friendly aggregations; sums of counts, disjunctions of booleans, sums and suprema of nonnegative weights are the typical instances. Write \( \omega_U(A) := \mathrm{Agg}_U\big((\omega_L(A))_{L \in U}\big) \).

9. Lawfulness and zero obstruction

Lawfulness and zero obstruction read the same phenomenon from two directions.

Proposition I.9.1 (Obstruction soundness). A valuation omega_L for law L is sound when the law's holding forces the selected obstruction to vanish: \( L(A) \rightarrow \omega_L(A) = 0 \). Equivalently, under the zero/positive dichotomy, \( \omega_L(A) > 0 \rightarrow \neg L(A) \).

Proposition I.9.2 (Completeness). An obstruction family for L is complete when a law failure is always detected as a selected obstruction: \( \neg L(A) \rightarrow \omega_L(A) > 0 \). Equivalently, under the zero/positive dichotomy, \( \omega_L(A) = 0 \rightarrow L(A) \).

Theorem I.9.3 — Certified bounded inference

Lawfulness–Zero Obstruction

Lean: proved

Assume, for a law universe \( U \):

forall L in U, omega_L is sound.
forall L in U, omega_L is complete.
Agg_U is zero-reflecting.
Lawful_U(A) iff forall L in U, L(A).

Then

\[ \mathrm{Lawful}_U(A) \iff \omega_U(A) = 0. \]

Proof reading
\( \mathrm{Lawful}_U(A) \iff \forall L \in U,\ L(A) \iff \forall L \in U,\ \omega_L(A) = 0 \iff \omega_U(A) = 0 \). The second equivalence is soundness with completeness; the last is zero-reflecting aggregation.

10. Architecture as algebra

AAT's objects, maps, laws, and obstructions form an algebra.

Definition I.10.1 (Object algebra). An object algebra consists of the data Obj, Op, Inv, Law, Ob, Sig: architecture objects, operations, invariants, the law universe, obstruction circuits, and signatures.

Definition I.10.2 (Operation). An operation is a map of architecture objects, op : A -> B; at the atom level it is a transformation of configurations, op_At : Config(F) -> Config(G).

Definition I.10.3 (Signature). A signature is a multi-axis presentation collecting the selected quantities of an architecture object: Sig(A) = (q_1(A), q_2(A), ..., q_n(A)). A signature is not a single value: different law failures appear on different axes.

Proposition I.10.4 (Operation reading). An operation op : A -> B carries one or more of the following roles: 

preservation      : preserves selected invariants
reflection        : exposes obstructions
repair            : decreases the obstruction valuation
extension         : extends functionality by enlarging the atom family
synthesis         : constructs an object satisfying the laws
translation       : changes the representation

Chapter summary

Part I fixes the generative base of AAT. The atom axiom system supplies primitive typed facts and forbids laws and observations from generating them; atom families, configurations, and molecules organize those facts; every architecture object is generated from an atom family (Proposition I.5.3); invariants, laws, and obstruction circuits read structure over those objects; and under soundness, completeness, and zero-reflecting aggregation, lawfulness coincides with zero obstruction (Theorem I.9.3, a certified bounded inference). Theorem I.10.5 packages the whole tower — atom to signature — as an object algebra.

Part II takes this tower geometric: architecture context categories, coverage, AAT sites, and sheaves of architectural data over them.

Terms

Atom
A primitive typed architectural fact a = (kind, axis, subject, predicate, payload); the generator of AAT.
Extraction doctrine
The tuple D = (V, Γ, R, ρ, E, N) fixing vocabulary, use-context, use-rules, resolution, source semantics, and normalization; canonical atom families are unique relative to a doctrine (Axiom A8).
Configuration / molecule
C = (F, R, E): an atom family with relations and identifications; a molecule is a finite meaning-unit arrangement inside it.
Architecture object
A = (C, S, Q): a configuration with structure maps and selected quantities; always atom-generated.
Law universe
The family of laws considered together, with required / optional / derived roles, witness families, and coverage and exactness assumptions.
Obstruction circuit
A finite atom-and-relation arrangement witnessing a law failure.
Zero-reflecting aggregation
An aggregation whose zero forces every per-law valuation to be zero; the hypothesis under which lawfulness and \( \omega_U(A) = 0 \) coincide.
Object algebra
The data Obj, Op, Inv, Law, Ob, Sig in which AAT's constructions close.
