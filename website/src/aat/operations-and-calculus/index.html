Operations and Calculus

Architecture changes become mathematical only when they state their source, target, preconditions, preserved invariants, witness transport, repair measure, or synthesis certificate. AAT treats design work as bounded operations, not as unconditional improvement.

Research question

When does an architecture change preserve enough structure to be justified rather than merely described as a refactoring?

AAT answers by demanding an operation package. The package records which architecture state changes, which preconditions are visible, which invariant or obstruction family is selected, how witnesses are transported, which conclusion is licensed, and which readings are excluded. The operation name is only a tag until that evidence is supplied.

Architecture operation

An architecture operation is a selected move from a source architecture state to a target architecture state. It may be a replacement, split, merge, protection step, migration, repair, or synthesis action. The operation becomes mathematical when its precondition and proof obligation say what the move preserves, reflects, improves, localizes, translates, or assumes.

Definition 1.1. An ArchitectureOperation is an evidence-indexed partial architecture-state move carrying source and target states, an operation kind, a precondition, a generated proof obligation, witness predicates before and after the move, a target-to-source witness map, a soundness proof for that map, and recorded non-conclusions.

Proposition 1.2. The operation schema exposes a one-way witness bridge: every selected target witness can be transported back to a selected source witness through the stored witness map.

Proof. Lean stores witnessMap and witnessMappingSound as fields of ArchitectureOperation. The accessor theorem exists_sourceWitness_of_targetWitness packages the transported source witness existentially.

Kinds and roles

Operation kinds classify the move. Operation roles classify how a theorem package reads the move. This separation matters: a repair tag does not prove repair, and a preserve role tag does not prove preservation. The proof obligation supplies the conclusion.

Vocabulary Examples What it does not prove by itself
Operation kind compose, refine, abstract, replace, split, merge, isolate, protect, migrate, reverse, contract, repair, synthesize Preservation, improvement, safety, or solvability.
Operation role preserve, reflect, improve, localize, translate, transfer, assume The selected role conclusion until the proof package is discharged.

Proposition 2.1. A discharged preserve-role package yields selected invariant preservation only when its bounded conclusion is exactly the source-to-target preservation statement.

Proof. preservesInvariant_of_discharged_preserve requires the role tag, the conclusion-shape equality, a discharged role package, and visible assumptions. The tag alone is not enough.

Proposition 2.2. A discharged improve-role package yields only the selected improvement relation named by that package.

Proof idea. selectedImprovement_of_discharged_improve reads the conclusion through a caller-supplied relation improves : State -> State -> Prop. No all-axis monotonicity follows.

Operation laws

AAT names calculus laws such as identity, associativity, edge-union composition, refinement / abstraction compatibility, edge equivalence, protection idempotence, runtime localization, migration compatibility, reverse involution, witness mapping, synthesis soundness, and no-solution soundness. Each is a bounded theorem package with visible assumptions.

Proposition 3.1. A bounded calculus law yields its recorded conclusion from its recorded assumptions.

Proof. ArchitectureCalculusLaw.conclusion_of_assumptions applies the law package's sound field to boundedUniverse, compatibility, coverage, exactness, and observation-equivalence assumptions.

Example 3.2. finiteComposeEdgeUnionLaw proves a concrete edge-union statement for finite graph composition. It is not an unconditional associativity or flatness-preservation theorem.

Example 3.3. localReplacementPreservationLaw reads a LocalReplacementContract as projection soundness and LSP compatibility. This is separate from graph-level edge equivalence replacement.

Invariant correspondence

The calculus links operation families and invariant families. For selected invariants \(S\), Ops(S) is the family of operations that preserve every invariant in \(S\). For selected operations \(T\), Inv(T) is the family of invariants preserved by every operation in \(T\).

Theorem 4.1. Operation families and invariant families form a weak preservation correspondence: T subset Ops(S) iff S subset Inv(T).

Proof. Lean proves operationInvariant_galois directly from the definitions of Ops, Inv, and predicate-family inclusion. The antitone and closure theorems make the design tradeoff explicit: stronger invariant requirements reduce allowed operations; broader operation families reduce the invariants preserved by all operations.

Feature extension

Feature extension is the software-evolution operation in which an existing core is embedded into an extended architecture and a feature view is added. It is not automatically safe. Static split, feature-view soundness, lifting, runtime coverage, semantic coverage, and obstruction classification are separate evidence packages.

Definition 5.1. A feature extension is a bounded operation that expands a core by adding feature-owned atoms, relations, views, and interaction evidence while keeping the selected core embedding visible.

Proposition 5.2. Extension obstruction classification is explanatory and may be multi-label: inherited core obstruction, feature-local obstruction, interaction obstruction, lifting failure, filling failure, complexity transfer, and residual coverage gap are not required to be disjoint.

Proof idea. Lean defines ExtensionObstructionClass, ExtensionObstructionWitness, and MultiLabelExtensionObstructionWitness. The conversion toMultiLabel preserves the selected classification without asserting an exhaustive disjoint decomposition of reality.

Repair

Repair is an operation read through a selected obstruction universe and selected measure. A strict repair step decreases that selected measure. It does not claim all-axis improvement, runtime safety, semantic safety, or global flatness unless those are additional package conclusions.

Definition 6.1. A repair step is strict for a selected obstruction universe when the target's selected obstruction measure is strictly smaller than the source's selected obstruction measure.

Proposition 6.2. An admissible repair rule decreases the selected obstruction measure for the selected witness, rule, and step.

Proof. repairStepDecreases_of_admissible applies the decreasesSelectedMeasure field of an AdmissibleRepairRule. The law-package entrypoint repairStepDecreasesLaw_conclusion exposes the same selected-measure decrease through the bounded calculus-law interface.

Proposition 6.3. A finite repair package clears only the selected obstruction universe at the target.

Proof. BoundedRepairPlan.selectedObstructionsCleared uses the package's zero-measure-to-clearing bridge. FiniteRepairPackage.selectedObstructionsCleared exposes that selected conclusion. The package still records non-conclusions separately.

Non-conclusion 6.4. Selected repair can transfer complexity or obstruction to another axis. Decreasing one measure is not proof of total architecture improvement.

Synthesis

Synthesis is the operation of producing an architecture candidate that satisfies a selected constraint system. AAT separates candidate soundness from no-solution soundness. A candidate is useful when a soundness package proves it satisfies every required constraint. Failure to produce a candidate is not a proof that no solution exists.

Proposition 7.1. A sound synthesis package proves that its produced candidate satisfies the selected constraint system.

Proof. SynthesisSoundnessPackage.candidate_satisfies returns ArchitectureSatisfies C candidate. The calculus entrypoint synthesisCandidateSoundnessLaw_conclusion exposes that conclusion as a bounded synthesis law.

Proposition 7.2. A no-solution statement requires a valid no-solution certificate.

Proof. NoSolutionCertificate.sound_of_valid requires ValidNoSolutionCertificate. The bounded law noSolutionCertificateSoundnessLaw_conclusion keeps that validity premise visible as its bounded-universe assumption.

Worked example

Consider adding coupons to a payment service. The feature extension adds discount policy, coupon validation, rounding, tax, ledger update, and receipt effects. AAT does not ask whether the feature "looks clean." It asks which operation package is being claimed.

The repair may clear the selected static hidden-dependency witness while leaving a semantic rounding diagram non-commuting. That is not a failure of the calculus. It is exactly the point of the calculus: each operation reports the axis it changes and the axes it does not claim.

Counterexamples and non-conclusions

Counterexample 9.1. A repair operation kind without an admissible repair rule is only a tag. It does not decrease any obstruction measure.

Counterexample 9.2. A selected repair can decrease a static hidden-dependency measure while increasing semantic mismatch or runtime exposure.

Counterexample 9.3. A solver returning no candidate is not a no-solution certificate. No-solution soundness requires a valid certificate package.

Counterexample 9.4. A feature extension can preserve a static split policy while introducing an effect-ordering or rounding-law obstruction.

Formal reading

The Lean surface for this chapter is concentrated in Formal/Arch/Operation, Formal/Arch/Extension, and Formal/Arch/Repair. The important boundary is that operation tags, law tags, and repair suggestions are not theorem conclusions until the corresponding package assumptions are visible.

Formal status 10.1. Operation kind, operation role, proof obligation, architecture operation, operation-invariant correspondence, bounded calculus laws, selected repair, finite repair, synthesis soundness, and no-solution certificate schemas are defined in Lean.

Formal status 10.2. The key bridge theorems are proved: witness reflection from target to source, operation-invariant Galois correspondence, bounded calculus-law conclusion extraction, selected repair measure decrease, selected obstruction clearing for finite repair, synthesis candidate satisfaction, and valid no-solution certificate soundness.

Chapter summary

Operations are the algebra of architecture change. They let AAT ask what a change preserves, what it reflects, what measure it improves, which witness family it repairs, and which constraints a synthesized candidate satisfies.

The claim discipline is the main result. Operation tags do not imply improvement, repair does not imply all-axis monotonicity, feature extension does not imply safety, and solver failure does not imply no solution. Each conclusion belongs to a bounded theorem package.

Terms

ArchitectureOperation
A source-to-target architecture-state move carrying precondition, proof obligation, witness transport, and non-conclusions.
Operation role
The selected reading of an operation: preserve, reflect, improve, localize, translate, transfer, or assume.
ArchitectureCalculusLaw
A bounded theorem package for an operation law under explicit compatibility, coverage, exactness, and observation assumptions.
Feature extension
A local evolution operation embedding a core into an extended architecture with a selected feature view and evidence packages.
Selected repair
A repair claim relative to a chosen obstruction universe, selected witness, rule, and measure.
Synthesis soundness
A theorem package proving a produced candidate satisfies the selected constraint system.
