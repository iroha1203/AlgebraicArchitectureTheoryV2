Computational core

SFT computes over field transitions. The core turns a living development situation into artifact actions, operation support, operation policy, step relations, and bounded paths while preserving the selected field boundary.

Lean-facing SFT isolates the theorem-bearing fragment: fields, supported operations, step relations, and bounded paths. Those paths become the carrier for reachable futures, review projections, and feedback records.

Artifact action

Part 2 begins before code changes. A PRD, spec, issue, design note, or AI proposal acts on the field by opening candidate update paths. It changes what developers consider natural, reviewable, cheap, forbidden, or urgent. SFT calls this an artifact-mediated change.

Vocabulary

ArtifactAction

A set-valued action from a current field to candidate update descriptors under an explicit interpretation boundary.

Source
PRD, issue, spec, review request, incident report, or AI-generated proposal.
Effect
Changes candidate support, policy, constraints, observation, and forecast boundary.

The artifact supplies candidate pressure. SFT reads that pressure as a family of possible field updates, each with its own support, policy, and observation boundary.

Object What it carries
RawArtifactAction source artifact, target field components, interpretation boundary, candidate updates, action boundary, composition boundary, observable boundary, and non-conclusions.
ForceDescriptor action class, target architecture regions, candidate operation support, affected policy, constraints, governance, observation, and uncertainty boundary.
ForecastedEffect forecast boundary, expected comparable axes, expected delta ranges, obstruction witness families, and unknown remainder.
ObservedEffect observed transition, before/after signatures, comparable measured delta, unexpected witnesses, review, CI, and runtime outcome.

Computation rule. The raw action supplies candidate pressure; path computation is generated from that pressure under a field model.

Operation support

Operation support is the SFT object that captures what the field makes possible, natural, low-cost, reviewable, or selected. It is the field-relative shape of what can plausibly happen next under the selected field model.

Vocabulary

OperationSupport

The operation family available at a field state after selected constraints, governance, evidence, and policy are taken into account.

Question
Which changes are available, natural, low-cost, reviewable, or selected?
Boundary
Support records the field-relative space of plausible next operations.

Policy and selection

Support says what is available. Policy says what is easy, preferred, cheaper, or likely to be selected inside that support. This distinction is the computational version of field force: two operations can both be possible while one is much more natural for the current organization, tooling, review culture, and codebase memory.

Vocabulary

OperationPolicy

A preorder, cost model, selection relation, or distribution over supported operations in a selected field.

Minimal core
Preorder, cost, or selected-operation predicate.
Extended semantics
Probability distribution only when a stochastic model is explicitly chosen.

Step relation

A StepRelation says when a supported operation can move one selected field estimate to another. Once steps exist, software evolution becomes a path object with witnesses, support, and observation.

Vocabulary

StepRelation

The relation that says a supported operation can move one selected field estimate to another.

Input
Field F, operation op, candidate next field F'.
Output
A witnessed step that can be composed into bounded field paths.
