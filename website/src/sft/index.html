Software Field Theory

SFT makes software evolution computable. It treats the whole development organization — requirements, code, reviews, agents, incidents, and governance — as a field of forces with attractors that pull a codebase toward some futures and away from others.

Why now

AI has made code generation fast. A requirement, prompt, or review comment can now become a patch before its future costs are visible.

Every patch changes the field around the next patch: tests, dependencies, ownership, review habits, examples, incidents, and the paths developers and agents learn to copy. The speed of generation turns software evolution itself into the engineering object.

Object

SFT studies software evolution as field-shaped computation. Its object is the organization-code field that makes some futures cheap, visible, repeatable, and reviewable while pushing other futures out of reach.

SFT treats requirements, issues, reviews, CI, incidents, prompts, ownership, and codebase memory as forces that shape operation support: which changes can be made, selected, justified, repeated, and observed.

Construction

SFT is built as a stack. AAT defines architecture as a local algebra of components, operations, invariants, and theorem boundaries. ArchSig observes bounded signatures of that algebra. SFT lifts those signatures into field dynamics and computes reachable architecture futures.

The formal core gives the theory its working vocabulary: state, operation support, policy, path, horizon, observation, and boundary. That vocabulary makes software evolution an object that can be modeled, measured, and steered.

  • 2-3. ForecastCone Computes bounded reachable futures from supported paths, horizons, reachability regions, and cone narrowing.

  • 3-4. Lifecycle and Closed Loop Closes the evolution loop through repair, migration, contraction, deletion, and posterior updates.
