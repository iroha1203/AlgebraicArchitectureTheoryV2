SFT computational problems

SFT defines problem families instead of remaining a metaphor. Each problem should name inputs, outputs, soundness boundary, and claim level.

  • Field reconstruction Build a bounded `SoftwareFieldEstimate` from artifacts, codebase, review, CI, and incident traces.
  • Operation support inference Estimate which operation families are natural, possible, risky, or low cost in the current field.
  • ConsequenceEnvelope generation Translate PRDs, specs, issues, or AI proposals into reachable path classes and boundary reports.
  • Cone narrowing Synthesize specification, issue, or review constraints that exclude a selected witness family while preserving the intended feature direction.
  • Support safety checking Check whether operation support or governance intervention preserves a selected safe region.
  • Feedback update Update a posterior field model from forecast differences and observed PR, review, CI, incident, or runtime outcomes.
  • AI proposal governance Keep generated proposal support inside a bounded field model and record shortcut witnesses.
  • Lifecycle decision Diagnose repair, migration, contraction, or end-of-life choices from signatures, costs, risk, and capacity.

This contract is what makes the computing-system layer falsifiable. A result that cannot name its boundary is only an explanatory heuristic, not a mature SFT computation.

Representative problem contracts
Problem Input Output Claim level
Field reconstruction DevelopmentField, modeling boundary, artifact / review / CI / incident traces. SoftwareFieldEstimate with evidence categories and unknown remainder. L1 trace-grounded; L4 only after held-out validation.
ConsequenceEnvelope generation Field estimate, artifact, observation boundary, support extractor, and horizon. ConsequenceEnvelope with reachable path classes, affected axes, witness candidates, and missing invariants. L2 if set-valued reachability is defined; L4 after calibrated path and axis validation.
Cone narrowing Target artifact direction, selected witness family, support extractors, and step simulation relation. Constraint, specification, or review intervention that preserves the intended direction while excluding the selected witness family. L2 under support inclusion and step simulation; L4 only with calibration.
AI proposal governance AI proposal stream, prompt / policy boundary, field estimate, theorem boundary, and review / CI traces. Bounded proposal support, shortcut witness report, review / CI intervention, and posterior field update. L1 when trace-grounded; L2 for formal support restriction; L4 for calibrated shortcut detection.

ArchSig-SFT pipeline

The current ArchSig-SFT tooling turns Markdown PRD, Spec, Issue, or AI proposal text into a bounded chain of forecast artifacts. This is a tooling estimate: it preserves source refs, selected support, finite horizon, unknown remainder, and non-conclusions rather than claiming a future outcome.

Input, output, and boundary by stage
Stage Input Output Soundness boundary
ArtifactDescriptor Selected artifact text and source refs. Action class, scope, target regions, missing evidence. Not a ground-truth architecture object.
OperationSupportEstimate Descriptor refs and policy hints. Candidate operation families, known forbidden support, unknown remainder. Not accepted PR history or actual future support.
ForecastConeSkeleton Operation support and bounded horizon config. Finite-support path class candidates. No probability, global risk reduction, or trajectory-level safety claim.
ConsequenceEnvelope Forecast skeleton and reporting boundary. Affected axes, obstruction candidates, missing boundaries, review / CI recommendation. Not a causal proof, Lean theorem, or forecast correctness certificate.

The public ArchSig pages describe the same surface from the tooling side: see the workflow overview, artifact catalog, and tooling roadmap.

PRD-to-ConsequenceEnvelope simulator

The simulator is a way of making SFT computational. It is not a PRD-to-PR predictor; it generates bounded consequence envelopes rather than a single future.

Practical value starts when the tool reports affected axes, witness candidates, missing boundaries, and review or issue decomposition recommendations. Empirical prediction claims depend on later weighting and calibration. Part VI follows the same simulator stages through a detailed Coupon PRD worked example from input artifact to field update.

The current artifact pipeline supports bounded descriptor, support, cone, envelope, and calibration-hook records. It does not support probability weighting, causal forecast, extractor completeness, or framework-specific semantic completeness without additional adapters and datasets.

  • Level 3 minimum A bounded `ForecastCone` or reachable path class is generated from explicit support and horizon.
  • Level 4 practical report `ConsequenceEnvelope` adds signature axes, witness candidates, missing boundaries, and recommendations.
  • Level 6 scientific loop Observed issue, PR, review, CI, or incident outcomes are compared against the report and used for field update.
Simulator maturity ladder
Level Reading Minimal output
0 Explanatory metaphor. Field / force language only.
1 Action classification. PRD action class or descriptor.
2 Support generation. Candidate operation support.
3 Cone generation. Bounded ForecastCone or path classes.
4 ConsequenceEnvelope. Signature axes, obstruction witnesses, and missing boundaries.
5 Weighting. Path class weight, probability, and unknown remainder.
6 Calibration. Observed issue, PR, review, CI, or outcome comparison with field update.
Capability and remaining boundary
Surface Current capability Remaining gap
Markdown PRD / Spec Can be converted through the forecast pipeline into forecast artifacts and a report. Real dataset calibration and held-out outcome validation remain future work.
GitHub Issue / AI proposal Representable as artifact text under the same boundary vocabulary. Native JSON adapters remain outside the current evidence claim.
Review / CI recommendation Report can name missing boundaries, theorem boundaries, and decomposition targets. Recommendation quality is not a theorem and needs empirical feedback loops.

AI-agent governance

SFT reads an AI coding agent as a field participant: artifact interpreter, proposal generator, local pattern amplifier, shortcut generator, review load shifter, and hidden assumption reproducer.

SFT does not prove general AI safety. It bounds proposal support with prompt, policy, theorem boundary, review, CI, shortcut witness reports, and posterior field update.

Allowed operation support is a bounded candidate set for review. Shortcut witness reports name forbidden edges, missing invariants, runtime boundaries, theorem boundaries, and review load without turning AI capability evaluation into forecast correctness.

Review, CI, and type systems

Review, CI, type systems, architecture rules, ownership boundaries, and runtime guards are governance interventions. They shape future operation support and selection policy, not just pass or fail a change.

  • Restrictive Remove unsafe support from the accepted operation set.
  • Redirective Raise shortcut cost and lower lawful path cost.
  • Instrumenting Add observation axes, tests, runtime checks, or evidence records.

Good governance does more than reject unsafe shortcuts: it lowers the cost of lawful paths, finds missing invariants, and writes the result back into field memory.

Lifecycle and benchmarks

SFT includes birth, growth, stabilization, drift, repair, migration, contraction, and end-of-life. End-of-life can be a field reconfiguration decision rather than failure.

Benchmarking is required before calibrated empirical status: PRD-to-issue, issue-to-PR signature, review mediation, incident feedback, AI shortcut, and lifecycle decision benchmarks give the workbench something falsifiable to test.

Lifecycle intervention comparison
Intervention Reading Main comparison axes
Repair Keep the existing field and add local support or invariants. Repair adoption, missing invariant, runtime regression risk, ownership capacity.
Migration Move support and field memory from an old field into a new field. Projection mismatch, dual-run cost, consumer compatibility, rollback path.
Contraction Reduce an unsupported surface and reallocate field capacity. Removed operation families, affected user or runtime boundary, simplified signature axes.
Deletion Remove a field component or subsystem from the lifecycle. Dependent operation loss, data or API retirement boundary, incident avoidance, unknown remainder.
  • PRD-to-Issue Benchmark Checks whether artifact decomposition preserves feature direction, missing evidence, and reviewable boundaries.
  • Issue-to-PR Signature Benchmark Compares forecast items with later signature deltas and affected architecture regions.
  • Review Mediation Benchmark Tests whether envelope items become useful review comments, CI requests, or issue splits.
  • Incident Feedback Benchmark Connects runtime or incident outcomes back to calibration hooks and field update records.
  • AI Shortcut Benchmark Evaluates whether shortcut witnesses and governance constraints catch risky generated paths.
  • Lifecycle Decision Benchmark Compares repair, migration, contraction, and deletion recommendations against later evidence.

Calibration therefore does not reduce SFT to whether a single prediction came true. It also evaluates whether unknown remainder, missing evidence, shortcut witnesses, and obstruction witnesses were classified with the right boundary.

Systems boundary and non-conclusions

Part V describes computing systems that can be built from SFT, but their outputs remain bounded estimates. A simulator is not a PRD-to-PR predictor, an AI governance report is not general AI safety, and benchmark calibration is required before empirical forecast claims are promoted.

  • Input discipline Reports must state the selected artifact, field estimate, observation boundary, support extractor, and horizon.
  • Output discipline Outputs are envelopes, affected axes, witness candidates, missing boundaries, recommendations, and update records.
  • Claim discipline Heuristic path classes, formal reachability, calibrated weights, and deployed governance outcomes remain separate statuses.
