Question

How should a bounded forecast become something a reviewer can actually read?

From cone to map

A ForecastCone is a theoretical object. It may contain many bounded path families, support witnesses, unknowns, and boundary-relative assumptions. A reviewer cannot work directly with the whole cone. They need a map: what kinds of paths are open, what architecture axes may move, what obstructions may appear, and what is still outside the model.

That map is the ConsequenceEnvelope. It is not a summary in the casual sense. It is a loss-aware projection. It deliberately throws away detail while preserving the references and warnings needed to keep the projection honest.

This makes the envelope one of the most practical objects in SFT. The cone is how the theory thinks. The envelope is how a developer, reviewer, or architect can decide what to ask next.

Report projection

The formal object is the cone. The practical object is the envelope. A reviewer rarely needs every path in a cone family. They need to know what path classes matter, which axes may move, which obstruction candidates appear, which invariants are missing, and what question governance must answer before accepting the work.

Each field in the envelope exists to prevent a common failure. Path classes keep reviewers from seeing only one implementation story. Affected axes keep architecture movement visible. Obstruction candidates prevent "looks fine" from replacing evidence. Missing invariants name what the artifact did not say. Unknown remainder keeps unobserved evidence from being silently treated as safe.

Definition 2.13. A ConsequenceEnvelope is a loss-aware report projection from selected forecast-cone evidence to a developer and reviewer readable artifact.

SFT status report projection Source: SFT section 12 and ArchSig SFT artifact boundary.

Loss-aware reading

The projection is intentionally one-way. An envelope summarizes selected cone evidence, but a reader cannot reconstruct the unique cone family from the report. That is why the report must preserve references, boundaries, and unknowns rather than compressing everything into a pass/fail label.

This loss is not a defect. It is what makes the object usable. A perfect cone dump would be unreadable. A one-line conclusion would be dishonest. The envelope lives between those extremes: small enough for review, explicit enough to preserve the boundary of the computation.

Review habit. Read an envelope as a map of modeled consequences, not as proof that unlisted consequences are absent.

What reviewers can do with it

A good ConsequenceEnvelope changes review from taste to inquiry. Instead of asking only whether the proposed diff looks reasonable, a reviewer can ask whether a path class is missing, whether the affected axes match the artifact, whether an obstruction witness needs a test, whether governance should split the issue, or whether an unknown remainder is too large for the claim being made.

The envelope also gives AI governance a surface. If an agent proposes a locally plausible shortcut, the envelope can record the shortcut path family, the missing invariant, the policy question, and the calibration hook for the eventual outcome. That turns review from isolated judgment into field update.

Boundary

Non-conclusion 2.14. A ConsequenceEnvelope cannot be inverted into a unique ForecastCone family and does not prove that the reported path classes will occur.
