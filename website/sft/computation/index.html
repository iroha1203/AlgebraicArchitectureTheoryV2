Making Software Evolution Computable

Question

How can a field too large to know completely still support useful computation about reachable architecture futures?

Bounded computation

Part 1 described the development organization as a field. That object is intentionally larger than a repository graph. Part 2 asks how to cut a finite, explicit, reviewable slice from that field without pretending that the slice is the whole system.

The result is a bounded computation problem. A selected field estimate records observed evidence and unknown remainder. AAT supplies local architecture algebra for the projected architecture state. ArchSig supplies observation artifacts. SFT adds operation support, policy, path semantics, forecast cones, reviewer-facing envelopes, and calibration hooks.

Definition 2.0. A computable SFT problem is a selected field estimate together with observation axes, admissible operation families, support and policy assumptions, a horizon, and an explicit boundary for unknown or unmodeled evidence.

Computation pipeline

Boundary

Non-conclusion 2.0. A computable slice is not complete reconstruction, a ForecastCone is not deterministic prediction, an ArchSig report is not a Lean theorem, and calibration is not obtained from a report format alone.
