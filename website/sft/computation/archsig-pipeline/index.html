Question

What does the implemented tooling surface produce, and what does it deliberately not conclude?

Artifact chain

The ArchSig-SFT pipeline is where the theory touches files. It does not attempt to implement the whole SFT formal core. It creates bounded artifacts that can be read by humans, compared across runs, and linked to later outcomes.

The pipeline keeps artifact refs, support refs, horizon, measurement boundary, theorem precondition status, unknown remainder, and non-conclusions visible. That shape makes it useful for review and later calibration because the report can be joined with observed PR, review, CI, incident, and outcome references.

This is a practical compromise. The tool can preserve the shape of a forecast without claiming that it has enumerated every reachable field path. It can record support candidates without proving that the support relation is complete. It can produce a ConsequenceEnvelope without claiming that the envelope is the future.

The pipeline is therefore an interface between research ambition and daily review. It gives reviewers a concrete artifact to inspect while keeping enough structure for later calibration. A future benchmark can ask whether envelope items matched observed outcomes because the refs were preserved from the beginning.

The important word is "surface." The pipeline does not expose every internal object SFT may eventually formalize. It exposes the parts that can be safely shared in review: artifact identity, support estimate, path-class skeleton, envelope items, claim boundaries, and calibration hooks. That surface is enough to make future-oriented review repeatable without pretending the tool has solved the theory.

Definition 2.21. The ArchSig-SFT pipeline is the implemented tooling surface that projects selected artifact input into bounded forecast and review artifacts while preserving evidence boundaries.

Claim boundary

The pipeline output can say: under this artifact, this support estimate, these selected axes, and this horizon, these path classes and affected boundaries should be reviewed. It cannot say: this future will happen, this theorem is proved, this organization is safe, or this forecast has been calibrated.

That boundary is what makes the tooling useful rather than misleading. A review artifact should sharpen questions: which invariant is missing, which axis is affected, which path class is risky, and what evidence would update the field after the PR. It should not collapse those questions into a green or red label.

Computation habit. Use tool output as a structured review surface, not as an oracle.

This also keeps ArchSig aligned with the AAT / SFT interface. A report can carry AAT observables and SFT forecast surfaces side by side, but the artifact must keep their claim levels separate. A theorem boundary note, a support estimate, and an envelope item should be readable together without being collapsed into one kind of truth.

For command and artifact details, read ArchSig commands and ArchSig artifacts.

Boundary

Non-conclusion 2.22. ForecastConeSkeleton and ConsequenceEnvelope artifacts do not certify a Lean theorem, forecast correctness, causal effect, architecture lawfulness, or global safety.
