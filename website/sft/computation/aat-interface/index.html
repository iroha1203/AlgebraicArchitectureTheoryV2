Question

What can formal architecture algebra contribute to a field theory without becoming a forecast engine?

One-way dependence

AAT is a local theory of architecture objects, operations, laws, signatures, and witnesses. It can be studied without organizations, PRDs, review queues, AI agents, or calibration datasets. SFT is larger and less pure: it studies how those local objects appear inside a development field.

The dependence therefore runs one way. SFT may use AAT as a coordinate system and as a source of theorem preconditions. AAT theorems do not become SFT forecast theorems unless an additional field model, support relation, horizon, and bridge theorem have been supplied.

This is a productive asymmetry. AAT gives SFT local law: what an operation preserves, which witness marks an obstruction, which signature axis can be measured. SFT gives those local objects a field context: who can propose the operation, which artifacts made it visible, which policy selected it, and which future paths remain reachable afterward.

The interface is therefore not a merger of theories. It is a disciplined translation layer. AAT keeps its mathematical status. SFT uses that status as a premise inside a larger computation over field-shaped evolution.

This is what prevents Part 2 from collapsing into either pure formalism or pure process commentary. Without AAT, SFT would have no local algebraic coordinates for architecture. Without the SFT boundary, AAT would be asked to explain proposal pressure, organizational selection, and calibration outcomes it was not designed to explain. The interface lets each theory remain strong in its own domain.

The practical reading is simple: when a field step touches architecture, SFT can ask whether the projected local operation satisfies an AAT premise. That answer can constrain the path. It cannot, by itself, tell us which path the field will select next.

This is why the interface belongs in Part 2, not in an appendix. The moment SFT tries to compute reachable futures, it needs local architectural coordinates. But the moment it uses those coordinates, it must also prevent theorem status from leaking into forecast status. The interface is the guardrail that lets the calculation proceed.

How SFT uses AAT

architecture projection
The selected part of a field estimate read as an AAT-style architecture object.
observable coordinate
A signature axis or witness family that tooling can report under a measurement boundary.
local transition law
An operation-level premise that can constrain one field step, not every future path.
admissibility premise
A theorem precondition that must be checked before a formal AAT claim can be cited.

Definition 2.3. The AAT / SFT interface is the boundary where local architecture algebra is projected into field computation while preserving the distinction between formal theorem status and bounded forecast status.

The key discipline is to avoid promotion by vocabulary. An AAT invariant does not become an organization-level guarantee. A zero signature axis does not make unmeasured axes safe. A lawful local transition does not prove that a policy will keep selecting lawful transitions. SFT can use AAT precisely because it does not ask AAT to say more than it proves.

The reward for this discipline is composability. SFT can use local theorem premises inside larger field paths, compare paths that preserve or violate those premises, and then project the result into forecast and review artifacts. The local theorem remains local; the field computation remains bounded.

Computation habit. Use AAT as local law inside the field, not as a shortcut from local theorem to future forecast.

Boundary

Non-conclusion 2.4. An AAT theorem can justify a local architectural claim under its premises. It does not by itself predict which operation a team will select, which PR will merge, or which field path will be reached.

For the formal theorem boundary, read AAT Status.
