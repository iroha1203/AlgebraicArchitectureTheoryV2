Can governance turn software evolution into a learning trajectory?

Governance

Governance is the field mechanism that turns feedback into future support. A review rejection records the evidence a path failed to carry. A CI failure names an observation axis. An incident makes a hidden invariant expensive. A postmortem turns runtime pain into tests, ownership, review rules, or new constraints.

Vocabulary

Governance

The field update mechanism by which review, CI, type checking, incidents, runtime observation, and postmortems change future operation support.

Acts through
Rules, checks, review precedents, incident memory, owner routes, tests, and observation axes.
Reading
Ask what the next proposal can see, select, avoid, or justify because governance remembered it.
Vocabulary

Governance feedback

Review, CI, type checking, runtime observation, incident response, and postmortem evidence when they are preserved as memory for later support and policy.

Role
Changes what the next proposal can see, select, or avoid.
Field effect
Adds constraints, tests, observation axes, ownership routes, or review rules.

This is the closed-loop heart of SFT. A field proposes futures, governance selects among them, observation reveals what the field failed to see, and the result becomes memory for the next proposal. A healthy architecture process is a learning system that changes the support of future changes.

  • Restrict Remove unsafe support and make protected paths explicit.
  • Redirect Raise shortcut cost and lower the cost of the lawful path.
  • Instrument Add tests, checks, telemetry, or runtime observations that make drift visible.
  • Learn Record review, CI, and incident outcomes as memory for the next field state.

Architecture futures

SFT studies futures as reachable path families. A future includes the field path, the architecture projection path, the signature movement, obstruction candidates, and the boundaries of what remains unobserved.

Vocabulary

Architecture future

A reachable family of field and architecture trajectories, together with signature movement, obstruction candidates, and the boundary of what was observed.

Object
Reachable path family with observed and unobserved boundaries.
Observation
Read through signature trajectory and forecast boundary.

The cone language keeps several futures visible at once: which changes are now closer, which have moved farther away, which architecture regions are likely to be touched, and which obstructions are likely to matter if the path continues.

Signature trajectory

Endpoint safety leaves path information behind. A path can end with an acceptable signature while making unsafe excursions along the way. SFT keeps trajectory language: what changed, in which order, on which axes, and with which evidence boundary.

This is where ArchSig becomes useful as an observation lens. It keeps a multi-axis record of movement: dependency direction, boundary preservation, semantic obstruction, runtime exposure, review evidence, and the places that remain outside the current observation boundary.

before

What did the field support?

Which operations were cheap, visible, reviewable, or hidden before the artifact arrived?

during

Which path was selected?

Which PR shape appeared, which invariants were preserved, and where did obstructions surface?

after

What did the field learn?

Which tests, review rules, owner routes, incident memories, or agent policies changed future support?

next

Which futures moved?

Which good paths became easier, which shortcuts became expensive, and which unknowns remain active?

Process feedback

Waterfall and agile can be read as different feedback geometries. Waterfall concentrates field updates at large phase boundaries: requirements, design, implementation, verification, release. Agile makes smaller field updates frequent through iteration, review, tests, demos, and retrospectives.

SFT reads both as governance designs. The question is how quickly evidence returns to the field, which observations are preserved as memory, and which future changes become easier after each cycle.

Vibe coding makes the pressure sharper. An improvised prompt can generate a cone-like spread of possible changes, and the resulting force can bend the field before the organization has named the invariant it wants to protect.

Process form Field update shape Governance reading
Waterfall Large updates at phase boundaries. Strong upfront memory, slower correction when observations arrive late.
Agile Small updates through repeated cycles. Fast feedback, with architecture memory depending on tests, review, and retrospectives.
Vibe coding Large force can erupt from an ambiguous prompt or PRD. Governance preserves stronger memory so generated speed keeps architectural direction.

Bridge to computation

Definition 1-4. Governance feedback is the field update pressure created by review, CI, runtime observation, incident response, and postmortem evidence. Architecture futures are the reachable field and architecture trajectories shaped by that updated memory.

SFT status conceptual core Part 2 introduces bounded path and cone objects for this reading.

Proposition 1-4. A governance outcome changes future architecture possibilities when it is preserved as field memory that alters support, policy, or observation.

Proof idea. A review, CI result, incident, or postmortem can add a missing invariant, forbid a shortcut, introduce a test, change ownership, or add an observation axis. Each of these changes the selected field estimate used by later support and path computation. Thus the feedback affects reachable futures by changing the state from which later paths are generated.

Example 1-4. A rounding incident after coupon launch can become a runtime observation, a missing invariant, a test, a review checklist item, and a future support constraint. The next coupon-related path begins from a different field.
