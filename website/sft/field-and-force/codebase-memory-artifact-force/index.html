What field is already shaping operation support before an artifact becomes a diff?

Development field

A field is the development situation that makes some architecture operations natural: codebase memory, PRDs, issues, developers, AI agents, ownership, review, CI, incidents, and lifecycle pressure acting together.

A force is the action of an artifact on that field. A PRD, issue, spec, review comment, incident report, or AI proposal can change what is named, what is hidden, which path is cheap, and which invariant becomes visible enough to protect.

In this sense, SFT is not a theory of the codebase alone. It is a theory of software evolution as shaped by artifacts, organization, tooling, feedback, and agents. AAT supplies the local algebra of architecture; SFT asks how the surrounding field makes future paths easier or harder to reach.

Definition 1.0. A development field is the selected organizational and technical state from which SFT reads supported software-evolution paths. AAT appears inside it as the local algebra of the architecture projection.

SFT status conceptual core Part 2 turns selected field readings into bounded computable objects.

Memory

A repository remembers more than syntax. It remembers old requirements, accepted compromises, abandoned migrations, ownership boundaries, incident repairs, review precedents, and local patterns that became easy to copy. Those memories keep acting because later work reads them as available structure.

SFT calls this codebase as field memory. The memory is not passive. It shapes which operations look normal, which invariants are easy to preserve, and which shortcuts can be copied without being named.

Vocabulary

Field memory

The remembered development state that makes later operations feel natural: code examples, review precedents, incidents, ownership routes, migrations, tests, and missing tests.

Role
Turns past changes into present operation support.
Reading
Ask what the codebase teaches the next engineer or AI agent to copy.

This is one reason architectural debt can feel stubborn even when everyone agrees with the desired design. The problem is not only that the code has the wrong shape. The field has learned a path. It has examples, review habits, missing tests, owner expectations, and short-term success stories that make the old path cheap to repeat.

  • Sediment Past decisions become local examples that future changes reuse.
  • Default path The field makes one operation family feel easier than its alternatives.
  • Hidden invariant A rule exists in practice but has not been made visible to artifacts, tests, or review.
  • Agent prompt The codebase and its documents become the context from which AI agents infer what to do next.

Artifact force

Artifacts act on that memory. A PRD can make one decomposition obvious. An issue can hide a boundary. A design memo can name a missing invariant. A review comment can become precedent. An incident report can add an observation axis that did not exist before.

The word force is useful here because the artifact need not determine a single change. It changes the shape of the possible next changes.

Vocabulary

Artifact force

The field-level action by which an artifact changes what can be proposed, seen, reviewed, selected, or remembered.

Carrier
PRD, issue, spec, review comment, incident report, AI proposal.
Effect
Candidate field updates, support shifts, policy shifts, and observation changes.

In ordinary process language, artifacts are often treated as coordination aids: a PRD explains the feature, an issue tracks work, a review comment asks for a revision, and an incident report records what happened. SFT gives them a stronger role. Each artifact can change the support of the field. It can make a lawful path easier to discover, a shortcut harder to justify, or a missing invariant visible enough to become a future test.

Mathematical spine

The field language is meant to carry mathematics, not remain a metaphor. The first move is to separate the broad development field from the computable slice that SFT can reason about. That slice keeps architecture projection, observed signature, support, policy, constraints, governance, history, and incoming artifacts in one state.

Vocabulary

SoftwareField

The computable slice of a broader development field. It is the state from which SFT reads supported operations and reachable architecture futures.

Projection
arch : SoftwareField -> ArchitectureObject
Contents
History, signature record, support, policy, constraints, observation, governance, and artifact inputs.

Force then becomes a set-valued action. An artifact does not have to choose a single future. It can produce candidate field updates, and each update changes the support and policy from which future paths are drawn.

This gives SFT its first computational question. Given a field and an artifact, which supported operation families become closer, which become harder, and which architecture projection will those families touch?

Coupon PRD

A coupon PRD in a checkout system is the smallest running example. The request can open a lawful discount-policy path, a payment-adapter shortcut, a UI-only drift, or a rounding semantic obstruction. The PRD has not changed the code, but it has changed the field of plausible implementations.

Artifact wording Field effect Likely PR shape
"Allow coupons at checkout." Feature is named, but ownership and invariants are left implicit. Nearby patch, adapter modification, or UI-only state.
"Coupons are discount policies composed before tax and payment authorization." Boundary, order, and semantic responsibility become visible. Policy insertion with tests for composition and rounding.
"Refunds and cancellations must preserve the original discount semantics." Runtime and lifecycle observations enter the field before implementation. Checkout, payment, refund, and accounting paths are reviewed together.

Reading habit. Do not ask only what the artifact says. Ask which operation families it makes visible, cheap, hidden, or reviewable.

Field reading

Definition 1-1. Codebase memory is the sedimented field state through which past requirements, decisions, repairs, and practices influence future operation support. Artifact force is the field-level action by which artifacts reshape that support, policy, and observation boundary.

SFT status conceptual core Part 2 turns selected support and observation boundaries into computable objects.

Proposition 1-1. If two codebases have the same selected module graph but different field memory or artifact context, SFT may assign different operation support to their next changes.

Proof idea. Operation support is a field-relative object, not a function of the module graph alone. Changing review precedents, incident memory, ownership routes, or PRD constraints changes the input to the support estimate while leaving the selected graph fixed. Therefore equality of the graph does not imply equality of supported operation families.

Example 1-1. A checkout codebase with a recorded discount invariant and review examples for policy insertion supports a different coupon path than a visually similar codebase whose only local examples patch payment adapters. The artifact says "coupon" in both cases; the field makes different next moves natural.
