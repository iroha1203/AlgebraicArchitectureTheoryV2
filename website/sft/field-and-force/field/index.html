Can we model the dynamics already acting on a codebase before the next change begins?

Development field

A field is the development situation that makes some architecture operations feel natural. It includes codebase memory, PRDs, issues, developers, AI agents, ownership, review, CI, incidents, and lifecycle pressure acting together.

SFT treats that situation as part of software engineering itself. AAT supplies the local algebra of architecture; SFT reads the surrounding field that makes future paths easier to find, harder to justify, cheaper to repeat, or visible enough to review.

Vocabulary

Field

The selected development situation around a codebase: the technical and organizational state from which future changes become supported, costly, visible, or natural.

Includes
Codebase memory, artifacts, agents, ownership, review, CI, incidents, governance, and lifecycle pressure.
Reading
Ask what the current development situation makes easy to propose, repeat, review, or notice.

Definition 1.0. A development field is the selected organizational and technical state from which SFT reads supported software-evolution paths. AAT appears inside it as the local algebra of the architecture projection.

SFT status conceptual core Part 2 turns selected field readings into bounded computable objects.

Memory

A repository remembers more than syntax. It carries old requirements, accepted compromises, abandoned migrations, ownership boundaries, incident repairs, review precedents, and local patterns that became easy to copy. Later work reads that memory as available structure.

SFT calls this codebase as field memory. The memory shapes which operations look normal, which invariants are easy to preserve, and which shortcuts can be copied without being named.

Vocabulary

Field memory

The remembered development state that makes later operations feel natural: examples, review precedents, incidents, ownership routes, migrations, tests, and missing tests.

Role
Turns past changes into present operation support.
Reading
Ask what the codebase teaches the next engineer or AI agent to copy.

This is why architectural debt can feel stubborn even when everyone agrees with the desired design. The field has learned a path. It has examples, review habits, missing tests, owner expectations, and short-term success stories that make the old path cheap to repeat.

  • Sediment Past decisions become local examples that future changes reuse.
  • Default path The field makes one operation family feel easier than its alternatives.
  • Hidden invariant A rule exists in practice but has not been made visible to artifacts, tests, or review.
  • Agent prompt The codebase and its documents become the context from which AI agents infer what to do next.

Mathematical spine

The field language carries mathematics. SFT separates the broad development field from the computable slice it can reason about. That slice keeps architecture projection, observed signature, support, policy, constraints, governance, history, and incoming artifacts in one state.

Vocabulary

SoftwareField

The computable slice of a broader development field. It is the state from which SFT reads supported operations and reachable architecture futures.

Projection
arch : SoftwareField -> ArchitectureObject
Contents
History, signature record, support, policy, constraints, observation, governance, and artifact inputs.

Field reading

Definition 1-1. A development field is the selected organizational and technical state around a codebase from which SFT reads supported software-evolution paths.

SFT status conceptual core Part 2 turns selected field slices into bounded computable objects.

Proposition 1-1. If two codebases have the same selected module graph but different field memory, SFT may assign different operation support to their next changes.

Proof idea. Operation support is field-relative. Review precedents, incident memory, ownership routes, and local defaults all enter the support estimate, even when the selected module graph stays fixed. The same graph can therefore carry different supported operation families.

Example 1-1. A checkout codebase with a recorded discount invariant and review examples for policy insertion supports a different coupon path than a visually similar codebase whose only local examples patch payment adapters. The selected graph can match; the field still makes different next moves natural.
