Can we interpret the change a pull request exerts on a codebase as force?

Force

A force is a field-level pressure that changes how a codebase is prepared to move. A pull request makes one path concrete. It gives reviewers a shape to accept, reject, amend, remember, or copy.

Vocabulary

Force

The field-level pressure that changes which software evolution paths become nearby, costly, visible, reviewable, or natural.

Acts on
Operation support, selection policy, observation boundaries, and field memory.
Reading
Ask how a new artifact changes the direction the codebase is prepared to move.

Artifact force

Other artifacts carry the same kind of pressure through ordinary software work: a PRD makes one decomposition obvious, an issue hides a boundary, a design memo names a missing invariant, a review comment becomes precedent, and an incident report adds a new observation axis.

SFT uses force to describe this change while preserving many possible next diffs. The artifact reshapes the space of possible next changes.

Vocabulary

Artifact force

The field-level action by which an artifact changes what can be proposed, seen, reviewed, selected, or remembered.

Carrier
PRD, issue, spec, pull request, review comment, incident report, AI proposal.
Effect
Candidate field updates, support shifts, policy shifts, and observation changes.

In ordinary process language, artifacts coordinate work: a PRD explains the feature, an issue tracks work, a pull request proposes a concrete path, a review comment asks for a revision, and an incident report records what happened. SFT gives them a stronger role. Each artifact can change field support. It can make a lawful path easier to discover, a shortcut harder to justify, or a missing invariant visible enough to become a future test.

Mathematical spine

The force language carries mathematics. Force becomes a set-valued action: an artifact produces candidate field updates, and each update changes the support and policy from which future paths are drawn.

This gives SFT a computational question. Given a field and an artifact, which supported operation families move closer, which become harder, and which architecture projection will those families touch?

Coupon PRD

A coupon PRD in a checkout system is the smallest running example. The request can open a lawful discount-policy path, a payment-adapter shortcut, a UI-only drift, or a rounding semantic obstruction. Before code changes, the PRD has already changed the field of plausible implementations.

Artifact wording Field effect Likely PR shape
"Allow coupons at checkout." Feature is named, but ownership and invariants are left implicit. Nearby patch, adapter modification, or UI-only state.
"Coupons are discount policies composed before tax and payment authorization." Boundary, order, and semantic responsibility become visible. Policy insertion with tests for composition and rounding.
"Refunds and cancellations must preserve the original discount semantics." Runtime and lifecycle observations enter the field before implementation. Checkout, payment, refund, and accounting paths are reviewed together.

Reading habit. Read the artifact by the operation families it makes visible, cheap, hidden, or reviewable.

Force reading

Definition 1-2. Artifact force is the field-level action by which artifacts reshape operation support, policy, and observation boundary.

SFT status conceptual core Part 2 turns selected support and observation boundaries into computable objects.

Proposition 1-2. If two artifacts name the same feature but expose different boundaries, responsibilities, or observations, SFT may assign different operation support to their resulting fields.

Proof idea. Operation support is field-relative. Artifact wording changes the candidate field update: it can expose a lawful path, hide a boundary, add an observation axis, or make a shortcut costly to justify.

Example 1-2. The artifact "allow coupons at checkout" opens a broader cone than "coupons are discount policies composed before tax and payment authorization." The second artifact applies more specific force to operation support.
