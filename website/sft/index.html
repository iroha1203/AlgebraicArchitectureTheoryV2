What if the object of software engineering is not only the current codebase, but the field that shapes the next changes?

Thesis

Software Field Theory begins from a simple observation: software does not merely change. It evolves inside a field of artifacts, habits, agents, review systems, operational memory, and lifecycle pressure. That field does not determine one future, but it changes which futures become cheap, visible, repeatable, reviewable, or almost unthinkable.

A PRD is not just text. It can make one issue decomposition feel natural and another disappear. A review rule is not just a gate. It can raise the cost of shortcuts and lower the cost of lawful paths. A codebase is not just state. It is memory that teaches developers and AI agents what kind of move looks nearby.

SFT makes that moving object computable. Its target is not a single predicted PR. Its target is the shape of reachable architecture futures under a selected field model, operation support, policy, observation boundary, and horizon.

This is the central claim: software evolution has a field structure. Requirements, incidents, ownership, tests, reviews, prompts, and old code exert force by changing which operations are supported and which paths are selected. SFT gives that force a computational reading instead of leaving it as anecdote.

Field reading

The field is the selected development reality around a codebase: requirements, issues, design notes, ownership, review, CI, runtime incidents, AI policy, old workarounds, and the patterns everyone has learned to copy. SFT reads these as active structure. They shape operation support: the set of changes that are plausible, low-friction, legitimate, and likely to be selected next.

This is why two repositories with nearly identical dependency graphs can have different futures. One field records invariants in review and turns incidents into tests. Another field leaves hard boundaries as folklore and makes local patching the cheapest move. The code may look similar, but the next possible moves have changed.

Core intuition. SFT studies the pressure that exists before the next diff: what the present field is making easy, hard, visible, hidden, repeatable, or costly.

Three movements

The public reading path is intentionally three large movements. The small formal and tooling notes remain available, but the main experience should read as a theory of software evolution, not as a directory of fragments.

  • 1. The Field Three chapters on codebase memory and artifact force, agents and organization, and governance feedback as the source of architecture futures.
  • 2. The Computation Three chapters on computable slices and observation, support and ForecastCone semantics, and ConsequenceEnvelope calibration.
  • 3. The Intervention Three chapters on governance and attractor engineering, AI proposal governance, and lifecycle / closed-loop evolution.

Mathematical guardrail

SFT is allowed to speak in large language because its formal core keeps the claims shaped. The prose can say that artifacts apply force, that codebases remember, and that good futures can be made natural. The mathematics then asks which support relation, which step semantics, which horizon, which observation axes, and which theorem boundary license the statement.

Lean is the discipline behind the rhetoric. A field is not a loose metaphor; it is represented through selected states, operations, support, paths, projections, and boundaries. A force is not a causal theorem by itself; it becomes a change to support, policy, observation, or feedback under a stated model. The website can carry the intuition because the formal core keeps the object typed.
