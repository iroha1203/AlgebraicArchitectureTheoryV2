Software Field Theory

Software Field Theorem

SFT makes software evolution computable by treating the whole development organization--requirements, code, reviews, agents, incidents, and governance--as a field of forces that pulls a codebase toward some futures and away from others.

Why now

AI has made code generation fast. A requirement, prompt, or review comment can now become a patch before the organization has fully understood what future that patch makes easier.

But software changes are not isolated events. Each change alters tests, dependencies, ownership, review habits, examples, incidents, and the paths that developers and agents will copy next. The faster change becomes, the more important it becomes to study the conditions under which future changes are made.

Object

This brings software evolution back to the center of the theory. Not only the current codebase. Not only the next diff. The object is the field that shapes which futures become cheap, visible, repeatable, reviewable, or nearly unthinkable.

SFT studies that object. It treats requirements, issues, reviews, CI, incidents, prompts, ownership, and codebase memory as forces that shape operation support: what changes can be made, selected, justified, repeated, and observed.

Construction

SFT is built on a stack. AAT gives architecture a local algebra: components, operations, invariants, and theorem boundaries. ArchSig makes parts of that algebra observable as bounded signatures. SFT then studies how fields move those signatures through reachable architecture futures.

The formal core keeps the language disciplined: state, operation support, policy, path, horizon, observation, and boundary. The overview gives readers enough structure to see software evolution as an object that can be modeled, measured, and steered.

Chapters

The main path has three movements. Read them as a progression: first the field that shapes change, then the computation that bounds reachable futures, then software evolution as an engineered loop that steers the field.

  • 1-3. Feedback and Futures How review, CI, operational feedback, and governance shape the field before the next diff.
  • 2. The Computation How slices, support, ForecastCone semantics, ConsequenceEnvelope projection, and calibration make evolution computable.
  • 2-1. Slice and Observation How SFT selects a bounded slice of development reality and states what can be observed.

  • 2-3. Envelope and Calibration How consequence envelopes, traces, unknown remainder, and feedback connect the model to evidence.
  • 3. Software Evolution How governance, attractor engineering, AI proposal policy, and lifecycle feedback make better futures easier to reach.
  • 3-1. Governance and Attractor How review, CI, type systems, stable regions, and field shaping pull evolution toward better basins.
  • 3-2. AI Proposal Governance How AI-generated proposal support is bounded by theorem boundaries, shortcut witnesses, review, and CI.
  • 3-3. Lifecycle and Closed Loop How repair, migration, contraction, deletion, and posterior updates close the evolution loop.
