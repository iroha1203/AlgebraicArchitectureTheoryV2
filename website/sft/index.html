Computational thesis

SFT makes software evolution a computable object. It studies how codebase memory, artifacts, practices, agents, governance, feedback, and lifecycle pressure reshape the operation space from which future architecture paths are drawn.

The central move is to treat reachable architectural futures as bounded objects of calculation: reconstruct a field slice, infer supported operations, shape support and policy, generate forecast cones, project them into consequence envelopes, and feed observations back into the field.

Core pipeline

The core pipeline is the positive computational reading of SFT. Artifacts and feedback do not select one future directly; they reshape support, policy, observations, and governance so that a family of reachable paths can be calculated and reported.

  • Field reconstruction Build a computable slice from codebase memory, artifacts, review, CI, incident, ownership, and runtime traces.
  • Attractor engineering Make desirable paths natural, cheap, visible, and reviewable while making shortcuts unsupported, costly, or observable.
  • Envelope generation Turn cone families into reviewer-facing path classes, affected axes, obstruction candidates, missing invariants, and governance questions.

Main contributions

  • Software evolution as a computable object SFT treats reachable architecture futures as bounded objects of calculation under an explicit model, support relation, policy, horizon, and observation boundary.
  • Field memory and artifact-mediated change Codebases are read as accumulated implementation, requirements, review, incident, ownership, workaround, and tooling memory; artifacts change operation support instead of selecting a single deterministic next state.
  • ForecastCone and ConsequenceEnvelope A ForecastCone is a set of reachable paths under stated support and horizon. A ConsequenceEnvelope is the boundary-aware report that summarizes selected cones, affected axes, witness candidates, and non-conclusions.
  • Governance as field intervention Review, CI, type checking, issue decomposition, AI policy, and runtime feedback are modeled as interventions that can reshape supported paths and update field memory.

AAT / ArchSig / SFT

SFT uses AAT and ArchSig, but it keeps their claim levels separate. AAT is the local algebra of architectural change. ArchSig is the observation layer for signatures and witnesses. SFT is field-shaped computation over future architectural trajectories under stated support, policy, and forecast boundaries.

  • AAT Local algebra of architectural change.
  • ArchSig Observation layer for signatures and witnesses.
  • SFT Field-shaped computation over future architectural trajectories.
  • AAT theorem A local theorem claim under selected universe, assumptions, coverage, exactness, and non-conclusions.
  • ArchSig tooling estimate A measurement or report with evidence boundaries. It does not become a Lean proof or ground-truth architecture object.
  • SFT forecast claim A bounded reachability or forecast statement whose status depends on the field model, support relation, calibration, and observation boundary.

Worked example preview

The running example is a coupon PRD that touches checkout and payment. SFT reads the PRD as an artifact action: it may make several operation families look available, not just request a feature endpoint. The useful question is which paths become supported, which boundaries are missing, and which review or specification interventions narrow the selected cone.

  • Coupon PRD case study Shows how a vague feature artifact becomes path classes, missing invariants, a ConsequenceEnvelope, and review governance questions.
  • PRD-to-ConsequenceEnvelope simulator Frames the eventual tooling problem: artifact, field estimate, support relation, horizon, evidence, and claim level must all be named.

Claim boundary

SFT claims are bounded by the stated field model, support relation, horizon, evidence boundary, and claim level. A ForecastCone names a set of reachable paths, not a point prediction; a ConsequenceEnvelope is a boundary-aware report, not a causal proof; and ArchSig output remains measurement evidence, not a Lean theorem.

The glossary fixes terminology boundaries, and the status page gives the full claim-level map.
