Can people, AI agents, and organization structure be modeled as recurrent pressure on architecture?

Field participants

Human developers and AI agents are field participants. They interpret artifacts, read local patterns, estimate review cost, generate proposals, and amplify what the field already makes plausible. Their choices are creative, but they are not made in a vacuum.

Vocabulary

Field participant

A human or AI agent that acts inside the selected field by reading artifacts, estimating cost, generating proposals, or reviewing paths.

Human
Brings judgement, ownership memory, review context, and operational fear.
AI agent
Amplifies the patterns and instructions made visible by code and artifacts.

AI makes this especially visible. A codebase is also a prompt. If the field makes shortcuts local and invariants implicit, an agent can reproduce that pattern quickly. If the field records boundaries and examples of lawful extension, the same agent can amplify better structure.

This is why SFT treats AI governance as more than permission control. A policy can forbid a class of changes, but the field still teaches the agent what looks normal. The deeper question is whether the codebase, PRD, issue, tests, review examples, and architecture notes make the good path the easiest path to propose.

A codebase is not only an implementation. In the AI era, it is also a training surface and a prompt for the next change.

Conway through SFT

Conway's Law is the observation that organizations tend to produce systems whose structure resembles their communication structure. If four groups must coordinate through four communication channels, the system often grows four visible seams. If a feature requires three teams to negotiate, the resulting architecture often carries that negotiation cost.

The usual reading is descriptive: look at the org chart and you can often recognize its shadow in the software. SFT turns that into a mechanism. The organization does not directly command an architecture. It changes which work is easy to formulate, who must approve it, where handoffs occur, which reviews are cheap, and which changes are repeatedly selected.

In SFT terms, Conway's Law is field pressure. Communication paths, ownership boundaries, approval flows, and review routes reshape operation support and policy. Over time, repeated low-friction operations become recurrent PR shapes, and those recurrent PR shapes settle into architecture.

Vocabulary

Organization field

The ownership, communication, approval, review, and agent context that changes which operation families are repeatedly selected.

Mechanism
Changes issue decomposition, review cost, coordination cost, and recurrent PR shape.
Conway reading
Architecture reflects repeated low-friction operations, not a one-time copy of an org chart.
Organization pressure SFT reading Architectural effect
A team owns checkout and another owns payment. Cross-boundary changes have higher coordination cost. Checkout/payment boundaries become stable, or shortcuts appear at the seam.
Reviewers are assigned by technical layer. Layer-local patches become cheaper than capability-wide changes. The architecture tends to preserve layer-shaped PRs.
AI agents implement from local examples. Visible local patterns become low-cost operation support. Good examples or shortcuts are amplified faster.

The important move is to replace a static resemblance story with a dynamic one. The organization does not stamp the org chart onto the codebase in one event. It changes which issues are easy to write, which PRs are easy to assign, which review conversations happen quickly, and which changes require expensive coordination. Architecture absorbs those repeated choices.

Recurrent path shape

SFT does not need to claim that organization determines architecture. It is enough to observe that repeated operation shapes matter. A team boundary can make certain cross-cutting changes expensive. A review route can make boundary-preserving work easier to justify. A product-capability organization can make capability-preserving changes more natural.

layer split

Frontend / Backend / Data / Infra

Issues and approvals tend to follow technical layers, so cross-cutting capability work can become coordination-heavy.

capability split

Search / Checkout / Billing

Feature work can preserve product boundaries more naturally, while shared platform changes need explicit governance.

agent split

Human review / AI implementation

The proposal stream becomes faster, so artifacts and tests must carry more of the architectural memory.

incident split

On-call / Product / Platform

Operational pain changes which invariants become visible and which regions receive future protection.

Support and policy

The mathematical object behind organization pressure is not an org chart. It is a support and policy structure over operations. Support says which operations are available in the current field. Policy says which of those operations are cheaper, easier, more reviewable, or more likely to be selected.

Vocabulary

Support and policy

Support is the set of operations available in a field. Policy is the ordering, cost, or selection relation that makes some supported operations more natural than others.

Support
OperationSupport(F): admissible operations in field F.
Policy
op1 <=_F op2: op1 is no harder or no less natural than op2.

Recurrent architecture pressure appears when the same low-cost region of this support is selected again and again. A field does not need to force a single operation. It only needs to make one family of operations repeatedly easier than its rivals.

Design implication

Definition 1-2. An organization field is the ownership, communication, review, and agent context that changes the recurrent operation families selected by a development organization.

SFT status conceptual core Part 2 treats recurrent paths as support and policy objects rather than organizational destiny.

Design habit. If an architecture boundary matters, design the organization field so changes preserving that boundary are easy to propose, review, and repeat.

Proposition 1-2. Organization structure changes architecture by changing the recurring operation support from which ordinary PRs are drawn.

Proof idea. Ownership boundaries, communication paths, and review routes alter the cost and visibility of candidate operations. Repeated low-cost operations become recurrent PR shapes. Recurrent PR shapes become recurring architecture operations. SFT treats that recurrence as field pressure rather than deterministic command.

Counter-reading 1-2. Conway's Law is not a proof that every architecture mirrors the org chart. It is a field reading: the organization changes the support and policy through which architecture is repeatedly modified.
