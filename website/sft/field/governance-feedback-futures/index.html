Can feedback make software evolution a trajectory rather than a sequence of endpoints?

How the field learns

Governance produces memory. A review rejection records that a path was not acceptable under current evidence. A CI failure names an observation axis. An incident makes a hidden invariant expensive. A postmortem can turn runtime pain into future support, tests, ownership, or review rules.

Vocabulary

Governance feedback

Review, CI, type checking, runtime observation, incident response, and postmortem evidence when they are preserved as memory for later support and policy.

Not just a gate
It changes what the next proposal can see, select, or avoid.
Field effect
Adds constraints, tests, observation axes, ownership routes, or review rules.

This is the closed-loop heart of SFT. A field proposes futures, governance selects among them, observation reveals what the field failed to see, and the result becomes memory for the next proposal. A healthy architecture process is not merely a gate. It is a learning system that changes the support of future changes.

  • Restrict Remove unsafe support when a path should not be available.
  • Redirect Raise shortcut cost and lower the cost of the lawful path.
  • Instrument Add tests, checks, telemetry, or runtime observations that make drift visible.
  • Learn Record review, CI, and incident outcomes as memory for the next field state.

Architecture futures

SFT studies futures as reachable path families, not endpoints. A future includes the field path, the architecture projection path, the signature movement, obstruction candidates, and the boundaries of what was not observed.

Vocabulary

Architecture future

A reachable family of field and architecture trajectories, together with signature movement, obstruction candidates, and the boundary of what was observed.

Object
Path family, not a single predicted endpoint.
Observation
Read through signature trajectory and forecast boundary.

This language is intentionally different from prediction as a single answer. SFT is interested in a cone of reachable paths: which future changes are now closer, which have moved farther away, which regions of architecture are likely to be touched, and which obstructions are likely to matter if the path continues.

Signature trajectory

Endpoint safety is not enough. A path can end with an acceptable signature while making unsafe excursions along the way. SFT therefore keeps trajectory language: what changed, in which order, on which axes, and with which evidence boundary.

This is where ArchSig becomes useful as an observation lens. The point is not to compress architecture into one score. The point is to keep a multi-axis record of movement: dependency direction, boundary preservation, semantic obstruction, runtime exposure, review evidence, and the places where the field still cannot see.

before

What did the field support?

Which operations were cheap, visible, reviewable, or hidden before the artifact arrived?

during

Which path was selected?

Which PR shape appeared, which invariants were preserved, and where did obstructions surface?

after

What did the field learn?

Which tests, review rules, owner routes, incident memories, or agent policies changed future support?

next

Which futures moved?

Which good paths became easier, which shortcuts became expensive, and which unknowns remain active?

Forecast and update

The mathematical heart of SFT is a path object. Given a field, a support function, and a finite horizon, SFT reads a forecast as a bounded family of reachable field paths. The future is not a point prediction. It is a cone of paths generated by supported steps.

Vocabulary

ForecastCone

The bounded set of reachable field paths from a starting field under selected operation support and horizon.

Inputs
Start field F, support U, horizon h.
Reading
Which path classes have become reachable or closer under the current field.

Governance closes the loop. A review, CI result, type check, runtime observation, rollback, or postmortem is not only a verdict on one change. It is an update to the field from which the next cone will be generated.

Bridge to computation

Definition 1-3. Governance feedback is the field update pressure created by review, CI, runtime observation, incident response, and postmortem evidence. Architecture futures are the reachable field and architecture trajectories shaped by that updated memory.

SFT status conceptual core Part 2 introduces bounded path and cone objects for this reading.

Proposition 1-3. A governance outcome changes future architecture possibilities when it is preserved as field memory that alters support, policy, or observation.

Proof idea. A review, CI result, incident, or postmortem can add a missing invariant, forbid a shortcut, introduce a test, change ownership, or add an observation axis. Each of these changes the selected field estimate used by later support and path computation. Thus the feedback affects reachable futures by changing the state from which later paths are generated.

Example 1-3. A rounding incident after coupon launch can become a runtime observation, a missing invariant, a test, a review checklist item, and a future support constraint. The next coupon-related path begins from a different field.

Chapter summary

Governance is field feedback. Review, CI, incidents, postmortems, and runtime observations do not merely accept or reject a finished change; they update the memory from which the next operation support is computed. Architecture futures begin in that updated field.

The mathematical guard is trajectory, not endpoint rhetoric. SFT keeps field paths, architecture projections, signature movement, obstruction candidates, and forecast boundaries visible so Lean-facing theorem schemas can state exactly which path properties are preserved.

The practical ambition is bigger than inspection. SFT points toward attractor engineering: designing PRDs, reviews, tests, AI policies, ownership routes, and operational feedback so that good changes become easier to find and bad shortcuts become visible before they become the obvious path.
