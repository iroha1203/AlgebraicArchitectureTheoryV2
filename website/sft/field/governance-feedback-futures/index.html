Can feedback make software evolution a trajectory rather than a sequence of endpoints?

How the field learns

Governance produces memory. A review rejection records that a path was not acceptable under current evidence. A CI failure names an observation axis. An incident makes a hidden invariant expensive. A postmortem can turn runtime pain into future support, tests, ownership, or review rules.

Architecture futures

SFT studies futures as reachable path families, not endpoints. A future includes the field path, the architecture projection path, the signature movement, obstruction candidates, and the boundaries of what was not observed.

Signature trajectory

Endpoint safety is not enough. A path can end with an acceptable signature while making unsafe excursions along the way. SFT therefore keeps trajectory language: what changed, in which order, on which axes, and with which evidence boundary.

Bridge to computation

Definition 1-3. Governance feedback is the field update pressure created by review, CI, runtime observation, incident response, and postmortem evidence. Architecture futures are the reachable field and architecture trajectories shaped by that updated memory.

SFT status conceptual core Part 2 introduces bounded path and cone objects for this reading.

Proposition 1-3. A governance outcome changes future architecture possibilities when it is preserved as field memory that alters support, policy, or observation.

Proof idea. A review, CI result, incident, or postmortem can add a missing invariant, forbid a shortcut, introduce a test, change ownership, or add an observation axis. Each of these changes the selected field estimate used by later support and path computation. Thus the feedback affects reachable futures by changing the state from which later paths are generated.

Example 1-3. A rounding incident after coupon launch can become a runtime observation, a missing invariant, a test, a review checklist item, and a future support constraint. The next coupon-related path begins from a different field.

Chapter summary

Governance is field feedback. Review, CI, incidents, postmortems, and runtime observations do not merely accept or reject a finished change; they update the memory from which the next operation support is computed. Architecture futures begin in that updated field.

The mathematical guard is trajectory, not endpoint rhetoric. SFT keeps field paths, architecture projections, signature movement, obstruction candidates, and forecast boundaries visible so Lean-facing theorem schemas can state exactly which path properties are preserved.

Chapter glossary

architecture future
A field path together with architecture projections, signature movement, obstruction candidates, and forecast boundary.
signature trajectory
The ordered movement of observed signature records along a path, not just endpoint delta.
feedback memory
Review, CI, runtime, or incident evidence preserved for later support and policy computation.
