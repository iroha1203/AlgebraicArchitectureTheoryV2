Architecture Futures

SFT studies reachable architecture futures, not a single predicted endpoint. A future is read through trajectories, signature movement, obstruction candidates, and default paths under an explicit field boundary.

Future as path family

An architecture future is not merely an end state. It is a field path: a sequence of supported operations, observations, review mediations, and feedback updates. The path matters because a safe-looking endpoint can still require unsafe intermediate states or hidden boundary crossings.

SFT therefore asks about trajectory, not only destination. Which signature axes move? Which obstruction candidates appear? Which default path becomes cheaper? Which missing invariant prevents reviewers from comparing futures?

Reachability, not certainty

A field changes which futures are reachable and which are naturally selected, but it does not decide one future in advance. The selected field model, operation support, policy, observation boundary, and horizon define the bounded problem.

This is the bridge from Part 1 to Part 2. Once the development organization is read as a field, SFT can make software evolution computable by naming the slice, support relation, step relation, ForecastCone, and report projection.

Boundary

ForecastCone is not defined on this page. Part 1 keeps the reader at the level of field intuition and reachable futures; Part 2 introduces the formal and tooling-facing objects.
