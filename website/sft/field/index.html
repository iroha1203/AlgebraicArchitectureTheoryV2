SFT Part 1

The Development Organization as a Field

SFT begins by changing the object of study. The object is not only a repository snapshot, a process chart, or an org chart. It is the field that makes some future changes natural: codebase memory, artifacts, developers, AI agents, ownership, review, CI, incidents, and lifecycle pressure acting together before the next diff exists.

What is already acting on the next change before anyone opens the next pull request?

The object

The first move of SFT is to stop treating the repository as the whole object. A repository snapshot tells us what has settled. It does not tell us why a certain change will feel obvious next, why another change will be expensive to justify, or why a missing invariant will stay invisible until production.

The field is that larger object. It includes the source tree, but also the requirements that named the work, the issue boundaries that split it, the review comments that became precedent, the tests that make one path cheap, the owners who must be consulted, the incident memory that changed what the team fears, and the AI agent context that makes some local pattern easy to reproduce.

SFT studies this field because architecture futures are born there. Before a PR changes the codebase, the field has already changed what can be proposed, what can be seen, what can pass review, and what will be remembered afterward.

Field habit. Begin by asking what the present system makes easy, hard, visible, hidden, repeatable, and nearly unthinkable.

Codebase memory

A codebase is not only implementation. It is memory. Old requirements, design compromises, migration scars, ownership boundaries, incident repairs, review habits, and copied local patterns are sedimented into it. They remain active because later developers and AI agents read them as examples of what is normal.

This memory is why two codebases with the same module graph can evolve differently. One codebase makes boundary-preserving changes legible because its field remembers the boundary. The other makes local patching cheap because its field remembers shortcuts better than invariants.

Artifacts and agents

SFT gives artifacts a more active role than ordinary process language does. A PRD, issue, design memo, review comment, or incident report is not merely a description of work. It changes the field by naming regions, proposing decompositions, hiding or exposing invariants, and making some operation families easier to select.

Developers and AI agents then act inside that shaped field. They do not generate proposals from nowhere. They interpret the artifact, look for nearby patterns, follow or resist review pressure, and produce candidate changes that the field makes plausible. In this sense, a codebase and its artifacts become a prompt for the next change.

Artifact action. A document can act on architecture without touching code because it changes operation support: which changes are named, cheap, expected, reviewable, or ignored.

Coupon PRD

The running example is a coupon PRD in a checkout and payment system. The artifact may look small: "allow coupons at checkout." But that sentence can open several architecture futures. One path adds a lawful DiscountPolicy. Another patches PaymentAdapter. Another creates a UI-only discount drift. Another exposes a rounding or refund semantic obstruction.

This is why SFT cares about the field before the code changes. A better PRD does not simply add more prose. It changes which paths are easy to find by naming boundaries, invariants, observation axes, and review questions before the implementation collapses the future into one diff.

Conway as field pressure

Conway's Law also fits naturally here. Organization structure does not command the architecture directly. It changes issue decomposition, ownership routes, approval flows, review friction, and the recurrent shape of PRs. Repeated operation shapes then settle into architecture.

SFT reads that as field pressure. A team split around frontend, backend, data, and infrastructure will tend to make different changes natural than a team split around search, checkout, and billing capabilities. The theory does not need to moralize the difference. It asks which support each field creates.

Definition 1.0. A development field is the selected organizational and technical state from which SFT reads supported software-evolution paths. AAT appears inside it as the local algebra of the architecture projection.
