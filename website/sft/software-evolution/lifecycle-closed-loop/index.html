Can repair, migration, contraction, and end-of-life be read as field transformations?

Lifecycle trajectory

Lifecycle is not outside architecture. Birth, growth, stabilization, drift, repair, migration, contraction, and end-of-life all change future operation support. End-of-life is not simply failure; it can be an explicit field reconfiguration.

Vocabulary

LifecycleTrajectory

The field path through birth, growth, stabilization, drift, repair, migration, contraction, and end-of-life.

Why it matters
Lifecycle phase changes available support, ownership capacity, runtime risk, and migration options.
Intervention reading
Repair, migration, contraction, and deletion are field transformations.

Decision surface

SFT compares repair, migration, contraction, and deletion by looking at architecture signatures, ConsequenceEnvelopes, runtime risk, ownership capacity, migration support, rollback paths, and unknown remainder.

Vocabulary

Field capacity

The support, ownership, staffing, runtime guard, and migration capacity available for future changes.

Not just
Team size or project budget.
Used for
Comparing repair, migration, contraction, deletion, and end-of-life decisions.
Intervention Reading Comparison axes
Repair Keep the field and add local support or invariants. repair adoption, missing invariant, runtime regression risk, ownership capacity.
Migration Move support and memory from old field to new field. projection mismatch, dual-run cost, compatibility, rollback path.
Contraction Reduce unsupported surface and reallocate capacity. removed operation family, affected users, runtime boundary, simplified axes.
Deletion Remove a field component from the lifecycle. dependent operation loss, data or API retirement, incident avoidance, unknown remainder.

A lifecycle decision report should keep selected inputs, affected regions, field capacity impact, runtime risk boundary, ownership boundary, calibration refs, and non-conclusions for each intervention. It is not a project-management oracle. It is a comparable SFT surface for future support.

Closed loop

Forecasts become useful when they return to the field as evidence. Observed PRs, reviews, CI failures, incidents, repairs, and adoption outcomes update the posterior field record. The next forecast begins from that updated memory.

Field update

The loop is not a claim that forecasts automatically become better. It is a record-preservation discipline. Forecast error, missing evidence, unexpected witnesses, review outcomes, runtime feedback, and policy drift are written into the posterior field so later computation begins from a richer state.

Vocabulary

Posterior field record

The updated field estimate that preserves observed outcome, forecast error, unexpected witnesses, missing evidence, and refreshed support.

Guarantees
Record preservation under the selected update rule.
Does not guarantee
Automatic accuracy improvement, calibration, or governance effectiveness.

Workbench loop

A closed-loop SFT workbench ties forecast artifacts, governance projections, operational feedback, and calibration hooks into one review cycle. The point is not automatic decision making. The point is to keep the same artifact graph alive from proposal to forecast to intervention to observed outcome.

This loop is where Part 1, Part 2, and Part 3 meet. The field supplies memory and force. Computation produces cones and envelopes. Intervention changes support, policy, observation, and feedback. The observed result becomes the next field memory.

Calibration boundary

Calibration links forecast items to later observations. It does not promote forecast artifacts into correctness theorems. The categories matter because missing, private, unavailable, and not-comparable evidence must not be silently read as success.

Stage Boundary
Forecast capture Store bounded forecast before observing the later PR or outcome.
Observation capture Record PR, review, CI, incident, rollback, and private / unavailable data boundary.
Linkage Connect forecast item refs to observed refs as matched, unmatched, unavailable, private, or notComparable.
Review classification Store false positive, false negative, inconclusive, and actionability notes as policy input.
Hypothesis refresh Retain, revise, or reject empirical hypotheses without claiming causal proof.

Workbench direction

Definition 3-3. Closed-loop evolution is the SFT workflow in which forecast artifacts, observed outcomes, and field-update notes are joined so future estimates preserve what the system learned.

Lifecycle habit. Do not treat repair, migration, contraction, and deletion as project-management labels. Treat them as changes to the field's future capacity.

Proposition 3-3. Lifecycle decisions are field interventions because they change available support, ownership capacity, migration paths, runtime risk, and the unknown remainder carried into future forecasts.

Proof idea. Repair adds local support or missing invariants. Migration transfers support and memory across old and new fields. Contraction removes unsupported operation families. Deletion removes a field component from future support. Each decision changes the input to later ForecastCone and ConsequenceEnvelope generation.

Example 3-3. A payment discount subsystem with repeated rounding incidents can be repaired by adding invariants, migrated behind a new policy service, contracted by removing unsupported coupon combinations, or deleted if the field can no longer support the surface. SFT compares the future capacity each choice creates.
