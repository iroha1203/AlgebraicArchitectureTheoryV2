How do we build a field where good changes are the natural next move?

Designing the field

SFT intervention is the design side of the theory. Part 2 asks which futures are reachable under a selected field model. Part 3 asks what a team can change when that reachable shape is not the one it wants to inhabit.

The answer is rarely a single gate. A review rule may remove a shortcut from support. A type boundary may make an invariant cheap to preserve. A runtime check may make a hidden path observable. A lifecycle decision may retire a surface the field can no longer support. These are not after-the-fact quality controls. They are changes to the future shape of the system.

This is the practical promise of SFT: once software evolution is visible as a field-shaped computation, engineering can move upstream. We can design the conditions under which the next good change becomes obvious.

Definition 3.0. A field intervention is a bounded change to support, policy, observation, feedback update, or lifecycle capacity intended to reshape the reachable architecture futures of a selected SFT field model.

Governance dynamics

Governance is often read as a final checkpoint: approve, reject, request changes. SFT reads it as part of the dynamical object. Review, CI, type checking, ownership rules, architecture constraints, runtime guards, and AI policy all change future operation support.

A good governance system does not only block bad changes. It lowers the cost of lawful paths, makes missing invariants visible, routes ambiguous proposals into design work, preserves observed failures as field memory, and gives AI agents better local structure to imitate.

Attractor engineering

Attractor engineering is the most vivid application of SFT: design the field so desirable futures are natural, cheap, visible, reviewable, and repeatable, while shortcut paths become costly, observable, and mediated by review.

This reframes "good design" as field design. We do not only ask whether the current architecture looks clean. We ask whether the field makes the next boundary-preserving operation easy to find, easy to implement, easy to review, and easy to reinforce with feedback.

Intervention habit. Do not only add guardrails around a bad default path. Change the field so the better path becomes the default path.

AI and lifecycle

AI coding agents make this problem urgent. An agent tends to follow the path made visible by the existing codebase and surrounding artifacts. If the field teaches shortcut patterns, the agent can amplify them quickly. If the field teaches explicit boundaries and preserved invariants, the agent can also amplify those structures.

Lifecycle decisions belong to the same surface. Repair, migration, contraction, and deletion all reshape support and capacity. End-of-life is not a failure of growth; it can be a field intervention that closes unsupported futures and frees capacity for better ones.

Closed loop

SFT becomes most powerful when forecast and outcome meet. A ConsequenceEnvelope names expected affected regions, path classes, signature axes, obstruction candidates, and missing boundaries. Later PRs, reviews, CI outcomes, runtime incidents, and repairs can be joined back to those forecast items.

The field then learns in a disciplined way. It records where the forecast was too narrow, where support was wrong, where a missing invariant appeared, where a review rule helped, and where unknown remainder should become visible next time.
