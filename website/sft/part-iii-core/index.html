Reader model

SFT does not say

A PRD, issue, or AI proposal applies a physical force to a codebase, determines one future, or carries its own forecast. A field update is not an automatic accuracy improvement theorem.

SFT says

Inside a selected field model, artifacts induce bounded candidate updates. Support, policy, observation, and governance determine reachable path sets and the reportable consequence envelope.

Non-conclusion III.0. Part III is a theorem-bearing computational fragment, not a complete model of social organization, runtime causality, or empirical prediction quality.

Claim status boundary statement Checked Lean anchors are listed where available; empirical prediction quality remains outside this claim.

SoftwareField and architecture projection

`SoftwareField` is a computable slice of a development field. It is not itself an AAT `ArchitectureObject`; the AAT object is obtained through architecture projection.

Definition III.1. A SoftwareField is the selected computable state: contextual dynamics, architecture projection, observed signature record, history, support, policy, constraints, observation model, governance model, and exogenous artifact inputs.

Claim status conceptual schema Defined as SFT vocabulary; Lean formalization currently exposes selected supporting records and theorem boundaries.

  • Contextual state History, constraints, observations, governance, and exogenous artifact inputs belong to the field.
  • Architecture projection AAT reasoning applies to the projected architecture object, not to every field component at once.

Artifact-mediated change

The formal term is `ArtifactAction` or `ArtifactMediatedChange`. Force remains an informal field-level reading. PRDs, specs, issues, and AI proposals can produce multiple candidate updates rather than a single next field.

Definition III.2. An ArtifactAction maps a field to a set of candidate update hypotheses under an explicit interpretation boundary. A deterministic action is only the singleton special case.

Claim status set-valued schema Forecasted effects are outputs of forecasting, not fields of the raw action.

Forecasted effects are outputs of forecasting, not part of the raw action definition. This prevents an artifact from already containing the forecast that it is supposed to generate.

  • Raw action Source artifact, target components, interpretation boundary, candidate updates, action boundary, and observable boundary.
  • Force descriptor Action class, target regions, candidate support families, affected policy or governance components, and uncertainty boundary.
  • Observed effect Transition, before / after signatures, comparable deltas, unexpected witnesses, and review / CI / runtime outcome.

Operation support and policy

SFT asks which operations are lawful, but also which operations appear natural, possible, low cost, or likely to be selected in a field.

Definition III.3. OperationSupport(F) is the admissible operation set after constraints and governance interventions. OperationPolicy(F) is a preorder, cost, selection relation, or explicitly supplied probability distribution on that support.

Claim status computational schema Probability semantics are not assumed unless a distribution is part of the model.

In larger field models, support is not a single undifferentiated relation. SFT separates generated operations, review candidates, theorem-admissible steps, governed support, and accepted trajectory steps.

Cone-narrowing claims must state which support layer is being compared. The standard governance reading is a pointwise inclusion of governed support, often after policy has been compiled into SelectedSupport_pi.

A probability distribution mu_F : Distribution SupportedOperation is added only when the selected semantics explicitly includes one.

  • OperationSupport The admissible operation set after constraints and governance interventions.
  • OperationPolicy A preorder, cost, selection relation, or explicitly probabilistic distribution on supported operations.

ForecastCone and ConsequenceEnvelope

`ForecastCone` is the formal set of reachable field paths from a starting field, support relation, and horizon. For set-valued artifact actions, SFT forms a family of cones and summarizes them for practice as a `ConsequenceEnvelope`.

Definition III.4. ForecastCone(F, U, h) is the set of field paths starting at F, bounded by horizon h, whose adjacent transitions are realized by operations selected from support U.

Claim status formal object schema The cone is set-valued reachability, not a point prediction.

Bounded reachability is not automatically computability. For a cone to be used as a finite computation or report input, the selected support must be finite or enumerable, support membership must be decidable, step witnesses must be decidable or semi-decidable, the horizon must be bounded, the observation projection must be finite, and the unknown remainder must be explicit.

Definition III.5. A ConsequenceEnvelope is the report form that gathers selected forecast cones, reachable path classes, affected architecture regions, comparable signature axes, obstruction witnesses, missing boundaries, recommendations, forecast boundary, and unknown remainder.

Claim status report schema The envelope interprets one or more cones for diagnosis and tooling.

The projection is one-way: a consequence envelope does not uniquely recover the cone family and does not supply causal correctness, probability, or calibrated point prediction.

Each reported path class, affected axis, or obstruction candidate must identify whether it is supported by a reachable path witness, a heuristic reference, or empirical evidence. This keeps the envelope connected to the formal core without pretending that every report item is a Lean theorem.

Observation, governance, and FieldUpdate

Review, CI, type checking, architecture rules, AI policy, and runtime guards are governance interventions because they can change support, policy, observation, and feedback update.

`FieldUpdate` records the difference between forecast and observation: signature delta, unexpected witnesses, review or CI outcome, runtime feedback, missing evidence, and non-conclusions.

Definition III.6. An ObservationBoundary records measured axes, unavailable evidence, private evidence, extractor version, and non-conclusions. A GovernanceIntervention changes support, policy, observation, feedback update, or escalation.

Claim status boundary schema Review, CI, type checking, architecture rules, AI policy, and runtime guards are modeled here.

  • Restrictive intervention Removes unsafe support from later accepted operations.
  • Redirective intervention Raises shortcut cost and lowers lawful path cost inside the selected field model.
  • Instrumenting intervention Adds observation axes, tests, runtime checks, or evidence records.
  • Escalating intervention Converts a proposal into a review, design, deferral, or incident record.
  • Learning intervention Preserves observed outcomes and forecast error in posterior field memory.

Field update preserves forecast error and boundary evidence. It does not automatically prove that future forecasts become more accurate.

Schema III.7. A FieldUpdate takes a prior field, a forecast cone or envelope, observed signature delta, unexpected witnesses, review / CI outcome, and runtime feedback, then produces a posterior field that records the forecast boundary and non-conclusions.

Claim status Lean record package Formal/Arch/Evolution/SFTFieldUpdate.lean proves record-preservation accessors; forecast-quality calibration remains separate.

Core boundary and non-conclusions

Part III defines the theorem-bearing fragment, not the whole social reality of a development organization. `ForecastCone` is a reachable path set under selected support and horizon; `ConsequenceEnvelope` is the reader-facing report that explains the cone family, unknown remainder, and forecast boundary.

The core does not assert that artifacts have unique effects, that policy is probabilistic unless a distribution is supplied, or that a field update automatically improves later forecast quality.
