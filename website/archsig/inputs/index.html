Input map

  • Repository sourceLean or Python source roots used to build the component and dependency universe.
  • PolicyForbidden edges, boundary rules, adopted law refs, and organization review thresholds.
  • Runtime edgesObserved runtime dependency or exposure evidence that static import graphs do not fully express.
  • PR metadataAuthor, branch, files, labels, reviewers, and review context used by diff and report commands.
  • Proposal textPRDs, specs, issues, and AI proposals used by SFT-facing artifact commands.
  • Operational recordsIncidents, thresholds, repairs, outcome ledgers, ownership monitors, and calibration records.

Scan inputs

The root selects the repository universe. Python mode also needs source and package roots so import names can be interpreted against the selected package boundary.

Policy file

A policy file turns organizational architecture expectations into measured policy axes. It can express forbidden dependencies, boundary expectations, abstraction rules, adopted law references, and threshold decisions. Omitted policy does not mean no violation exists; it means the policy axis was not selected for that scan.

Runtime evidence

Runtime edges add evidence from traces, incidents, service calls, job execution, event flow, or other operational sources. Use them when static dependency structure understates the architecture risk.

Runtime evidence should describe what was observed and what was not observed. A missing runtime path is not proof that no runtime interaction exists.

PR metadata

PR metadata gives signature diffs and review reports their review context. Useful metadata includes PR number, title, author, base/head revisions, changed files, labels, reviewers, linked issues, and generated-change provenance.

Proposal inputs

ArchSig can normalize PRD, spec, GitHub Issue, and AI proposal artifacts into descriptor records for SFT-facing workflows.

The descriptor records source refs, scope, action class candidates, missing evidence, measurement boundary, and forecast non-conclusions. It does not by itself generate a forecast probability.

Operational inputs

Operational feedback inputs let teams calibrate signals over time: report outcome ledgers, calibration review records, threshold policies, ownership boundary monitors, repair adoption records, incident correlation monitors, and hypothesis refresh cycles. These records support empirical learning without turning correlation into a theorem.
