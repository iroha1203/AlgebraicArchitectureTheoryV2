Input map

  • ArchMap observationSupplied archmap-observation-map-v0 with atom observations, molecule observations, semantic observations, concern hints, gaps, provenance, and non-conclusions.
  • LawPolicySelected law universe, witness rules, molecule patterns, obstruction definitions, signature axes, coverage requirements, and exactness assumptions.
  • Adapter evidenceOptional Lean or Python Sig0 import-graph evidence for bounded conflict checks or comparison workflows.
  • Runtime edgesObserved runtime dependency or exposure evidence that static import graphs do not fully express.
  • PR metadataAuthor, branch, files, labels, reviewers, and review context used by projections and reports.
  • Proposal textPRDs, specs, issues, and AI proposals used by FieldSig-facing artifact commands.

ArchMap input

ArchMap is supplied evidence, not an automatic claim that the tool extracted a complete architecture. Each observation should point back to source refs and each missing area should remain a gap.

LawPolicy file

A LawPolicy file selects the laws ArchSig should use to read one ArchMap. It can express witness rules, molecule patterns, obstruction circuit definitions, signature axes, coverage requirements, exactness assumptions, and non-conclusions. It is not AAT theory itself.

Runtime evidence

Runtime edges add evidence from traces, incidents, service calls, job execution, event flow, or other operational sources. Use them when static dependency structure understates the architecture risk.

Runtime evidence should describe what was observed and what was not observed. A missing runtime path is not proof that no runtime interaction exists.

PR metadata

PR metadata gives review projections their review context. Useful metadata includes PR number, title, author, base/head revisions, changed files, labels, reviewers, linked issues, and generated-change provenance.

Proposal inputs

FieldSig normalizes PRD, spec, GitHub Issue, and AI proposal artifacts into descriptor records for SFT-facing workflows. ArchSig receives the resulting artifact refs only as review evidence.

The descriptor records source refs, scope, action class candidates, missing evidence, measurement scope, and forecast review follow-ups. It does not by itself generate a forecast probability.

Operational inputs

Operational feedback inputs let teams calibrate signals over time: report outcome ledgers, calibration review records, threshold policies, ownership monitors, repair adoption records, incident correlation monitors, and hypothesis refresh cycles. These records support empirical learning without turning correlation into a theorem.
