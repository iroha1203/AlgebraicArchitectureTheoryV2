Input map

ArchSig does not guess every architecture fact from source files. It reads a selected repository universe, then adds optional evidence supplied by the team. Missing input stays visible as unmeasured or unavailable evidence.

  • Repository root The checkout to scan. For Lean, this usually contains lakefile.toml or lakefile.lean. For Python, pair it with source and package roots.
  • Policy file Local layer, boundary, or abstraction rules that turn intended architecture constraints into measurable policy axes.
  • Runtime edge evidence Observed runtime dependencies supplied by instrumentation, traces, or hand-curated evidence.
  • PR and proposal metadata PR JSON, review refs, PRD / Spec / Issue text, or AI proposal records used by review and SFT forecasting commands.

Scan inputs

The smallest scan needs only a root and an output path. Add language and package options when scanning Python packages, and add policy or runtime evidence when those axes should be measured rather than left unknown.

Policy file

A policy file is how a team tells ArchSig which architectural boundaries matter locally. Without a policy file, policy axes may appear as placeholder values with unmeasured status.

After adding policy input, first inspect policyViolations, boundaryViolationCount, abstractionViolationCount, and the corresponding metricStatus entries.

Runtime edges

Runtime evidence lets ArchSig report dependencies that do not appear as static imports. Treat this file as measured evidence: if no runtime edge file is supplied, runtime exposure remains outside the scan.

PR metadata

PR metadata connects a diff report to a review event. It gives attribution candidates and PR comments enough context to say which change introduced or removed evidence.

  • Use with signature-diff, air, feature-report, policy-decision, and pr-comment.
  • Include PR number, base and head refs, changed files, author or actor refs, and any review evidence available to automation.

Proposal inputs

SFT commands start from proposal text: PRDs, specs, issues, or AI proposal records. The first command normalizes that text into an ArtifactDescriptor, then later commands estimate operation support and consequence envelopes.

If evidence is not supplied, ArchSig should say that it is missing. Do not replace missing input with a safe-looking zero.
