Fixture scan

Begin with the minimal fixture so the shape of the artifact is visible before any live repository noise appears. The fixture includes policy input and runtime edge evidence, so the first output already shows how structural, policy, and runtime axes sit beside their measurement status.

The scan chooses a root, walks the supported source language, records components and dependency edges, attaches optional policy and runtime evidence, and writes a Sig0 JSON document. It does not claim that unsupported dynamic behavior has been observed.

Open .lake/sig0-fixture.json after the command. The first fields to check are schemaVersion, components, edges, signature, and metricStatus.

Validate Sig0

Validation checks the component list, dependency edge closure, external targets, and policy metric status in an existing Sig0 JSON file. It does not rescan source files, and it does not strengthen an unmeasured axis into a measured claim.

Treat validation as the artifact's first reader. It asks whether the bounded universe is internally coherent enough for later comparison, reporting, and review projection.

Open .lake/sig0-fixture-validation.json. A useful first validation read asks whether duplicate components, edge closure problems, external targets, and policy status warnings are present.

Repository scan

A repository scan writes Sig0 for the current checkout. The same reading rule applies: the scan is a bounded observation of supported constructs, not a complete architecture model. Keep policy and runtime evidence explicit when those axes matter; omitted inputs can leave placeholder values with unmeasured status.

For a real repository, keep the first run boring. Do not tune policy yet. First learn what static structure the scanner can see, then add policy and runtime evidence in a second pass.

Python scan

Python mode extracts import and from ... import ... edges with the standard AST parser. Dynamic import, plugin loading, and framework conventions remain boundary items.

Use --source-root and --package-root to make the selected import universe explicit. This is what lets later reports distinguish a measured dependency from a dependency family that the adapter does not yet observe.

Expected files

After the first session, you should have at least one Sig0 file and one validation report. These are the files that later commands consume.

  • .lake/sig0-fixture.json Fixture signature. Use it to learn the JSON shape before reading your repository's output.
  • .lake/sig0-fixture-validation.json Validation report for the fixture signature. Use it to see how ArchSig reports artifact quality.
  • .lake/sig0.json Your repository signature. This becomes the input to snapshot, diff, AIR, and review commands.

First reading

Do not read the numeric signature alone. Read metricStatus, unmeasuredAxes, validation status, and non-conclusions before deciding whether an axis is a measured zero, measured nonzero, unmeasured, or out of scope.

The next chapter gives the full output reading recipe. For the first run, focus on three questions: did the scan find the components you expected, did it find dependency edges that make sense, and which axes are unmeasured?

  • Measured zero The adapter observed the axis in the selected universe and the measured value is zero.
  • Unmeasured The artifact may contain a placeholder value, but the status says the axis was not observed.
  • Not comparable Before / after artifacts do not share enough coverage for a meaningful delta on that axis.
  • Out of scope The selected artifact family or adapter does not attempt to cover that axis.

A placeholder value of zero is not evidence that the risk is zero unless the corresponding metric status says it was measured.

After the first run

Getting started deliberately stops at the Core surface. Move to Review only after Sig0 and validation output are readable; move to SFT forecasting only when the input artifact and bounded horizon are explicit; move to Operational feedback only when outcome records and privacy boundaries are available.

  • Review surface Use AIR, theorem precondition checks, Feature Extension Report, policy decision, and PR comment output for PR review cues.
  • SFT surface Use sft-forecast for bounded report projection, not point prediction or causal proof.
  • Operational surface Use datasets and B10 artifacts for calibration and feedback, not theorem promotion.
