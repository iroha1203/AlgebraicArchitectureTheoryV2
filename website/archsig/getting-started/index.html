Install and prerequisites

ArchSig currently runs from the repository tooling crate. You need Rust and Cargo available, then run commands from the repository root.

The primary workflow starts from supplied ArchMap observation JSON and a selected LawPolicy JSON. Adapter scans are optional bounded evidence, not the default review path.

Fixture workflow

Begin with the minimal fixture so the artifact shape is visible before live repository noise appears.

The output directory contains ArchMap validation, LawPolicy validation, the ArchSig analysis packet, analysis validation, an LLM interpretation packet, and bounded AIR / theorem-check / feature-report / observable-bundle projections.

Step-by-step commands

Use the separate commands when you want to inspect each boundary.

Adapter evidence

Use adapter-scan when you need bounded Lean or Python import-graph evidence. Adapter output can support conflict checks and older comparison workflows, but it is not an ArchMap generator and it does not prove extractor completeness.

First reading

  • atomObservationsSource-grounded observations. They do not create atom truth.
  • concernHintsReview cues from the observation stage. They are not obstruction circuits.
  • selectedLawPolicyRefThe policy under which this packet reads the ArchMap.
  • architectureSpectrumReportACTS hotspots, witness clusters, recurrent obstruction supports, coverage gaps, and review focus for current-state inspection.
  • obstructionCircuitsLaw-relative analysis outputs computed by ArchSig from ArchMap plus LawPolicy.
  • signatureAxesMulti-axis readings, not a single quality score.

Read the analysis packet first; read AIR and feature reports as projections from it.

Next moves
