CI shape

The strongest ArchSig CI path validates the supplied ArchMap, validates the selected LawPolicy, builds an ArchSig analysis packet, validates that packet, and then emits bounded review projections for humans and LLM interpretation.

ArchMap and policy

Keep the observation map and policy file as first-class CI inputs. Record which source refs were read, which gaps remain, which laws were selected, and which axes are in scope for this review.

Use adapter-scan only when bounded import-graph evidence is needed. Adapter output can support comparison workflows, but it is not the source of truth for the current review path.

Analysis and reports

The analysis packet is the first result to read. AIR, theorem checks, feature reports, policy decisions, PR comments, and FieldSig handoff artifacts are projections from that analysis state.

PR comment

The PR comment should summarize review action, not dump raw JSON. A good comment tells reviewers which architectural movement matters, what evidence supports it, what was not established, and which files or assumptions deserve inspection.

Review gate policy

  • Block on artifact invaliditySchema or validation failures can block because the review artifact is not trustworthy.
  • Warn on measured riskMeasured architectural risk should usually become reviewer attention unless the organization has an explicit policy threshold.
  • Do not block on unknown as if it were failureUnknown, unmeasured, unsupported, and private evidence states should be visible, not silently converted into pass/fail.
  • Record suppressionsBaseline suppressions need justification so the signal can be recalibrated later.

Minimal workflow

A minimal first CI integration can stop after ArchMap validation, LawPolicy validation, and ArchSig analysis packet validation. Add AIR, feature reports, policy decisions, and PR comments once the team agrees on how analysis packets should be read.
