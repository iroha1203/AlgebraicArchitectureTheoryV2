Use ArchSig

Every artifact is a portable architectural reading.

ArchSig artifacts are designed to move between local runs, CI, PR review, policy decisions, and operational feedback without losing their evidence boundary.

Core artifacts

  • Sig0Baseline architecture signature: components, edges, metrics, metric status, policy status, runtime refs, and unsupported constructs.
  • Validation reportChecks duplicate components, edge closure, external targets, schema shape, and policy metric status.
  • SnapshotWraps a signature and validation report with repository, revision, scan, policy, and extractor metadata.
  • Signature diffCompares before and after snapshots and separates improved, worsened, unchanged, and unmeasured deltas.

Review artifacts

  • archmap-v0Evidence-carrying map candidate from selected source artifacts to architecture components, relations, semantic diagrams, and coverage records.
  • archmap-validation-report-v0Checks source inventory refs, semantic coverage, claim boundary, conflicts, missing evidence, and formal-promotion guardrails.
  • AIRArchitecture Interpretation Record. Connects signatures, diffs, and ArchMap projection to claims, evidence refs, coverage, theorem preconditions, and report projections.
  • Theorem checkReports whether a formal theorem-shaped reading has the needed assumptions and bridge evidence.
  • Feature reportTurns AIR into reviewer-facing explanation of feature extension quality, witnesses, and boundaries.
  • Policy decisionApplies organization policy to the feature report and records pass, warn, block, or review-required outcomes.
  • PR comment summaryHuman-readable review comment with evidence-backed action and scope.

SFT artifacts

  • Artifact descriptorNormalizes PRD, spec, issue, or AI proposal into source refs, scope, action class candidates, missing evidence, and non-conclusions.
  • Operation support estimateEstimates which operations the proposal supports under bounded evidence.
  • Forecast cone skeletonRepresents bounded reachable futures without claiming point prediction.
  • Consequence envelopeProjects likely review-relevant consequence regions and missing evidence.
  • AI proposal governanceRecords governance pressure around AI-generated proposals and required confirmation.

Operational artifacts

  • Dataset recordsPR history, feature extension, outcome linkage, and report outcome ledgers.
  • Calibration recordsReview records, team threshold policies, hypothesis refresh cycles, and detectable values catalogs.
  • Ownership and repair recordsOwnership boundary monitors, repair adoption records, architecture drift ledgers, and operation proposal logs.
  • Incident correlation recordsOperational correlation monitors that remain empirical evidence, not causal theorem proof.

Retention

CI should retain enough artifacts to audit a PR comment later: before/head Sig0, validation reports, snapshots, diff report, ArchMap and its validation report when used, AIR, theorem check, feature report, policy decision, and generated PR comment. Use report artifact retention manifests when teams need reproducible review history.

Diagnostic reading

When an artifact looks wrong, first identify whether the problem is source selection, ArchMap source inventory, policy input, runtime evidence, schema compatibility, validation failure, or claim projection. Do not debug a feature report before checking Sig0, ArchMap validation when present, and AIR.
