Signal shape

This structure is the core product idea. ArchSig makes architecture review practical by turning diffuse concern into bounded, inspectable signal.

What changed?

The first layer is architectural movement: dependency changes, boundary crossings, responsibility shifts, semantic coupling, runtime exposure, proposal scope movement, or operational feedback.

Why it matters?

A raw dependency is not automatically a design problem. ArchSig connects movement to an architectural reading: a port was bypassed, a feature is no longer independent, a runtime path crosses a protected boundary, or a proposal opens a future change region.

What evidence supports it?

  • Structural evidenceImports, dependency edges, component membership, boundary rules, ownership paths, and file changes.
  • Semantic evidenceBusiness rules, responsibilities, state transitions, assumptions, code-reading observations, and AIR records.
  • Operational evidenceTraces, incidents, latency, error propagation, runtime dependencies, and feedback records.

What should the reviewer inspect?

A good signal narrows attention. It should name the boundary, files, policy, runtime evidence, semantic assumption, or missing confirmation the reviewer should inspect.

What is not proven?

This is not a weakness. It is part of what makes the signal trustworthy. A report can guide review while saying runtime completeness was not established, semantic interpretation needs human confirmation, or a theorem precondition is missing.

A trustworthy architecture signal includes its own evidence boundary.

What should be observed next?

Signals should leave a next observation: add runtime evidence, confirm a business rule, extract an explicit port, add policy coverage, watch an incident correlation, or recalibrate a threshold after review outcome data arrives.
