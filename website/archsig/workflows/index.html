First adoption

  1. Create one ArchMapHave an LLM, agent, or reviewer record a narrow source-grounded observation map.
  2. Select one LawPolicyEncode the law universe, witness rules, obstruction definitions, and signature axes for that review.
  3. Build one analysis packetRun llm-native-workflow and read the obstruction, signature, flatness, repair, and gap sections manually.
  4. Review projections secondCheck whether the feature report and observable bundle helped reviewers inspect the right files and assumptions.

PR review

The PR workflow is the daily path: provide ArchMap observations, select LawPolicy, build the analysis packet, validate it, then project AIR, theorem-precondition checks, feature reports, policy decisions, and PR comments as needed.

Do not make the first rollout a hard gate. Start with visible comments and a small selected law universe.

Generated change

For AI-generated code, preserve the proposal and generation context as an artifact. The review question is not only whether tests pass, but whether the generated change moved a responsibility or dependency without making that movement explicit.

  • InputAI proposal, prompt summary, linked issue, touched files, and generated-change provenance.
  • ObservationStructural diff, semantic AIR, changed responsibilities, and missing confirmation.
  • ReviewHuman confirmation for semantic claims and explicit treatment of unmeasured runtime behavior.

Scheduled review

Scheduled reviews help teams notice architecture drift outside individual PRs. They should retain ArchMap observations, selected policies, analysis packets, trajectory reports, drift ledgers, and daily outcome records. The value is trend and calibration, not a sudden universal quality score.

Proposal forecast

SFT-facing workflows start from the ArchSig analysis packet or proposal descriptors and move through operation support estimates, forecast cone skeletons, consequence envelopes, calibration hooks, AI proposal governance, and SFT forecast reports. These artifacts describe reachable future regions and review pressure for planning; they do not prove forecast correctness.

Feedback loop

The loop closes when review outcomes, incidents, repairs, ownership changes, and calibration decisions feed back into future thresholds and interpretation. This is how ArchSig turns one-off PR comments into an architecture evidence layer.
