Core commands

  • Default scanCreate Sig0 from a repository root.
  • validateValidate an existing Sig0 artifact.
  • snapshotWrap Sig0 and validation with revision metadata.
  • signature-diff / diffCompare before and after architecture signatures.

Review commands

  • airCreate AIR from Sig0, validation, diff, metadata, policy, and adapter inputs.
  • archmapValidate supplied archmap-v0 JSON and produce archmap-validation-report-v0.
  • air-from-archmapProject validated ArchMap objects, relations, semantic diagrams, source refs, and coverage records into AIR.
  • validate-airCheck AIR reference integrity; --strict-measured-evidence can make missing evidence a failure.
  • theorem-checkReport theorem precondition status for formal promotion.
  • feature-reportGenerate reviewer-facing feature extension report.
  • policy-decisionApply organization policy to a feature report.
  • pr-commentCreate PR comment summary Markdown.
  • baseline-suppressionRecord suppressions with review context.

Policy and schema

  • organization-policyValidate organization policy shape.
  • report-artifactsValidate report retention manifests.
  • law-policy-templates, custom-rule-plugins, measurement-unitsValidate extension registry artifacts.
  • dynamics-measurementsValidate Architecture Dynamics measurement contracts.

FieldSig handoff

SFT forecast, IntentMap, workflow evidence, governance, calibration, and operational feedback commands now belong to tools/fieldsig. ArchSig keeps the AAT structural scan, validation, ArchMap, AIR, theorem-precondition, feature report, policy, and review cue surface.

  • artifact-descriptorNormalize PRD, spec, issue, or AI proposal artifacts.
  • operation-support-estimateEstimate operation support from selected inputs.
  • forecast-cone-skeletonCreate reachable-future skeletons.
  • consequence-envelopeProject review-relevant consequences.
  • forecast-calibration-hook, ai-proposal-governance, sft-forecastConnect proposals to governance and calibration artifacts.

Validation commands

Validation commands answer different questions. Use the narrowest validator for the artifact you already have.

ValidatorArtifact checkedMain failure means
validateSig0The repository observation is not internally coherent enough for comparison or report projection.
archmaparchmap-v0Source inventory refs, semantic coverage, conflicts, or claim records do not support safe AIR projection.
validate-airAIRClaims, evidence refs, or coverage records do not line up with their source artifacts.
organization-policyOrganization policyPolicy cannot be applied safely to feature reports.
report-artifactsRetention manifestCI cannot later audit which artifacts supported a review comment.
law-policy-templates, custom-rule-plugins, measurement-unitsExtension registry artifactsCustom rules or measurement units may not preserve schema or evidence meaning.
dynamics-measurementsArchitecture Dynamics measurement contractTrajectory or dynamics reports cannot be interpreted against the expected measurement contract.
schema-compatibilityBefore / after report schemasArtifacts should not be silently compared without migration or compatibility handling.

Source reference

The complete repository command guide remains the implementation-facing source for command examples and edge-case notes: tools/archsig/docs/commands.md and tools/fieldsig/docs/commands.md. This public page keeps the product-manual index: what each command consumes, what it emits, and when an engineer should choose it.

Command reading rule

A command produces an artifact. The artifact becomes trustworthy only when its schema, evidence, status fields, and review follow-ups are read together.
