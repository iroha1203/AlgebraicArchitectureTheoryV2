What ArchSig is

ArchSig reads architecture as an observed signature rather than as a single score. A checkout, policy file, runtime edge record, PR description, incident note, or proposal can become an artifact only after its evidence boundary is made explicit. The output is useful because it says both what was seen and what remained outside the observation.

The first object is a bounded component universe: components, dependency edges, selected policy edges, optional runtime evidence, unsupported constructs, and metric status. Later commands preserve that boundary while producing diffs, review reports, policy decisions, forecast envelopes, and operational feedback records.

  • Observation ArchSig records a finite view of a repository or proposal: modules, imports, policy edges, evidence refs, and unsupported constructs.
  • Signature The observed view is projected onto axes such as cycle risk, SCC size, depth, fanout, boundary violations, runtime exposure, and theorem-boundary status.
  • Report Reports translate the signature into reviewable changes: worsened axes, improved axes, missing evidence, precondition status, and recommended follow-up.
  • Feedback Operational artifacts connect signatures to calibration, thresholds, ownership, repair adoption, and outcome records without promoting correlation into theorem status.

Signature model

A signature axis is never just a number. It is a value together with a measurement status, the selected universe, evidence refs, and non-conclusions. This is the difference between an architectural diagnostic and a loose metric dashboard.

  • Structural axes Component count, dependency relation count, cycle flag, SCC maximum size, maximum depth, reachable cone, and fanout risk.
  • Policy axes Forbidden boundary edges, abstraction violations, adopted law refs, policy decision status, and measured policy gaps.
  • Runtime and semantic axes Runtime dependency projection, protected and unprotected paths, semantic witnesses, observation mismatch, and unsupported dynamic behavior.
  • Boundary axes Measured axes, unmeasured axes, private or unavailable evidence, theorem precondition status, exactness assumptions, and explicit non-conclusions.

The central reading rule is simple: a zero value says nothing until the artifact says the axis was measured in the selected universe.

Product surfaces

ArchSig is one manual with four product surfaces. The surfaces share artifact discipline but answer different questions: what exists now, what changed in review, what a proposal may open, and how an organization calibrates the signal over time.

  • ArchSig Core Builds a bounded architecture observation from Lean or Python import graphs, explicit policy input, optional runtime evidence, Sig0 validation, snapshots, and signature diffs.
  • ArchSig Review Turns before / after signatures into AIR, theorem precondition checks, Feature Extension Reports, policy decisions, PR comments, and baseline suppressions.
  • ArchSig SFT Reads PRDs, specs, issues, and AI proposals as bounded change artifacts, estimates operation support, constructs forecast cone skeletons, and projects consequence envelopes.
  • ArchSig Operational Maintains datasets, daily ledgers, calibration reviews, threshold policies, ownership monitors, repair adoption records, incident correlations, and hypothesis refresh artifacts.

Report life

  • 1. Observe A scan or descriptor command chooses the source universe and records components, relations, policy refs, runtime evidence, proposal text, or outcome records.
  • 2. Validate Validation checks duplicates, edge closure, external targets, schema shape, and whether placeholder values are being protected by metric status.
  • 3. Compare Diff reports preserve before / after comparability, mark unmeasured deltas, and separate improved axes from worsened axes.
  • 4. Project Review, policy, SFT, and operational commands project the same bounded evidence into the language used by reviewers, CI, planning, and calibration.

Manual map

The ArchSig pages are meant to be read as a self-contained manual. Repository documents and Rust implementation files can verify details, but the public manual should explain the concepts, artifacts, workflows, and boundaries without sending the reader away first.

  • Getting Started Run the smallest useful path and learn how to read the first Sig0 and validation artifacts.
  • Commands Use scan, validation, diff, review, policy, dataset, forecast, and feedback commands as a connected system.
  • Artifacts Understand Sig0, validation, snapshots, diffs, AIR, feature reports, policy decisions, datasets, and SFT forecast records.
  • Boundaries Read measured zero, unknown, theorem precondition status, non-conclusions, and empirical claims correctly.
  • Workflows Adopt ArchSig for first scan, PR review, generated-change provenance, scheduled scans, architecture dynamics, and SFT forecasting.
  • Schemas Preserve compatibility, migration meaning, required assumptions, deprecated fields, and non-conclusions.
  • Operational Feedback Close the loop with calibration records, thresholds, ownership monitoring, repair adoption, incidents, and hypothesis refresh.
  • Examples Practice report reading with good extensions, hidden interaction witnesses, semantic witnesses, and dynamics examples.
  • Roadmap Track the product surfaces and standardization targets without confusing roadmap status with artifact semantics.

Claim discipline

Architecture Signature is a multi-axis diagnostic, not a scalar quality score. Every report must be read together with measurement status, coverage gaps, theorem preconditions, and non-conclusions.

ArchSig gives architecture review a disciplined grammar: it can say that a witness was measured, that a formal theorem would require missing bridge assumptions, that an empirical pattern is only a calibration signal, or that a recommendation is a review policy decision. These are different claim levels, and the manual keeps them separated.

A measured witness can guide review. It does not by itself discharge a Lean theorem, prove extractor completeness, or establish architecture lawfulness.
