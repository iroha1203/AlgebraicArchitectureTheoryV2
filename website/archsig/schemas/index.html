Version policy

Every durable artifact should carry a schema version. A version change must preserve the meaning of existing evidence or declare the migration boundary explicitly.

Compatibility outcomes

  • CompatibleExisting consumers can read the artifact without meaning changes.
  • Compatible with warningNew fields are present or optional fields changed, but old meaning remains stable.
  • Migration requiredConsumers need explicit transformation before interpretation.
  • IncompatibleThe artifact should not be silently compared or consumed.

Catalog

Schema catalogs cover archmap-observation-map-v0, law-policy-v0, archsig-analysis-packet-v0, their validation reports, bounded AIR / theorem-check / feature-report / observable-bundle projections, Sig0 adapter evidence, signature diffs, policy decisions, PR comments, report retention, extension registries, FieldSig forecast artifacts, operational datasets, and dynamics reports.

Migration rules

A migration should say which fields changed, whether evidence refs preserve identity, whether metric status meaning changed, and which review follow-ups remain valid. Never migrate a placeholder value into a measured value without evidence.

Required assumptions

Schema validity is not architecture validity. A valid artifact can still have missing runtime evidence, unsupported constructs, incomplete semantic confirmation, or theorem precondition gaps. Schema checks protect format meaning; source inventories protect claim meaning.
