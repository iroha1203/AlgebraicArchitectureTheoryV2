The review gap

Architecture review is not the same task as syntax checking, test execution, or style enforcement. The hard question is whether a change moved a responsibility, crossed a boundary, introduced a semantic dependency, or created an operational coupling that future changes must now respect.

  • LintersGood at local rule violations, weak at explaining why a design movement matters.
  • Metrics dashboardsGood at trends, weak at telling a PR reviewer which file, boundary, or assumption needs inspection now.
  • AI review commentsUseful as observations, risky when treated as final judgment without evidence boundaries.
  • ArchSigTurns the change into a signal with evidence, status, scope, and next review action.

Outcomes

  • Better PR reviewReviewers see which boundaries moved, which responsibilities shifted, which assumptions changed, and where to focus attention.
  • Earlier architecture-debt detectionBoundary erosion, responsibility leakage, hidden business rules, runtime coupling, and negative attractor movement become reviewable signs before they harden.
  • Safer AI-agent developmentGenerated changes can be observed as architectural movement, connected to AIR, and returned to review as evidence-backed feedback.

AI-native development

AI agents change the review problem. They can produce large volumes of plausible code, including changes that pass tests while moving architectural responsibilities. ArchSig gives those changes an observation layer: static structure, semantic interpretation, runtime evidence, policy expectations, and review status are kept together.

  1. AI proposes a changeThe proposal or diff becomes an input artifact instead of a free-form conversation.
  2. ArchSig records evidenceStructural, semantic, and operational signals are attached to explicit evidence references.
  3. Review receives bounded signalsDangerous movements are sent back to reviewers; safe changes can move faster because the evidence is visible.

Not a linter

ArchSig can report policy violations, but its product surface is broader than pass/fail enforcement. The main object is the review signal: what changed, why it matters, what supports the reading, what is not established, and which observation should happen next.

ArchSig helps reviewers make architectural decisions with explicit evidence boundaries.

When to use it

  • Use it during PR reviewWhen a change touches a boundary, feature extension, adapter, dependency surface, or operationally sensitive path.
  • Use it during AI-assisted developmentWhen generated code needs architectural interpretation before it becomes routine project structure.
  • Use it during architecture governanceWhen teams need a durable record of observed signals, decisions, thresholds, incidents, and repairs.
