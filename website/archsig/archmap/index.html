Product surface

ArchMap is the ArchSig surface for architecture that lives in conventions, tests, runtime traces, PR context, and domain language. Static dependency extractors can see imports and files. ArchMap records the semantic structure a selected observer can justify from a bounded source universe.

  • It is a map candidateThe artifact describes how selected source refs map to architecture components, relations, roles, diagrams, and claims.
  • It is evidence-carryingEach item keeps source refs, preserved structure, forgotten structure, coverage, uncertainty, missing evidence, and non-conclusions.
  • It is AI-native without making AI a judgeLLMs and agents can author maps; validation, AIR projection, reports, and reviewers decide how the map is used.

Flow

The useful ArchMap path starts with a selected source universe and ends with review artifacts that still remember what was inspected and what was not inspected.

Map shape

An archmap-v0 artifact carries map identity, generator and prompt provenance, a source inventory, target candidates, mapping items, preservation claims, forgetting claims, and coverage records. The source inventory matters because a map can only speak about the code, docs, tests, traces, and PR metadata that were actually selected.

Field familyReview meaning
Source inventoryIncluded, excluded, unavailable, private, and known blind-spot refs for the review universe.
Mapping itemsObject, relation, semantic role, policy boundary, operation support, and semantic diagram candidates.
Preserves / forgetsThe structure the map keeps, such as ownership or contract preservation, and the detail it intentionally discards, such as path multiplicity.
Coverage and uncertaintyMeasured, assumed, unmeasured, unavailable, private, out-of-scope, and missing-evidence records.
Non-conclusionsExplicit reminders that the map is not proof of semantic preservation, global completeness, lawfulness, or theorem discharge.

Semantic structure

ArchMap is valuable when the architecture signal is not visible as a raw import edge. It can carry business-rule movement, workflow order, contract preservation, observation equivalence, semantic commutation claims, and nonfillability witnesses into the review pipeline.

  • semanticDiagramA selected workflow, API path, operation order, or implementation route whose behavior should be compared.
  • semanticCommutationClaimA bounded claim that a selected diagram commutes or fails to commute under the inspected evidence.
  • contractPreservationA candidate that an API contract, schema compatibility rule, test oracle, or domain invariant is preserved.
  • observationEquivalenceA user-visible result, event stream, state transition, or trace observation that appears equivalent in the selected universe.
  • nonfillabilityWitnessA review cue that a diagram does not close, a path lacks a filler, or a feature creates an obstruction candidate.

From ArchMap to AIR

AIR consumes ArchMap as a bounded input, not as final judgment. air-from-archmap projects source refs into AIR artifacts and evidence, maps object and relation candidates into reviewable architecture records, and keeps missing evidence, conflicts, coverage gaps, and non-conclusions visible for downstream reports.

This is the bridge that lets Semantic AIR talk about business rules and responsibility movement without pretending that generated text is architecture ground truth.

Boundary

The ArchMap boundary is a product feature. It separates the positive telemetry surface from the proof surface that belongs to AAT, Lean theorem packages, or human architectural decision making.

ArchMap can carryArchMap does not prove
Components, relations, semantic roles, source refs, and preservation candidates.Global architecture completeness or exact reconstruction of the repository.
Semantic diagrams, contract preservation candidates, observation equivalence, and obstruction witnesses.Semantic preservation, lawfulness, theorem precondition discharge, or zero-curvature conclusions.
Coverage, missing evidence, private context, unavailable refs, and non-conclusions.That missing evidence is safe, measured zero, or resolved by model confidence.

Commands and artifacts

  • archmapValidates supplied archmap-v0 JSON and emits archmap-validation-report-v0.
  • air-from-archmapProjects the validated map into AIR while preserving source refs, semantic coverage, missing evidence, conflicts, and non-conclusions.
  • archmap-v0The evidence-carrying map candidate authored from the selected source universe.
  • archmap-validation-report-v0The validation report that checks source inventory refs, claim boundaries, semantic coverage, conflicts, and formal-promotion guardrails.
