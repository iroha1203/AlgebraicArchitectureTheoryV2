Why it exists

Architecture rarely lives in one place. Import graphs show files and dependencies, but AAT analysis often needs source-grounded atom, molecule, semantic, and gap observations: which responsibility is present, which contract is visible, which workflow changed, and which evidence is missing.

ArchMap treats an LLM, agent, or reviewer as an observer. The observer reads a selected slice of code, docs, tests, config, traces, and PR context, then supplies archmap-observation-map-v0 with source refs and non-conclusions.

What reviewers get

  • A map of the changeWhere responsibilities appear to move, which boundaries are crossed, which workflows changed, and which components now depend on each other.
  • Evidence attached to each claimA valid map keeps pointers back to the files, tests, specs, traces, or PR context that supported each observation.
  • Gaps that stay visibleIf runtime behavior, private context, generated code, or framework convention was not inspected, the report keeps that gap visible instead of smoothing it away.
  • A bridge into ArchSig analysisArchMap feeds the analysis packet first; AIR, feature reports, policy decisions, theorem checks, and PR comments are downstream projections from that state.

What it can notice

ArchMap is most useful when the architecture signal is semantic rather than syntactic. It can point to the places where the code says one thing, the tests assume another, and the workflow tells the real story.

  • Business rule movementA rule that used to live in a domain service is now duplicated in a controller, job, migration, or client-facing adapter.
  • Workflow orderA checkout, approval, ingestion, or rollback path changed order even though the import graph barely moved.
  • Contract preservationAn API, schema, event, or test oracle appears to preserve a consumer-facing contract across a refactor.
  • Observation equivalenceTwo implementation paths appear to produce the same user-visible result, event stream, state transition, or trace observation.
  • Obstruction cuesA path may no longer close cleanly: a fallback is missing, a dependency leaks, or a new feature creates an architectural tension reviewers should inspect.

How it moves

The ArchMap path starts with selected source context and ends when ArchSig reads it under a selected LawPolicy.

Evidence trail

An archmap-observation-map-v0 artifact records what the observer read and which source refs support each observation. The source inventory is part of the artifact because a review map should remember its reading list.

PartWhy reviewers care
Atom observationsSource-grounded observations of already-theorized atom predicates. Observation does not create atom truth.
Molecule observationsGroups of atom observations, roles, and patterns that ArchSig may later read under a selected law.
Concern hintsReview cues from the observation layer. They are not obstruction circuits.
Observation gapsPrivate, unavailable, unsupported, dynamic, or missing evidence that must not be rounded to measured zero.

From map to analysis

ArchSig combines a validated ArchMap with LawPolicy to compute law-relative obstruction circuits, signature axes, flatness reading, repair candidates, and coverage gaps. AIR and Feature Report outputs are downstream projections from that analysis state.

Artifact reference

  • archmap-observation-map-v0The supplied source-grounded observation map.
  • archmapValidates supplied ArchMap JSON and emits archmap-validation-report-v0.
  • law-policy-v0Selects laws, witness rules, obstruction definitions, signature axes, coverage, and exactness assumptions.
  • archsig-analysisBuilds the analysis packet from ArchMap + LawPolicy.
