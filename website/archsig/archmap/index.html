Why it exists

Architecture rarely lives in one place. Import graphs show files and dependencies, but the structure AAT and SFT need is often semantic: which contract is preserved, which workflow should be compared for commutation, which responsibility moved, and which implementation detail the map intentionally forgets.

ArchMap treats an LLM, agent, or reviewer as a semantic observer. The observer reads a selected slice of code, docs, tests, traces, and PR context, then supplies a bounded map from implementation artifacts into architecture artifacts with evidence attached to each observation.

What reviewers get

  • A map of the changeWhere responsibilities appear to move, which boundaries are crossed, which workflows changed, and which components now depend on each other.
  • Evidence attached to each claimA valid map keeps pointers back to the files, tests, specs, traces, or PR context that supported each observation.
  • Gaps that stay visibleIf runtime behavior, private context, generated code, or framework convention was not inspected, the report keeps that gap visible instead of smoothing it away.
  • A bridge into ArchSig reportsArchMap feeds AIR, feature reports, policy decisions, theorem checks, and PR comments with architecture context that static extraction alone cannot see.

What it can notice

ArchMap is most useful when the architecture signal is semantic rather than syntactic. It can point to the places where the code says one thing, the tests assume another, and the workflow tells the real story.

  • Business rule movementA rule that used to live in a domain service is now duplicated in a controller, job, migration, or client-facing adapter.
  • Workflow orderA checkout, approval, ingestion, or rollback path changed order even though the import graph barely moved.
  • Contract preservationAn API, schema, event, or test oracle appears to preserve a consumer-facing contract across a refactor.
  • Observation equivalenceTwo implementation paths appear to produce the same user-visible result, event stream, state transition, or trace observation.
  • Obstruction cuesA path may no longer close cleanly: a fallback is missing, a boundary leaks, or a new feature creates an architectural tension reviewers should inspect.

How it moves

The ArchMap path starts with the context selected for review and ends with artifacts that remember both the evidence and the gaps.

Evidence trail

An archmap-v0 artifact records what the agent read and how each observation maps into architecture language. The source inventory is part of the artifact because a review map should remember its reading list.

PartWhy reviewers care
Source inventoryThe code, docs, tests, traces, and PR metadata selected for the review, plus excluded or unavailable context.
Mapping itemsObserved components, relations, roles, policy boundaries, workflows, and operation support records.
Evidence refsLinks from each observation back to concrete source refs, so the reviewer can inspect the basis.
Coverage notesWhat was measured, assumed, unmeasured, private, unavailable, or outside the selected review slice.

From map to AIR

air-from-archmap projects a validated ArchMap into AIR. Source refs become AIR artifacts and evidence; object and relation mappings become reviewable architecture records; semantic diagrams and obstruction cues become claims that downstream reports can show.

This is what lets Semantic AIR talk about business rules, responsibility movement, contracts, and workflow behavior while keeping the evidence trail attached.

Artifact reference

  • archmap-v0The supplied map candidate authored from the selected source universe.
  • archmapValidates supplied archmap-v0 JSON and emits archmap-validation-report-v0.
  • archmap-validation-report-v0Reports source ref checks, coverage notes, conflicts, and promotion guardrails for the supplied map.
  • air-from-archmapProjects the validated map into AIR for feature reports, policy decisions, theorem checks, and PR comments.
