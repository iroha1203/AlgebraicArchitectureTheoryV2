Read the output like a review map.

Reading order

  1. Identify the artifact typeArchMap observation, LawPolicy, ArchSig analysis packet, validation report, AIR projection, feature report, policy decision, PR comment, adapter evidence, or FieldSig handoff artifact.
  2. Check schema and validationDo not interpret report conclusions from invalid artifacts.
  3. Find the selected policyFor analysis packets, read selectedLawPolicyRef before reading obstruction or signature output.
  4. Read gaps before conclusionsObservation gaps, coverage gaps, unsupported constructs, and unknown remainder are not measured zero.
  5. Follow evidence refsA review signal should point to files, observations, laws, policies, runtime evidence, semantic observations, or missing evidence.

Field lookup

Field familyHow to read it
schemaVersionCompatibility contract for the artifact format.
selectedLawPolicyRefThe law universe, obstruction definitions, axes, and coverage requirements used by this analysis.
moleculeReadingsHow ArchSig grouped ArchMap observations into law-readable architectural molecules.
obstructionCircuitsLaw-relative analysis outputs. Read the linked observations and policy rule before treating a circuit as review-relevant.
signatureAxesMulti-axis architecture readings, each with evidence and measurement status.
repairCandidatesOperation-shaped review suggestions, not automatic edits or merge gates.
coverageGapsUnknown, unavailable, or out-of-scope evidence that must remain visible.
evidenceRefsWhere the signal came from.
nonConclusionsWhat the artifact is explicitly not claiming.

Analysis packet

archsig-analysis-packet-v0 is the first artifact to read in the LLM-native workflow. It records the selected LawPolicy, molecule readings, law-relative obstruction circuits, signature axes, flatness reading, repair candidates, coverage gaps, evidence boundaries, LLM interpretation notes, and non-conclusions.

Validation report

Validation is the first quality check on the artifact. It should be read before diff or report interpretation. A validation warning may mean the selected universe is still useful, but it must travel with the review signal.

Adapter evidence

Adapter evidence records bounded import-graph structure when that structure matters. Read it as supporting evidence for analysis or comparison, not as a replacement for ArchMap or a claim of complete architecture extraction.

Feature report

A feature report is the main reader-facing architecture review artifact. It should explain the feature movement, witnesses, evidence, missing evidence, theorem precondition status, and suggested review action.

Policy and PR comments

A policy decision applies organization policy to the report. A PR comment translates the decision and evidence into reviewer language. The comment should be brief enough for PR review but specific enough to keep the source context intact.
