Reading order

Start with the artifact type, then read status fields before numeric values. This prevents the most common mistake: treating missing or unmeasured evidence as if it were a safe result.

  1. Check schema and validation status Confirm that the file is the artifact you think it is and that validation did not fail.
  2. Read measurement status Look for metricStatus, metricDeltaStatus, unmeasuredAxes, and coverageGaps.
  3. Inspect changed or risky axes Use worsenedAxes, improvedAxes, witness refs, and evidence diffs to choose what to review.
  4. Read non-conclusions They tell automation and reviewers what the report refuses to prove.

Sig0

Sig0 is the first architecture reading for one revision. It is the place to inspect components, dependency edges, raw signature values, policy violations, runtime evidence, and unsupported constructs.

  • signature.hasCycle true means the observed dependency graph has a static cycle.
  • signature.sccMaxSize Large strongly connected components are change-propagation candidates.
  • signature.maxDepth Deep chains make understanding and testing the affected region more expensive.
  • signature.fanoutRisk High fanout points to modules whose changes may touch many dependents.
  • metricStatus Read this before trusting any zero value.

Validation report

Validation asks whether a Sig0 file is coherent enough for the next command. If validation fails, fix the artifact or input boundary before reading the report as architecture evidence.

  • Duplicate components Two records describe the same component identity.
  • Edge closure An edge points to a component that is not in the selected local universe.
  • External targets A dependency leaves the selected universe. This may be valid, but it should be visible.
  • Policy metric status Policy axes should say whether they were measured, not simply show zero.

Signature diff

A diff report is the first artifact most PR reviewers should read. It answers: which axes worsened, which improved, which are not comparable, and which evidence changed.

If deltaSignatureSigned.axis is null, do not call the axis improved or worsened. Read the metric delta status first.

Feature report

The Feature Extension Report is the main reviewer-facing artifact. It summarizes whether a feature looks split under the selected observation, which witnesses explain concerns, and which repair suggestions are available.

  • splitStatus Whether the feature is read as split, non-split, blocked, or unknown under the selected universe.
  • introducedObstructionWitnesses Measured witness candidates that explain the review concern.
  • coverageGaps What the report did not observe.
  • repairSuggestions Advisory repair candidates, not automatic fixes.

Policy and PR comments

Policy decisions translate reports into local review actions. PR comment summaries translate the same evidence into Markdown that a reviewer can scan.

  • Fail The local policy says the PR should be blocked until the item is resolved or explicitly accepted.
  • Warn The reviewer should inspect the item, but the policy does not automatically block.
  • Advisory The report gives context or calibration data without requiring action.
  • Unmeasured No decision should pretend the missing axis was safe.

A good PR comment should show the changed axis, the evidence behind it, the missing evidence, and the action requested from the reviewer.
