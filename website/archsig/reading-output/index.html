Reading order

  1. Identify the artifact typeSig0, validation, snapshot, diff, AIR, feature report, policy decision, PR comment, or feedback artifact.
  2. Check schema and validationDo not interpret report conclusions from invalid artifacts.
  3. Find the selected universeRoot, source roots, policy, runtime edge input, PR metadata, or proposal source.
  4. Read measured status before valuesValues without measurement status are easy to overread.
  5. Follow evidence refsA review signal should point to files, edges, policies, runtime evidence, semantic observations, or missing evidence.

Field lookup

Field familyHow to read it
schemaVersionCompatibility contract for the artifact format.
components, edgesThe finite architecture universe selected by the scan or adapter.
signatureArchitecture axes. Read together with metric status.
metricStatusWhether an axis was measured, unmeasured, unsupported, private, unavailable, or out of scope.
evidenceRefsWhere the signal came from.
nonConclusionsWhat the artifact is explicitly not claiming.

Sig0

Sig0 is the first architecture signature emitted by ArchSig. Think of it as the baseline observation of your repository. It records the selected component universe, observed edges, signature axes, metric status, policy status, runtime evidence references, and unsupported constructs.

Validation report

Validation is the first quality check on the artifact. It should be read before diff or report interpretation. A validation warning may mean the selected universe is still useful, but it must travel with the review signal.

Signature diff

Signature diff compares two bounded observations. The key questions are which axes worsened, which improved, which were not comparable, and which deltas are unmeasured because one side lacked evidence.

Feature report

A feature report is the main reader-facing architecture review artifact. It should explain the feature movement, witnesses, evidence, missing evidence, theorem precondition status, and suggested review action.

Policy and PR comments

A policy decision applies organization policy to the report. A PR comment translates the decision and evidence into reviewer language. The comment should be brief enough for PR review but specific enough to preserve the evidence boundary.
