Local Law Packages

Projection, observation, substitutability, abstraction, and flatness are local theorem packages with explicit assumptions. AAT uses them to state local contract soundness without promoting that soundness into global layering, runtime flatness, or semantic completeness.

Reading model

AAT does not say

"SOLID proves the architecture is clean." A local contract law is not a global layering theorem, a runtime telemetry completeness theorem, or a semantic equivalence theorem for all possible clients.

AAT says

Under this selected projection, observation, client context universe, measurement boundary, and proof-carrying contract, these local projection, LSP, and DIP obligations hold. Stronger claims need their own theorem boundary.

projection
A map from concrete components or dependencies into a selected abstract graph.
observation
The selected output, behavior, trace, or diagnostic read from a concrete implementation.
local contract
A proof-carrying package that preserves selected local invariants such as projection soundness, LSP compatibility, and DIP compatibility.
non-conclusion
The global property that remains outside the theorem, even when the local package is proved.

Projection and observation

Plain reading. A projection says which concrete things count as the same abstraction. An observation says which selected behavior is visible for the claim.

Local law packages become theorem-grade only after the projection, observation, equality, and selected context universe have been named. The schematic kernel inclusion below is useful only relative to that boundary.

Definition 7.1. A local projection-observation boundary consists of an InterfaceProjection, an Observation, an observation equivalence relation, and a selected universe of concrete objects or client contexts to which the claim is allowed to apply.

Lean status defined only Anchored by InterfaceProjection, Observation, ObservationallyEquivalent, lspCandidatePairs, and ComponentUniverse. Source: Chapter 7 and the Projection / Observation entries in the Lean theorem index.

Proposition 7.2. If a selected observation factors through the selected projection, then same-abstraction pairs have equivalent selected observations. In finite measured universes, this supports zero LSP violation-count bridges under the stated closure assumptions.

Lean status proved anchors See lspCompatible_of_observationFactorsThrough, lspViolationCount_eq_zero_of_observationFactorsThrough, mem_lspViolationPairs_iff, and lspCompatible_of_lspViolationCount_eq_zero.

Proof idea. Observation factorization means the observation is determined by the abstract representative. Two concrete implementations in the same abstract fiber therefore receive the same selected observation. The finite violation-count bridge adds the explicit measurement-universe closure premise needed to read zero counted violations as graph-level LSP compatibility.

Projection / observation claim

Kernel inclusion is local to a selected observation model

proved bridge
Assumptions
Selected projection, selected observation, same-abstraction pair coverage, observation equivalence, and finite measurement closure where counts are used.
Boundary
The result is about selected observations in the named universe, not every possible client, effect, or runtime trace.
Non-conclusion
It does not prove global decomposability, projection completeness, semantic completeness, or runtime flatness.

Projection and DIP

Plain reading. DIP is not one theorem strength. Sound projection, complete projection, exact projection, and representative stability license different conclusions.

Definition 7.3. ProjectionSound states that concrete dependency evidence is represented in the abstract graph. ProjectionComplete states that abstract edges have concrete representatives. ProjectionExact bundles soundness and completeness. RepresentativeStable states the stability needed for quotient-style readings.

Lean status defined only Anchored by ProjectionSound, ProjectionComplete, ProjectionExact, RepresentativeStable, DIPCompatible, and StrongDIPCompatible.

Proposition 7.4. Exact projection together with representative stability yields strong DIP compatibility; strong DIP compatibility exposes exact projection, representative stability, quotient well-definedness, and ordinary DIP compatibility.

Lean status proved anchors See strongDIPCompatible_of_projectionExact_representativeStable, projectionExact_of_strongDIPCompatible, representativeStable_of_strongDIPCompatible, quotientWellDefined_of_strongDIPCompatible, and dipCompatible_of_strongDIPCompatible.

Proof idea. Lean keeps the strength assumptions as separate fields and proves accessor theorems from the bundled strong-DIP package. The accessors prevent a soundness-only projection from being silently read as an exact quotient or as an exhaustive abstraction.

DIP claim

Projection strength determines the licensed abstraction reading

proved bridge
Assumptions
Concrete graph, interface projection, abstract graph, exactness or soundness premise, and representative-stability premise where quotient readings are used.
Boundary
The claim is dependency-projection local; it says what the selected abstraction graph represents.
Non-conclusion
It does not say the abstract graph is acyclic, layered, complete for runtime dependencies, or semantically complete.
Docs source
Projection / DIP index and the Chapter 7 proof-obligation ledger.

Observation and LSP

Plain reading. LSP compatibility is selected substitutability: it is stated for named abstractions, observations, equivalences, and client contexts.

AAT reads LSP-style claims through observation. Two concrete implementations may be substitutable for checkout totals, rounding traces, and selected errors without being substitutable for every client, timing property, or hidden effect.

Definition 7.5. LSPCompatible is the selected observation compatibility predicate for implementations in the same abstract fiber. Its finite measurement form counts selected same-abstraction pairs whose observations diverge.

Lean status defined only Anchored by LSPCompatibleAt, LSPCompatible, LSPObstruction, lspViolationPairs, and lspViolationCount.

Proposition 7.6. In the current finite theorem package, LSP compatibility is equivalent to absence of selected LSP obstruction witnesses; when the finite measurement universe closes the relevant same-abstraction pairs, zero violation count yields LSPCompatible.

Lean status proved anchors See lspCompatible_iff_noLSPObstruction, lspViolationCount_eq_zero_of_lspCompatible, lspViolationCount_eq_zero_of_observationFactorsThrough, and lspCompatible_of_lspViolationCount_eq_zero.

LSP claim

Violation counts are theorem claims only inside the measured universe

proved bridge
Assumptions
Selected finite universe, pair coverage for same-abstraction candidates, observation equivalence, and decidable observation comparisons.
Boundary
The count is about selected pairs and selected observations. Unmeasured clients remain outside the claim.
Non-conclusion
No claim follows about all behavioral semantics, all effects, all clients, or every runtime dependency.
Docs source
Observation / LSP bridge ledger and Lean index.

Local contract packages

SOLID-style laws are useful because they are local enough to state with explicit theorem boundaries. In AAT, local contract replacement packages preserve selected projection soundness, LSP compatibility, and DIP compatibility. The package also records the global layering claim that it does not prove.

Proposition 7.7. A proof-carrying LocalReplacementContract gives the target state projection soundness, LSP compatibility, and DIP compatibility selected by the local contract invariant family. As a DesignPattern, the local contract package records that unconditional Decomposable or StrictLayered does not follow.

Lean status proved anchors See projectionSound_of_localReplacementContract, localReplacementOperation_preserves_lspCompatible, localReplacementOperation_preserves_dipCompatible, localContractDesignPattern_closure_law, and localContractDesignPattern_records_nonConclusion.

Proof idea. The local replacement operation carries the target-side contract as evidence. Lean extracts the selected invariants from that evidence, then places the operation family and invariant family into the general DesignPattern closure schema. The non-conclusion is a field of the pattern, so the global layering boundary travels with the package.

Principle reading Status Assumptions Boundary Non-conclusion
LSP proved for selected observation bridges Selected abstraction fiber, client contexts, observation equivalence, and finite pair coverage when measured. Local substitutability for the named observation universe. Not all behavioral semantics or all clients.
DIP proved for projection / local contract bridges Selected concrete graph, interface projection, abstract graph, and representative stability where strong readings are used. Local dependency abstraction soundness or exactness. Not global acyclicity, layering, or runtime dependency completeness.
SOLID as a family defined only / proved for bounded representative packages Each principle is read through its selected local invariant family. Local contract layer; other design layers require separate packages. SOLID is not a universal architecture theorem.

SOLID belongs to the local contract layer in this taxonomy; Layered and Clean Architecture are global structural layers, state-transition patterns are algebraic layers, and runtime protection or distributed convergence require their own evidence.

Three-layer flatness

Plain reading. Static flatness, runtime flatness, and semantic flatness are different axes. A static split is not a proof of runtime or semantic flatness.

Definition 8.1. ArchitectureFlatWithin is the bounded, coverage-aware flatness predicate that bundles selected static, runtime, and semantic flatness with required-axis coverage and exactness boundaries.

Lean status defined only Anchored by ArchitectureFlatnessModel, StaticFlatWithin, RuntimeFlatWithin, SemanticFlatWithin, NoUnmeasuredRequiredAxis, and ArchitectureFlatWithin.

Proposition 8.2. A lawful split-feature or extension package yields bounded ArchitectureFlatWithin only when extension coverage, static side conditions, runtime coverage / flatness, semantic coverage / flatness, and recorded non-conclusions are supplied.

Lean status proved anchors See LawfulExtensionPreservesFlatness, LawfulExtensionPreservesFlatness_of_runtimeSemanticSplitPreservation, architectureFlatWithin_of_splitFeatureExtensionWithin, and splitFeatureExtensionWithin_recordsNonConclusions.

Non-conclusion 8.3. Static split, static flatness, projection soundness, or local contract compatibility does not by itself imply runtime flatness, semantic flatness, global ArchitectureFlat, extractor completeness, telemetry completeness, or semantic universe completeness.

Lean status proved / recorded boundary Anchored by SplitFeatureExtensionWithinNonConclusions, splitFeatureExtensionWithin_recordsNonConclusions, GlobalFlatCertificate.nonConclusions_recorded, and the static / semantic counterexample package.

Flatness claim

Flatness is a three-axis theorem package, not a static split slogan

proved bridge
Assumptions
Static lawfulness, runtime coverage and protection, semantic coverage and commutation, no unmeasured required axis, and explicit non-conclusions.
Boundary
The conclusion is bounded ArchitectureFlatWithin unless exhaustive coverage and exact observation promote it through a separate certificate.
Non-conclusion
No runtime telemetry completeness, global semantic completeness, extractor completeness, or automatic global flatness follows.

Example and counterexample

Example 8.4. In a coupon extension, replacing a discount implementation can be LSP-sound for selected checkout clients when observed totals, rounding traces, and selected error behavior remain equivalent. That local theorem still needs separate static, runtime, and semantic packages before it can be read as bounded flatness.

Lean status proved representative package Related anchors include the coupon / discount semantic examples, ThreeLayerFlatnessPositiveExample.architectureFlatWithin, ThreeLayerFlatnessPositiveExample.noUnmeasuredRequiredAxis, and ThreeLayerFlatnessPositiveExample.recordsNonConclusions.

Counterexample 8.5. A package can satisfy strong DIP-style projection conditions and selected LSP compatibility while the abstract graph remains cyclic and therefore not Decomposable. This is the precise failure mode that prevents SOLID from being treated as a universal architecture theorem.

Lean status proved counterexample Anchored by StrongAbstractCycleComponent.strongDIPCompatible, StrongAbstractCycleComponent.lspCompatible, StrongAbstractCycleComponent.abstractGraph_not_decomposable, and StrongAbstractCycleComponent.strongDIPCompatible_and_not_decomposable.

Scenario What is proved or measured What remains outside
Coupon replacement Selected checkout observations can remain equivalent under the named client-context universe. Background jobs, caches, callbacks, hidden runtime dependencies, and unmeasured semantic diagrams.
Strong abstract cycle Projection exactness, representative stability, strong DIP compatibility, and LSP compatibility can hold. Global decomposability and abstract-layer acyclicity.
