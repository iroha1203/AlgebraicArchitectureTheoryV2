Related Work and Novelty

Boundary note

This page states how AAT should be read relative to adjacent work. It does not claim that AAT replaces architecture description standards, review methods, model checking, or empirical architecture metrics.

Positioning

AAT is not a new notation for every architecture document, a replacement for scenario-based review, or a single quality metric. It is a local algebra for stating what architecture operation is being considered, which invariant family is in scope, which obstruction witnesses count as failures, and which theorem boundary licenses a conclusion.

ISO 42010

ISO/IEC/IEEE 42010 is the natural neighbor for architecture description: stakeholders, concerns, viewpoints, views, and model kinds organize how an architecture is documented. AAT uses that kind of boundary discipline, but its object is not primarily an architecture description package.

AAT asks a different question. Given a selected architecture object and observation boundary, which invariant family is required, which witness universe can expose a violation, which signature coordinates were measured, and which theorem boundary prevents over-reading the result?

Neighbor concept AAT reading Boundary
Viewpoint / view Selected observation context and evidence boundary. A view does not by itself prove invariant preservation.
Model kind Carrier, graph, diagram, metric, or report shape used by a theorem or tool claim. Different model kinds require explicit bridge claims.
Concern Invariant family, required law, or signature axis. A concern is not automatically a measured zero axis.

ATAM

ATAM and related scenario-based methods organize quality attribute reasoning around scenarios, risks, sensitivity points, and tradeoffs. AAT does not replace that review practice. It supplies a formal claim discipline for the parts of a review that can be stated as bounded laws, witnesses, signature axes, and theorem packages.

  • Quality attribute scenario AAT can read a scenario as a selected observation context plus required invariant family when the evidence is precise enough.
  • Risk and tradeoff AAT represents selected risks as obstruction witnesses or nonzero signature axes; tradeoffs are axis-local movements, not a global score.
  • Review conclusion AAT requires the theorem boundary or artifact boundary to say what the conclusion does not cover.

Architecture metrics

Architecture metrics often summarize dependency, coupling, complexity, volatility, or risk. AAT deliberately does not make quality a single scalar. `ArchitectureSignature` is a multi-axis diagnostic whose coordinates carry measurement status, evidence, universe, and non-conclusions.

A lower value on one measured axis is not a total ordering of architectures. Static split, semantic commutativity, runtime exposure, projection exactness, and empirical outcomes remain separate axes unless a theorem or empirical protocol connects them.

Graph analysis

AAT reuses graph-level dependency analysis for selected static and runtime relations: reachability, layering, acyclicity, paths, adjacency-style representations, and finite metric calculations. The distinctive move is to prevent those graph facts from becoming stronger architecture claims without a bridge.

  • Reused Directed dependency graphs, finite component universes, reachability, witness counts, path and matrix-style representations.
  • Added by AAT Law families, obstruction witnesses, signature status, and theorem boundaries that state which graph fact supports which architecture claim.
  • Non-conclusion A clean selected graph does not imply complete extraction, semantic flatness, runtime safety, or empirical cost improvement.

Formal methods

Formal methods, model checking, type systems, theorem proving, and refinement techniques provide ways to verify precise specifications. AAT uses theorem proving as an anchor for selected architecture statements, but the whole theory is not reduced to a single model-checking problem.

The current Lean role is inspectable formal anchoring: selected finite static lawfulness, selected required witnesses, and selected required signature axes under explicit coverage and exactness assumptions. Runtime extraction completeness, semantic coverage, and empirical outcome claims remain outside that anchor unless separate theorem packages are supplied.

Category-theoretic models

AAT uses categorical vocabulary where it clarifies structure, composition, projection, lifting, and obstruction. It does not claim that every architecture concern is solved by naming a category, and its current `ComponentCategory` is intentionally thin: it forgets path counts and walk lengths.

Quantitative path information belongs to `Walk`, `Path`, adjacency matrices, finite metric representations, or future free-category constructions. This separation keeps categorical structure from swallowing measurement semantics.

Novelty claim

The novelty claim is not that AAT invented architecture views, scenario review, dependency graphs, software metrics, theorem proving, or category theory. The claim is that AAT packages them into a bounded architecture-reasoning discipline where design principles are operations that preserve or improve invariant families, failures are explicit obstruction witnesses, and quality is read as a multi-axis signature under theorem boundaries.

Area What AAT reuses What AAT contributes
Architecture description Views, model boundaries, and concern separation. Theorem boundary, witness universe, and signature status attached to claims.
Architecture evaluation Scenario, risk, and tradeoff vocabulary. Bounded law and obstruction readings that prevent over-generalized review conclusions.
Metrics Finite measurements and axis-specific readings. Multi-axis ArchitectureSignature with measured, unmeasured, out-of-scope, and non-conclusion states.
Graphs and formal methods Dependency graphs, reachability, theorem proving, and bridge lemmas. Architecture-specific packaging of invariant preservation, witnesses, and proof boundaries.
