How do finite skeletons make theorem packages and counterexamples readable?

Canonical examples

A canonical AAT example is a finite architecture presentation that exposes one operation, one selected evidence package, and the witnesses read through that package. The component names are intentionally small, so the proof shape can become the main object of attention.

Definition 15.0. A canonical AAT example is a finite presentation consisting of an architecture object, an operation or extension, an invariant family under test, a witness universe, measured signature axes, a theorem package, and recorded counterexamples or non-conclusions.

  • Static witness A selected direct dependency that bypasses the declared interface route.
  • Semantic witness A selected observation difference between two paths through the feature.

Example template

Each canonical example follows the same proof-relevant shape: an architecture object, an operation, a witness universe, a signature reading, and the conclusions attached to that reading.

The same template can carry static, semantic, and runtime readings while keeping each axis attached to its own evidence.

Finite extension skeleton

The running finite skeleton uses a feature component, a declared interface, a core service, and an internal adapter endpoint. The example is small enough to inspect, and rich enough to display static, semantic, runtime, and signature readings side by side.

Example 15.1. The obstructed extension contains the selected hidden static dependency witness CouponService -> PaymentAdapter.internalCache. The repaired extension routes the selected interaction through the declared port and restores the selected static split for that finite skeleton.

Proof idea. The finite skeleton names the core components, the feature component, the declared port, and the internal endpoint. Lean exhibits the hidden edge as a selected witness for the obstructed extension and proves the selected static split obstruction. The repaired extension removes that selected direct edge and supplies the static split package again.

  • Object A selected finite universe containing a feature component, a declared interface, core components, and selected relations among them.
  • Declared route The feature interaction passes through the declared interface, with a feature view and selected section laws recording how the feature is observed.
  • Static obstruction A direct dependency from the feature component into an internal adapter endpoint records the selected static split obstruction.
  • Semantic obstruction Two implementation paths produce different observations under the selected client context.
  • Runtime obstruction A retry, callback, invalidation, or protection witness appears on the selected runtime surface.
  • Scope Static, semantic, runtime, and repository-extraction readings are separate evidence axes.

Read as an AAT object, the skeleton has six visible parts. Each part contributes one piece of evidence and names the surface on which that evidence is read.

  • ComponentUniverse CheckoutService, CouponService, PaymentPort, selected payment core / adapter components, and selected static / runtime relations form the finite universe.
  • Declared static route CheckoutService -> CouponService, CouponService -> PaymentPort, and PaymentPort -> PaymentAdapter give the interface path under the declared policy.
  • Static witness set A direct CouponService -> PaymentAdapter or CouponService -> PaymentCache edge supplies the selected witness when it bypasses the interface route.
  • Semantic observation Total, rounding trace, error result, or discount ordering can be read under selected checkout contexts and observation bridges.
  • Runtime surface Retry, callback, cache invalidation, or protection evidence lives on the runtime surface and is read through its own package.
  • Signature coordinates Static obstruction count, semantic witness status, runtime exposure status, and residual coverage gap become axis-local readings.

Signature reading

The finite skeleton is read as a multi-axis diagnostic. The static axis is measured inside the finite witness universe, while semantic and runtime axes carry their own evidence and measurement status.

Signature reading 15.2. The snapshot separates some 0, some n, and unmeasured none axes. A repaired selected static axis can be available-and-zero while a semantic rounding-order axis is unmeasured or measured nonzero, and while runtime exposure belongs to a runtime evidence package.

  • static_hidden_interaction The obstructed extension reads as measured_nonzero(1) for the selected direct cache edge. After repair, the selected static witness universe reads as measured_zero. This is Lean-backed when the finite witness list and theorem package are cited.
  • semantic_curvature The static-only report leaves this axis unmeasured. A selected semantic diagram can supply measured_nonzero(delta), including the rounding-order diagram in the repaired extension.
  • runtime_exposure The static-only report leaves runtime exposure unmeasured. Runtime evidence can then supply retry, callback, cache behavior, or protection readings on its own surface.
  • Repository completeness Repository extraction is recorded as out_of_scope for both snapshots and belongs to a separate evidence problem.

Counterexamples

Static repair can satisfy selected static split while a semantic diagram carries a non-commutation witness. A repair can reduce a selected static obstruction while transferring risk to runtime or semantic axes. SOLID-style local contracts can sit beside distinct structural witnesses on the global graph.

Counterexample 15.3. There is a selected repaired static skeleton that satisfies selected static flatness while the selected semantic diagram carries a non-commutation witness. The example separates static flatness from semantic flatness and global architecture flatness.

Proof idea. Lean reuses the repaired finite static skeleton for the static side and supplies a selected semantic diagram whose two paths differ under the chosen observation. The static package and the semantic obstruction coexist, displaying the separation between selected static flatness and selected semantic flatness.

Counterexample 15.4. A repair step can decrease the selected static obstruction measure while increasing a runtime or transferred-complexity axis. Repair monotonicity is therefore read at the selected measure; other architecture axes are read separately.

AAT readings

AAT is an algebraic framework for software architecture. The examples on this page show static, semantic, repair, and signature readings as explicit evidence packages that can be inspected, cited, and recombined.

Non-conclusion 16.1. The examples on this page prove or illustrate package-aware readings under selected evidence: finite static witnesses, semantic diagrams, runtime observations, repair transfers, and multi-axis signature readings.

Layered reading

The finite skeleton exposes the theorem package. A static report measures whether a feature bypasses a declared interface. A semantic report measures whether selected observations commute. A runtime report measures retry, cache, or callback exposure. Together they form connected readings over separate axes.

SOLID readings

SOLID-style contracts are local law packages. A component can satisfy selected substitution tests and interface separation checks while the abstract dependency layer is cyclic. Conversely, a layered graph can be acyclic while selected LSP observations carry their own witness.

Counterexample 15.5. There are representative finite components whose selected DIP or strong DIP compatibility and LSP-style observations hold, while the abstract dependency graph is not Decomposable. Local SOLID-style compatibility and global strict layering are different readings.

  • Local success A selected client context or interface contract satisfies its observation law.
  • Global structure Layering, decomposition, or acyclicity can carry a distinct structural witness beyond the local contract.
  • Cross-layer bridge A theorem package connects one law family to another when such a bridge is available.

Chapter summary

The examples show how AAT makes finite architecture skeletons readable. A static witness, semantic diagram, runtime observation, repair measure, or signature coordinate can be placed in its own evidence record and then related to the other readings through explicit bridges.

This is the practical role of examples in the theory. They demonstrate how selected evidence becomes a theorem-facing object, how counterexamples separate local and global readings, and how architecture quality can be read across several axes while preserving the structure of those axes.

Terms

finite skeleton
A small selected architecture object used to expose the theorem package, witness family, or counterexample shape.
layered reading
A reading that separates static, semantic, runtime, repair, and signature evidence into distinct axes.
local success
A selected contract, client context, or interface law satisfies its own observation condition.
global structure
The selected dependency, layering, decomposition, or structural reading of the architecture object.
