Can software architecture be observed quantitatively as a multi-axis signature?

Reading contract

AAT quantifies selected coordinates of an ArchitectureSignature. Each coordinate tells a reader what was observed, which architectural law it touches, how strongly it is supported, and how it can be compared with another revision. The result is a vocabulary for asking where architecture improves, where pressure accumulates, and which repair direction has measurable support.

Axis A.1. AAT quantifies obstruction, preservation, coverage, and comparison coordinates by pairing each axis with its selected universe, measurement status, and axis-local order.

Axis catalog

The table below is a public reading catalog. It shows the architectural quantities AAT can make visible across the mathematical theory, signature artifacts, reports, theorem-precondition checks, and architecture evolution readings.

Axis family What is quantified Typical coordinate What it enables
Static structure Finite dependency shape and forbidden structural evidence. component count, relation count, cycle flag, SCC size, max depth, forbidden edge count. Layering, decomposition, cycle pressure, and forbidden-edge review become measurable.
Propagation and concentration How dependency pressure spreads from a component and where it concentrates. fanout risk, max fanout, reachable cone size, SCC excess, weighted SCC risk. Hotspots, propagation cones, and weighted structural pressure become visible as separate coordinates.
Boundary policy Edges or accesses that cross a declared architectural boundary. boundary violation count, direction policy, policy evidence status. Architectural boundaries become reviewable policy coordinates instead of informal diagrams.
Abstraction and interface Dependency on abstraction, concrete leakage, projection loss, and substitution evidence. DIP violation, LSP counterexample, projection mismatch, contract coverage level. Local contract pressure becomes visible at projection, substitution, and dependency seams.
Feature extension How an operation changes the architecture and where extension obstruction appears. added component count, changed relation count, split status, hidden interaction witness. Feature work can be read as inherited, local, and interaction change instead of one undifferentiated diff.
Repair and transfer Selected obstruction decrease and possible movement into another axis. eliminated witness count, residual witness count, transfer candidate, repair adoption status. Repair gains and transferred pressure can be tracked as separate architectural movements.
Architecture evolution Bounded sequences of feature, drift, repair, migration, policy, runtime, and semantic steps. path length, transition kind count, lawful-step status, target-flat status, reported drift witness. An architecture change becomes a typed path whose steps, endpoints, and witnesses can be inspected.
Signature dynamics How observed signatures move across an architecture evolution path. signature trajectory, per-step delta sequence, endpoint delta, recurrent region witness. Architectural movement can be read as a trajectory through signature space.
Runtime dependency Runtime edges, protected paths, unprotected paths, and exposure witnesses. runtime edge count, protected path count, unprotected path count, exposure witness. Runtime exposure becomes a signature coordinate alongside static structure.
Semantic and homotopy Whether paths, diagrams, and observations agree under the selected semantic reading. semantic diagram count, filler claim, nonfillability witness, observation mismatch. Equivalent-looking paths can be compared by the semantic diagrams they preserve or fail to fill.
State and effect Command, event, retry, compensation, snapshot, migration, and effect-law obstruction. effect-pair order, handler coverage, authority crossing, migration alignment. State-transition and effect design become algebraic law surfaces with measurable witnesses.
Graph and matrix Finite graph readings transported into walk, matrix, nilpotence, or spectral coordinates. closed-walk absence, adjacency nilpotence, walk count, propagation bound. Structural facts can move between graph, walk, matrix, and analytic representations.
Analytic valuation Numeric or ordered values assigned to selected witnesses or representations. obstruction valuation, aggregate drift, zero-preserving or zero-reflecting value. Witness families can be summarized by ordered values while retaining their architectural meaning.
Coverage and exactness Which observations are available and which assumptions are ready for discharge. measured axes, measurement backlog, construct coverage, exactness assumption list. Reviewers can distinguish available evidence, future measurement work, and assumption discharge.
Theorem boundary Whether a measured report can discharge a theorem package precondition. discharged assumption, precondition work item, formal readiness status. Measured artifacts can be organized into the precondition checklist for formal theorem use.

Tool surface

ArchSig gives the catalog an executable surface. It reads Lean and Python import graphs, validates Sig0 artifacts, compares signature snapshots, builds AIR records, checks theorem preconditions, summarizes feature extensions, and records policy decisions. These reports let architecture review move from prose judgement to axis-level evidence.

  • Sig0 Records components, edges, policies, signature values, metric status, runtime evidence, supported constructs, and runtime dependency graph entries.
  • Feature Extension Report Reads split status, preserved and changed invariants, introduced and eliminated witnesses, complexity transfer candidates, theorem checks, and repair suggestions.
  • Theorem precondition check Organizes discharged assumptions, precondition work, coverage evidence, exactness evidence, and formal-claim readiness.

Comparison

Signature comparison is axis-local. A before/after report can show which coordinates improved, worsened, or stayed stable when both revisions measure the same axis under a compatible schema, unit, projection, coverage, and order. This makes architectural movement readable as a vector of local changes.

Precision

AAT's precision comes from keeping each coordinate attached to its observation universe, evidence source, measurement status, and theorem package. The same page can therefore show a measured zero, a measured obstruction, a comparable delta, and a theorem-ready precondition as different useful pieces of architectural knowledge.

This is the point of the axis catalog. A structural count, a propagation cone, a signature trajectory, a runtime exposure, a semantic mismatch, an effect-law witness, and a theorem-precondition status can sit in one signature as distinct coordinates. The reader sees the shape of architectural evidence directly.

Axis A.2. The catalog turns architecture evidence into named coordinates: it lets a reviewer measure where a law is preserved, where an obstruction appears, how a repair moves the signature, and which observations are ready to support a stronger theorem package.
