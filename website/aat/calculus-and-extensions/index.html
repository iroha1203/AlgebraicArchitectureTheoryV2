Can architectural change be treated as a calculus of operations?

Architecture operation

AAT treats architectural change as algebraic action. An operation has a source, a target, a precondition, a transition, an invariant family, and a map that carries witnesses across the move. This turns change language into objects that can be composed, compared, audited, and repaired.

Definition 9.1. An architecture operation is an evidence-indexed partial morphism between selected architecture states. It becomes a theorem package when its preconditions, witness mapping, invariant family, conclusion, and recorded non-conclusions have been supplied.

Proposition 9.2. The operation schema preserves its generated operation kind and exposes a one-way witness bridge from post-operation witnesses back to pre-operation witnesses.

Proof idea. The schema stores the generated obligation and witness mapping as fields. The accessor theorems recover those fields and the existential witness bridge as reusable proof data.

Operation laws

The calculus names familiar architecture moves: compose, replace, refine, abstract, split, merge, isolate, protect, migrate, reverse, contract, repair, and synthesize. The name gives the operation a stable algebraic handle; the law package supplies the assumptions, conclusion, and proof data.

Law packages state exactly which observation, interface, coverage, exactness, finite universe, obstruction universe, or certificate makes a law readable as proved. The same surface phrase can then become graph law, runtime law, migration law, repair law, or synthesis law according to the evidence it carries.

Proposition 9.3. An architecture calculus law yields its recorded conclusion from its recorded assumptions, while unconditional algebraic laws require their own theorem packages.

Invariant correspondence

The operation calculus is paired with invariant families. For a selected family of invariants S, Ops(S) is the operation family that preserves them. For a selected family of operations T, Inv(T) is the invariant family preserved by them.

This correspondence gives design principles an algebraic form: they can be compared as operation/invariant packages. It is a weak Galois-style reading: stronger invariant families select a smaller operation space, while richer operation families reveal the invariants they preserve.

Feature extension

Feature extension is one of AAT's original moves: software evolution read as a local operation. A system grows by embedding an existing core into a larger architecture, adding a feature view, and carrying evidence for the interface route, runtime behavior, semantic preservation, and coverage. This gives SFT one of the AAT-side algebraic handles it later uses to compute with software evolution.

Definition 10.1. A feature extension is a local operation from a selected core into a selected extended architecture. Static split evidence, declared interface factorization, lifting evidence, runtime preservation, semantic preservation, and coverage travel as focused packages around the base extension.

This layered presentation lets a page, proof, or report name the layer that carries the relevant witness. Static hidden dependencies, semantic filling obstructions, and runtime axes each carry their own evidence.

Extension evidence

Extension proofs accumulate evidence. A base extension becomes informative through evidence for static split, feature view, lifting, ArchitectureFlatWithin, runtime, and semantic readings.

  • Static split StaticSplitFeatureExtension records core edge preservation, declared interface factorization, static edge discipline, and embedding policy preservation.
  • Feature view FeatureViewSectionPackage and FeatureViewSound state that the selected feature observation is read correctly through the extension.
  • Lifting SplitExtensionLiftingData and SplitExtensionLiftingPreservationPackage make selected feature steps lift through declared interfaces while preserving selected core and feature invariants.
  • ArchitectureFlatWithin SplitFeatureExtensionWithin and architectureFlatWithin_of_splitFeatureExtensionWithin produce ArchitectureFlatWithin through coverage, static side conditions, runtime evidence, and semantic evidence.

Proposition 10.2. A split feature extension yields ArchitectureFlatWithin through the selected universe and the static, runtime, semantic, coverage, and non-conclusion evidence carried by the package.

Proof idea. The theorem reads the supplied split package as coverage-aware flatness evidence. Static, runtime, and semantic axes assemble ArchitectureFlatWithin and prepare any stronger global flatness reading.

Extension obstruction

The structural extension formula classifies selected obstruction witnesses. The classification is explanatory: it records where a witness can be read, and it allows the same payload to be embedded into a multi-label layer.

Proposition 10.3. Under the selected extension-obstruction universe, a selected single-label witness is covered by at least one of the seven structural obstruction predicates.

Non-conclusion 10.4. The extension formula does not prove that classes are mutually exclusive, that every possible real-world obstruction has been extracted, that every extension is safe, or that unmeasured runtime and semantic axes are zero.

Example and counterexample

A feature extension adds a feature component to a selected core and chooses how the feature reaches the core. The structured case routes the interaction through a declared port; the obstruction case introduces a direct dependency from the feature component to an internal implementation component. Repair restores the declared interface route.

This example shows the calculus-level shape: selected static split evidence, a selected hidden-dependency witness, and a repaired static split package.

Chapter summary

The calculus treats architecture change as a selected operation. Each move records its source, target, support, preconditions, transition, invariant family, witness map, conclusion, and evidence record.

Feature extension is the chapter's software-evolution move. AAT can describe structured extensions, classify selected obstruction witnesses, and read repairs as operations that restore the intended interface route under explicit assumptions.

Terms

operation package
The source, target, support, precondition, transition, invariant family, witness map, conclusion, and non-conclusion record for an architecture move.
law package
A theorem wrapper that carries an operation kind, a law kind, explicit assumptions, a conclusion, and reusable proof data.
feature extension
A first-class addition schema with core and feature embeddings plus a selected feature view, together with static split, lifting, runtime, and semantic evidence packages.
extension obstruction
A selected witness classified as inherited core, feature-local, interaction, lifting failure, filling failure, complexity transfer, or residual coverage gap.
