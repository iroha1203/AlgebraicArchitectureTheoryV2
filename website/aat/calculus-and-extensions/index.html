Calculus and Extensions

AAT treats architecture moves as partial operations with explicit proof boundaries. Feature extension is the running example: a core may be embedded into a larger architecture, but preservation, lifting, filling, coverage, and obstruction classification each require separate evidence.

Reading model

AAT does not say

"This refactoring is an architecture operation, therefore every invariant is preserved." The family name is not a theorem, and neither is the existence of an extension.

AAT says

Under this operation boundary, selected observation model, witness universe, coverage or exactness assumption, and non-conclusion clause, a bounded conclusion can be read as theorem-grade.

operation boundary
The source, target, support, precondition, transition, invariant family, witness map, conclusion, and non-conclusion record for an architecture move.
law package
A bounded theorem wrapper that carries an operation kind, a law kind, explicit assumptions, and a conclusion. The law tag alone is only metadata.
feature extension
A first-class addition schema with core and feature embeddings plus a selected feature view. Static split, lifting, runtime, and semantic evidence are separate.
extension obstruction
A selected witness classified as inherited core, feature-local, interaction, lifting failure, filling failure, complexity transfer, or residual coverage gap.

Architecture operation

Plain reading. An operation is a proof interface for an architecture move. It says what changed, when the move is allowed, what witness mapping is available, and what claims remain outside the theorem.

Definition 9.1. An architecture operation is a boundary-indexed partial morphism between selected architecture states. It becomes a theorem package only after its preconditions, witness mapping, invariant family, bounded conclusion, and non-conclusions have been supplied.

Lean status defined only Anchored by ArchitectureOperation, OperationProofObligation, ArchitectureOperationKind, and OperationRoleSchema.

Proposition 9.2. The operation schema preserves its generated operation kind and exposes a one-way witness bridge from post-operation witnesses back to pre-operation witnesses.

Lean status proved anchors See ArchitectureOperation.generatedObligation_kind, ArchitectureOperation.witnessMapping_sound, and ArchitectureOperation.exists_sourceWitness_of_targetWitness.

Proof idea. The schema stores the generated obligation and witness mapping as fields. The accessor theorems recover those fields and the existential witness bridge without adding any new preservation claim.

Operation laws

The calculus names familiar architecture moves: compose, replace, refine, abstract, split, merge, isolate, protect, migrate, reverse, contract, repair, and synthesize. A name makes the operation auditable; it does not discharge the law.

Bounded law packages state exactly which observation, interface, coverage, exactness, finite universe, obstruction universe, or certificate makes a law readable as proved. The same surface phrase can therefore have different proof boundaries in graph, runtime, migration, repair, and synthesis settings.

Proposition 9.3. A bounded architecture calculus law yields its recorded conclusion from its recorded assumptions, but it does not yield an unconditional algebraic law for every operation.

Lean status proved anchors See ArchitectureCalculusLaw.conclusion_of_assumptions and the concrete law entrypoints in OperationLaws.lean.

Invariant correspondence

The operation calculus is paired with invariant families. For a selected family of invariants S, Ops(S) is the operation family that preserves them. For a selected family of operations T, Inv(T) is the invariant family preserved by them.

This correspondence is useful because it makes design principles compare as operation/invariant packages instead of slogans. It is intentionally weak: it does not classify all operations, prove a lattice isomorphism, or promote local contracts into global decomposability.

Lean status proved anchors See operationInvariant_galois, Ops_antitone, Inv_antitone, and the DesignPattern closure-law accessors.

Feature extension

Plain reading. A base feature extension says that an existing core and a new feature are embedded into a larger architecture with a selected feature view. It does not automatically say the extension is static-split, liftable, runtime-flat, semantic-flat, or complete.

Definition 10.1. A feature extension is a local operation from a selected core into a selected extended architecture. Static split evidence, declared interface factorization, lifting evidence, runtime preservation, semantic preservation, and coverage are separate packages that may or may not accompany the base extension.

Lean status defined only / proved accessors Anchored by FeatureExtension, StaticSplitFeatureExtension, FeatureViewSectionPackage, SplitExtensionLiftingData, and SplitFeatureExtensionWithin.

The base extension and the evidence packages are intentionally separated so a page, proof, or report can say exactly which layer failed. A static hidden dependency is not the same object as a semantic filling failure, and an unmeasured runtime axis is not measured zero.

Extension evidence

Extension proofs accumulate evidence. The useful question is not whether an extension exists, but which selected evidence is available and what conclusion it can discharge.

  • Static split StaticSplitFeatureExtension records core edge preservation, declared interface factorization, absence of forbidden static edges, and embedding policy preservation.
  • Feature view FeatureViewSectionPackage and FeatureViewSound state that the selected feature observation is read correctly through the extension.
  • Lifting SplitExtensionLiftingData and SplitExtensionLiftingPreservationPackage make selected feature steps lift through declared interfaces while preserving selected core and feature invariants.
  • Bounded flatness SplitFeatureExtensionWithin and architectureFlatWithin_of_splitFeatureExtensionWithin produce bounded ArchitectureFlatWithin only when coverage, static side conditions, runtime evidence, and semantic evidence are supplied.

Proposition 10.2. A split feature extension yields bounded flatness only inside the selected universe and only with the static, runtime, semantic, coverage, and non-conclusion evidence carried by the package.

Lean status proved anchors See architectureFlatWithin_of_splitFeatureExtensionWithin, LawfulExtensionPreservesFlatness, and ThreeLayerFlatnessPositiveExample.architectureFlatWithin.

Proof idea. The theorem reads the supplied split package as coverage-aware flatness evidence. Static, runtime, and semantic axes are separate inputs, so the conclusion remains bounded to the selected universe and does not become global ArchitectureFlat.

Extension obstruction

The structural extension formula classifies selected obstruction witnesses. The classification is explanatory: it records where a witness can be read, and it allows the same payload to be embedded into a multi-label layer. It is not a disjoint sum theorem.

Proposition 10.3. Under the selected extension-obstruction universe, a selected single-label witness is covered by at least one of the seven structural obstruction predicates.

Lean status proved anchors See ArchitectureExtensionFormula_structural, ExtensionObstructionWitness.toMultiLabel_label_iff, and the filling / lifting / complexity / residual-coverage bridge theorems.

Non-conclusion 10.4. The extension formula does not prove that classes are mutually exclusive, that every possible real-world obstruction has been extracted, that every extension is safe, or that unmeasured runtime and semantic axes are zero.

Example and counterexample

The coupon feature extension is the canonical small example. A good extension routes the feature interaction through a declared payment port. A bad extension introduces a direct dependency from the feature service to the payment adapter's internal cache. Repair restores the declared interface route.

Positive example

Declared interface route carries selected static split evidence

proved anchors
Lean anchors
CouponStaticDependencyExample.good_selectedStaticSplitFeatureExtension, extensionOf_featureViewSound, good_featureObservationCoverage
Claim boundary
Selected static split and selected feature observation coverage for the coupon example.
Non-conclusion
No runtime flatness, semantic flatness, or extractor completeness theorem follows from this static example alone.
Use
Shows what evidence a feature extension needs before it can be read as a bounded theorem package.
Counterexample

Hidden dependency refutes the selected static split predicate

proved anchors
Lean anchors
CouponStaticDependencyExample.hiddenDependencyWitnessExists, bad_not_selectedStaticSplitFeatureExtension
Claim boundary
The witness is a selected static hidden dependency inside the coupon extension family.
Non-conclusion
The witness does not classify every possible runtime or semantic failure and does not imply solver or extractor completeness.
Repair reading
repaired_selectedStaticSplitFeatureExtension shows the selected static split predicate can be recovered for the repaired route.

The example is deliberately narrow. It demonstrates selected static split evidence, a selected hidden-dependency witness, and a repaired static split package; it does not promote the coupon model into a universal extension theorem.
