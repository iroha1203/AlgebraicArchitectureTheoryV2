Reader model

AAT does not say

"The architecture was repaired." A repair suggestion, a successful patch, a solver timeout, or a smaller metric is not automatically a theorem about all axes of a real repository.

AAT says

Within this selected universe, this repair step or path package discharges this recorded conclusion, while these axes, witnesses, and completeness claims remain outside the theorem.

selected obstruction universe
The witness family and measure that a repair theorem is allowed to decrease or clear.
repair step
A source-to-target operation tagged by a selected rule; its effect is theorem-bearing only through an admissible package.
finite repair package
A bounded plan, target clearing claim, coverage assumptions, and non-conclusions for selected witnesses.
no-solution certificate
A checked certificate whose validity implies no architecture satisfies the selected constraint system.
complexity transfer
The case where a selected obstruction decreases while residual risk or obstruction moves to another measured axis.
signature trajectory
The finite sequence of selected signature observations visited by an architecture evolution path.
diagram filler
Evidence that two operation paths are homotopic under selected generators and observation-preservation assumptions.
non-fillability witness
A selected witness value whose soundness refutes the existence of a filler for the selected diagram.

Repair step

Plain reading. A repair theorem is local to the witness universe it names. The same operation can be a valid selected repair and still leave runtime, semantic, policy, or coverage questions open.

Definition 11.1. A repair step is a source-to-target architecture operation read against a selected obstruction universe. The base step only records the operation skeleton; selected-measure decrease is a separate theorem boundary supplied by an admissible repair rule.

Lean status defined only / proved accessors Anchored by SelectedObstructionUniverse, RepairStep, RepairStepDecreases, and AdmissibleRepairRule.

Proposition 11.2. If a selected witness is in the selected universe, a rule is admissible for that witness, and the recorded step applies that rule, then the selected obstruction measure decreases from the source to the target.

Lean status proved anchors See repairStepDecreases_of_admissible, extensionObstructionMeasure_decreases_of_admissible, and repairStepDecreasesLaw_conclusion.

Proof idea. The admissible rule carries exactly the decrease theorem needed by the selected step. The conclusion is a measure comparison in the selected universe, so no unselected axis is folded into the theorem.

Finite repair and synthesis

Bounded repair packages lift a single step into a finite plan. The plan exposes a finite step bound and target clearing of the selected obstruction universe. It still depends on coverage and exactness assumptions, and it records what the plan does not conclude.

Synthesis has a different obligation. A produced candidate needs satisfaction evidence for the selected constraints; a no-solution claim needs a valid certificate, not just a failed search.

Proposition 11.3. A finite repair package exposes selected-obstruction clearing, a synthesis soundness package exposes candidate satisfaction, and a valid no-solution certificate exposes absence of a selected satisfying architecture. None of these packages proves solver completeness outside its recorded universe.

Lean status proved anchors See FiniteRepairPackage.selectedObstructionsCleared, SynthesisSoundnessPackage.candidate_satisfies, NoSolutionCertificate.sound_of_valid, synthesisCandidateSoundnessLaw_conclusion, and noSolutionCertificateSoundnessLaw_conclusion.

Proof idea. Each package stores its own soundness field and boundary assumptions. The accessor theorem returns that stored conclusion; it does not manufacture completeness for an unrecorded search space or an unsupported constraint language.

Repair and synthesis theorem boundaries
Package Lean anchors Theorem-grade reading Non-conclusion
Bounded finite repair BoundedRepairPlan, FiniteRepairPackage Selected obstruction universe is cleared at the target under explicit plan and coverage assumptions. No all-obstruction removal, repair optimality, or global flatness preservation.
Candidate synthesis SynthesisConstraintSystem, SynthesisSoundnessPackage.candidate_satisfies The produced candidate satisfies every selected required constraint. No completeness of generated candidates and no empirical success claim.
No-solution certificate ValidNoSolutionCertificate, NoSolutionCertificate.sound_of_valid A valid certificate implies no selected architecture satisfies the selected constraint system. Solver failure, timeout, unsupported syntax, and incomplete search are not certificates.

Complexity transfer

A repair can reduce a selected static witness while making a different measured axis worse. AAT treats this as a first-class boundary because "removed here" and "improved everywhere" are different claims.

Counterexample 11.4. There is a bounded repair skeleton in which the selected static obstruction measure decreases, while a runtime-axis witness appears after repair. Therefore selected repair decrease does not imply all-axis nonincrease.

Lean status proved anchors See RepairTransferCounterexample.selectedRepairStep_decreases, runtimeAxisMeasure_increases, selectedRepairStep_not_all_axes_nonincreasing, and staticSplitRepair_transfers_runtime.

Proof idea. The example has two states. The selected static universe reads the target as zero, while the runtime diagnostic universe reads the same target as carrying a measured witness. The repair theorem remains true because it was scoped to the selected static universe.

Non-conclusion 11.5. A repair report, repair suggestion, or adoption record does not certify repair correctness, empirical cost improvement, or absence of side effects unless a separate package states those premises and conclusions.

Signature trajectory

Dynamics reads a repair or migration path as a sequence of selected observations. Endpoint equality, endpoint delta zero, or a safer final state can miss excursions and transfer events that occur along the path.

Paths and homotopy

ArchitecturePath is a finite operation sequence indexed by its endpoints. It can be appended, measured by step length, and checked for stepwise invariant preservation. PathHomotopy is generated by selected path rewrite contracts and context closure.

Definition 12.1. An architecture path is an endpoint-indexed finite sequence of primitive architecture steps. A path homotopy is the generated relation produced by selected independent-square, same-contract, repair-fill, and context-closure generators.

Lean status defined only / proved accessors Anchored by ArchitecturePath, ArchitecturePath.append, ArchitecturePath.length_append, and ArchitecturePath.PathHomotopy.

Proposition 12.2. If every primitive step of a finite path preserves a selected invariant, then the path preserves that invariant. If the selected homotopy generators preserve a selected observation, then homotopic paths have the same selected observation.

Lean status proved anchors See ArchitecturePath.pathPreservesInvariant, PathHomotopy.observation_eq_append, PathHomotopy.observation_eq, and architectureHomotopyInvariance.

Proof idea. Path preservation proceeds by induction over the finite path. Observation preservation for homotopy is also generator-relative: every generator must carry the observation-preservation premise supplied to the theorem.

Diagram filling

A semantic diagram compares two endpoint-matched operation paths. A filler says the paths are homotopic under the selected finite path calculus. A non-fillability witness says that, for the selected observation and witness value, no such filler can exist.

Proposition 12.3. Under selected observation-preservation assumptions for the path generators, a diagram filler implies equality of selected observations. Conversely, a selected observation difference refutes the filler and can be packaged as a sound non-fillability witness.

Lean status proved anchors See diagramFiller_observation_eq, observationDifference_refutesDiagramFiller, observationDifference_nonFillabilityWitnessFor, and obstructionAsNonFillability_sound.

Proof idea. The filler is a path homotopy. The observation theorem for path homotopy turns it into selected observation equality. A supplied inequality contradicts that equality and becomes a refutation package for the selected witness value.

Non-conclusion 12.4. Diagram non-fillability is not a theorem that every split failure is semantic, that every semantic obstruction has been extracted, or that the observation model is complete.

Coupon path example

The coupon and discount example is the small concrete model for path order. Coupon-first and discount-first paths reach the same endpoint shape, but their selected rounding traces differ. That difference refutes the selected diagram filler and produces a non-fillability witness.

Example 12.4. In the selected coupon / discount diagram, the coupon-then-discount path has one rounding trace, the discount-then-coupon path has another, and the rounding-order witness refutes fillability for the selected diagram.

Lean status proved anchors See CouponDiscountExample.couponDiscountDiagram, couponThenDiscount_roundingTrace, discountThenCoupon_roundingTrace, roundingOrder_refutes_selectedDiagramFiller, and roundingOrder_nonFillabilityWitnessFor.

The example should be read as a bounded semantic obstruction, not as an empirical frequency claim about coupon systems. It also does not prove runtime flatness or repository extraction completeness for any real codebase.
