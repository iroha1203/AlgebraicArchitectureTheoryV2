Foundations

Boundary note

AAT does not claim complete extraction from real codebases. ComponentUniverse is proof-carrying measurement scope, not a promise that all components and relations have been observed.

First model

AAT does not say

"This whole repository is clean." AAT does not turn a bounded scan, diagram, review note, or theorem package into a repository-wide quality certificate.

AAT says

Inside this selected universe, under this observation model, and with these coverage and exactness assumptions, this selected obstruction family is zero, nonzero, or unmeasured.

  • universe The selected components, relations, diagrams, or measurements that a claim is allowed to quantify over.
  • coverage The premise that the selected evidence contains the components or relations needed by the claim.
  • exactness The premise that a measured value can be read as the intended in-scope relation, witness, or axis.
  • observation The static, runtime, semantic, review, or operational model through which evidence is being read.
  • evidence boundary The line between measured evidence and facts that remain outside the theorem package or report.
  • measured zero The selected evidence was measured and no in-scope violating relation or witness was found.
  • unmeasured outside The relation, effect, component, or observation was not selected, so absence is not asserted.
  • non-conclusion A statement that does not follow, even when the bounded claim has been proved or measured.

Central decomposition

AAT formalizes software architecture as a local algebra of objects, operations, invariant families, obstruction witnesses, signatures, and theorem boundaries. The review question is not "is this design good?" but "which invariant is being required, under which observation boundary, and which witness or theorem package supports the claim?"

This is a decomposition of the theory surface, not a theorem whose proof is carried by the current Lean package. The theorem-grade claims in this chapter begin with the numbered definitions and propositions below.

Object definition

The foundational record is deliberately small. AAT does not begin with every fact about a repository; it begins with the selected carrier, the policies used to read that carrier, and the observation context that states what evidence is in scope.

This schema is the first theorem-boundary guard. A claim about the object is never stronger than the carrier, policy, and observation context that presented it.

Definition 1.1. A bounded ArchitectureObject is the mathematical object selected by a carrier, a policy, and an observation context. It is cut from code, specifications, review evidence, or operational observations, but it is not the unbounded real codebase itself.

Lean status defined only Tracked through ArchitectureCore presentation declarations; no standalone complete-codebase object is claimed.

Proposition 1.2. Object claims are relativized claims. A sentence about an architecture object is licensed only inside the carrier, policy, and observation context that present it.

Lean status theorem candidate Website reading of the canonical boundary; Lean anchors remain the selected-presentation definitions below.

Proof idea. Unfold the object schema: ArchitectureObject is presented as ArchitectureCarrier plus ArchitecturePolicy plus ArchitectureObservationContext. Later theorems consume such a selected presentation as input. Their conclusions are therefore indexed by the selected carrier, policy, and observation context; changing any one of these changes the proposition being proved.

Core proposition

ArchitectureCore is the proof-carrying presentation of the bounded object. It bundles static and runtime evidence, boundary and abstraction policy, selected semantic diagrams, decidability assumptions, and a finite ComponentUniverse.

Proposition 2.1. Core presentations do not imply complete extraction. A core can expose the evidence needed by a theorem package while still omitting relations outside its declared coverage boundary. Presentation equivalence is therefore relative to the selected observation model.

Lean status defined only ArchitectureCore is defined-only; selected accessors are proved, but extraction completeness is not promoted.

Proof idea. A core presentation contains the evidence selected for the chapter: a carrier, policies, observation data, and a bounded measurement universe. Any claim derived from that presentation can use only those selected fields. Complete extraction of an ambient repository would require a separate premise saying that every relevant component, dependency, runtime interaction, and semantic observation factors through the core. That premise is not present in ArchitectureCore.

Counterexample 2.2. Let two ambient repositories have the same selected static evidence, the same selected finite component universe, and the same selected observation context. Now add to the second repository a runtime edge whose source or target lies outside that selected universe. The fields of the selected ArchitectureCore are unchanged, because the added edge is out of scope for the core. A complete extraction of the ambient repository would distinguish the two repositories, but the selected core presentation does not. Therefore the core presentation does not imply complete extraction.

Lean status theorem candidate Counterexample text preserves the non-implication boundary; no Lean theorem is added in this chapter.

Non-conclusion 2.3. This distinction is essential. Reading ArchitectureCore as complete extraction would promote every omitted runtime edge into evidence of absence. The record is intentionally weaker: it proves facts about the supplied universe, not about every edge a production system could emit.

Universe proposition

ComponentUniverse is the measurement scope used by finite bridge theorems. It records selected components, selected relations, coverage boundary, and exactness boundary. Outside that universe, AAT does not infer zero, absence, flatness, split, or lawfulness.

Definition 3.1. A finite measurement universe supplies the bounded component list and the proof obligations needed to read graph-level evidence as in-scope evidence.

Lean status defined only Anchored by ComponentUniverse, the finite measurement universe with proof-carrying fields.

Proposition 3.2. Finite bridges are coverage-relative. Executable or graph-level facts become proposition-level facts only under their supplied universe and closure assumptions.

Lean status proved anchors See ComponentUniverse.edgeClosed_of_covers, FiniteArchGraph.edgeClosed_components, and bounded reachability facts.

Proof idea. A finite universe supplies a measured list of components together with the assertion that the relevant graph evidence is covered by that list. Closure says that an in-scope edge does not leave the measured list. Under these two assumptions, a finite computation over the measured list can be read as a statement about the selected graph, because every component and every selected edge needed by the claim has already been accounted for. Reachability is likewise bounded: once the universe is finite, a reachability assertion can be witnessed by a finite path inside the supplied universe, rather than by an unbounded search over an unspecified system.

Counterexample 3.3. Without coverage, a bounded search can miss a component that carries the only violating edge. Without edge closure, an in-scope source can point to an out-of-scope target, so a zero result on the list is not a graph-level zero claim.

Lean status theorem candidate Boundary example for missing coverage or closure; the proved bridge theorems require those premises explicitly.

Remark 3.4. The same boundary explains the role of ComponentCategory: it remembers whether reachability exists and intentionally forgets path count and walk length. Path counts, adjacency powers, finite metrics, and future free-category constructions live outside that thin-category claim. Likewise, current Decomposable means StrictLayered; acyclicity, finite propagation, nilpotence, and spectral conditions remain separate theorems or representations.

Example and counterexample

Example 4.1. A repository slice containing a coupon service, a payment interface, a payment adapter, and selected adapter internals can be a valid component universe for a static split theorem. That same slice is not, by itself, evidence about runtime retries, cache invalidation, rounding order, or operational incidents. Those require their own observation contexts.

  • Measured zero No selected violating relation was found inside the chosen universe and exactness boundary.
  • Unmeasured outside No claim is made about components, relations, effects, or observations outside that boundary.
  • Presentation equivalence Different carriers can present the same object only relative to the selected observation model.

Bounded reading 4.2. In the good static presentation, the coupon service reaches the payment implementation only through the declared payment port. If the theorem package supplies the required coverage and closure assumptions, this supports the bounded statement that the selected evidence contains no coupon-to-internal-cache edge.

Non-conclusion 4.3. The conclusion is deliberately narrow. A runtime trace could still show CouponService calling PaymentAdapter.publicApi through a retry helper, or a later static scan could reveal the bad edge CouponService to PaymentAdapter.internalCache. Neither case is erased by the good-universe reading; each requires its own observation context and theorem boundary. The fuller worked version appears in the coupon case study.
