Research question

Can architecture be decomposed into primitive facts that cannot be split further inside the theory?

AAT answers this with Atom. Components, relations, capabilities, state, effects, authority, contracts, semantic readings, and runtime interactions can all be read as typed primitive facts when they are used as generators for later architectural structure.

Definition

An Atom is the minimal unit from which AAT builds an architecture. It is a primitive typed architectural fact: once selected as an atom, it is not further decomposed inside the theory before molecules, laws, obstructions, signatures, and operations begin to read it.

Atom existence does not come from an observation boundary, a design law, a tool report, or an SFT event. Those later layers may observe, evaluate, map, or summarize atoms, but they do not generate the atoms that pure AAT starts from. 

a = (kind, axis, subject, predicate, payload)

The tuple records how the primitive fact is read. The kind says which atom family it belongs to, the axis says which architectural coordinate it touches, the subject says what it is about, the predicate gives one atomic statement, and the payload carries the value, name, type, or evidence handle needed to read that statement.

Definition 1.1. An Atom is an indivisible generator for AAT's architecture algebra: a primitive typed fact whose existence is supplied by the atom axiom system and whose reading is organized by AtomKind, Axis, and AtomPredicate.

Remark 1.2. Primitive does not mean operationally simple. A payment capture, permission rule, event meaning, or runtime retry may be complex. It is primitive here when AAT treats it as a generator rather than as a theorem derived from a later law, score, or report.

Atom axiom system

AAT begins from an AtomAxiomSystem. This object supplies the atom type, the predicate type, and the maps that read each atom's kind, axis, and predicate. It also records the discipline that makes atoms a root: an atom is a single fact, its predicate reading is stable, and its existence is not created by observation, laws, tools, or SFT events.

Definition 2.1. AATFromAtomAxioms is the pure AAT root. Later AAT structures are parameterized by an atom axiom system, not by an observation record, a finite presentation, an ArchSig packet, or an SFT transition.

Proposition 2.2. Every atom supplied by an AtomAxiomSystem is a primitive typed fact for that system.

Proof. Lean defines AtomAxiomSystem.Primitive system atom as the conjunction of primitive kind, singleFact, predicatePreserving, boundaryIndependent, and lawIndependent. The theorem AtomAxiomSystem.primitive constructs this conjunction from the evidence fields stored in the axiom system.

Proposition 2.3. A law can evaluate atoms and configurations of atoms, but the selected law does not create atom existence.

Proof. The axiom system stores noLawCreatesAtoms, and the theorem AtomAxiomSystem.law_does_not_create_atoms exposes it. This is why a failed law produces a law-relative reading later, not a retroactive change to the atom surface.

Axiom reading

A0 Primitive existence
At is the type of primitive architectural facts. AAT objects are generated from it.
A1 Typing
Every atom has a kind and an axis, so the theory can keep family and reading coordinate separate.
A2 Single fact
One atom records one primitive fact. Composite claims are represented by finite atom families or configurations.
A3 Predicate stability
An atom's identity is read through its type, subject, predicate, and payload, not through a later report label.
A4 Composition
Finite atom families generate configurations, which later become molecules and architecture objects.
A5 Law non-generation
Laws constrain, preserve, test, or reject atom configurations; they do not create atom truth.
A6 Observation non-generation
Observation, measurement, inspection, and reporting read atoms; they do not generate the root facts.
A7 Operation preservation of atom-origin
Architecture operations map atom-generated configurations to atom-generated configurations.
A8 Essential uniqueness
For a fixed source and atom vocabulary, a canonical atom family is read before coarse projections are applied.

Atom families

The current Lean-facing core uses nine atom families. This is the current core grammar, not a claim that the taxonomy is globally complete.

  • Component A unit exists: module, service, class, package, table, queue, adapter, port, or process.
  • Relation A typed connection holds: calls, reads, writes, publishes, subscribes, owns, implements, or projects.
  • Capability A subject provides an ability: query, command, handler, port, storage access, serialization, or calculation.
  • State A component or object holds state: aggregate field, cache entry, event-log projection, configuration value, or session value.
  • Effect An internal or external effect occurs: message send, payment capture, database write, cache invalidation, event publish, or retry.
  • Boundary authority A permission, trust relation, owner, visibility rule, policy scope, access path, or encapsulation fact is present.
  • Contract specification An interface promise, precondition, postcondition, event shape, idempotency promise, or authorization rule is present.
  • Semantic interpretation A domain meaning, identity, ownership meaning, money meaning, event meaning, or selected interpretation is present.
  • Runtime interaction A runtime call, message, event handling path, retry, timeout, lock, transaction, or schedule constraint is present.

Non-conclusion 3.1. The current atom grammar is not a global completeness theorem. It is the current Lean-facing grammar. If a stable primitive architectural fact cannot be represented as a molecule of the existing families, the grammar may be extended under an explicit extension policy.

Component atoms

A component atom says that an architectural unit is present. The unit may be a module, service, class, package, process, database table, queue, adapter, port, or other selected architecture unit. It can later become a vertex in a graph representation, but the graph vertex is not the whole component reading. 

component(CheckoutService)

A component atom supplies a carrier for later structure: it can be the source or target of relation atoms, the owner of state atoms, the provider of capability atoms, or the subject of contract, effect, authority, semantic, and runtime atoms. Component existence by itself does not imply lawfulness.

Relation atoms

A relation atom says that a typed connection holds between subjects. Typical relations include imports, calls, reads, writes, publishes, subscribes, owns, implements, substitutes, projects, emits, and handles. 

relation_calls(CheckoutService, PaymentGateway)
relation_writes(capture, orderStatus)
relation_emits(capture, OrderPaid)

Relation atoms are the material from which layering, reachability, cycle, projection, substitution, and effect-order laws can later be evaluated. They are not automatically graph edges with one fixed meaning; the predicate determines which relation is being asserted.

Capability atoms

A capability atom says what a component or object can provide. It sits between mere connection and full contract: a relation says where something connects, a capability says what is available, and a contract says what is promised. 

capability(CheckoutService, create_order)
capability(PaymentGateway, capture_payment)

Capability atoms support questions such as whether a required capability is provided, whether use factors through a declared interface, and whether a provided capability is covered by a selected contract. A missing capability can later become an obstruction, but the capability atom itself is just the fact that an ability is present.

State atoms

A state atom says that an architecture object holds a state quantity. State may appear as a database column, aggregate field, cache entry, session value, event-log projection, configuration value, or other stateful coordinate. 

state(OrderStore, order_status)
state(CheckoutService, retry_count)

State atoms generate state spaces and transition questions: which operations read state, which operations update it, which invariants are preserved by updates, and which effects require a state update before or after they occur.

Effect atoms

An effect atom says that an architecture object can act on internal or external state. Message sends, email, payment capture, database writes, cache invalidation, event publication, file writes, remote calls, retries, and compensations can all be effect facts. 

effect(capture_payment)
effect(publish_order_created)
effect(invalidate_cart_cache)

Effect atoms keep architecture from collapsing into a static dependency graph. A system can be acyclic and still fail because an effect is not idempotent, lacks compensation, happens in the wrong order, or requires authority that has not been supplied.

Boundary authority atoms

A boundary authority atom records who may perform which action on which resource, or which trust, ownership, visibility, policy scope, access path, or encapsulation fact is present. Authority is not the same as reachability: a component may be able to call a resource without being authorized to perform the selected action. 

authority(CheckoutService, capture, PaymentGateway)
trust_relation(CheckoutService, PaymentGateway)
visibility(PaymentGatewaySecret, internal)

Authority atoms feed access, permission, encapsulation, and policy laws. When those laws fail, the later obstruction is law-relative; the authority atom itself only records the primitive boundary fact.

Contract specification atoms

A contract specification atom records a promise attached to an operation, capability, component, or effect. Contracts may describe input and output shape, preconditions, postconditions, return type, error behavior, event shape, idempotency, authorization, latency, or substitution requirements. 

contract(capture, PaymentRequest -> PaymentResult)
contract(PaymentGateway, idempotent_capture)
contract(OrderPaid, event_shape_v1)

Contract atoms are central to substitution laws. An implementation can match a type signature while failing a contract atom. In that case lawfulness does not follow from the local type shape alone.

Semantic interpretation atoms

A semantic interpretation atom records meaning: domain invariant, identity, ownership meaning, money amount, event meaning, authorization meaning, or another selected interpretation that must be preserved. 

semantic(capture_payment, charges_customer_once)
semantic(orderId, identifies_order)
semantic(OrderPaid, denotes_successful_capture)

Semantic atoms are not forced to be unique per subject. The same subject can carry several semantic facts. A coarse observation may merge those facts, but that is a projection of the semantic surface, not proof that the surface is ambiguous.

Runtime interaction atoms

A runtime interaction atom records a fact that matters through execution behavior: runtime call, RPC, message, event emission, event handling, retry, timeout, circuit breaker, bulkhead, lock, transaction, or schedule constraint. 

runtime_interaction(CheckoutService, PaymentGateway, retry)
runtime_interaction(capture, PaymentGateway.capture, timeout)
runtime_interaction(OrderPaidHandler, OrderPaid, event_handler)

Runtime interaction atoms are not failures by themselves. A timeout or circuit breaker is a primitive runtime fact that later runtime laws can evaluate. Static flatness can hold while a runtime interaction still creates a separate obstruction.

Composition

Individual atoms are meaningful, but AAT becomes architectural when several atom families are read together. A concern is rarely just a component, just a relation, or just a contract. It is usually a finite surface of component, relation, capability, state, effect, authority, contract, semantic, and runtime facts.

This finite surface is the material from which the next chapter builds molecules. A role such as adapter, repository, saga participant, policy boundary, or event handler is a reading of such a surface. It is not a primitive atom kind by itself.

Proposition 3.2. Atom composition is the first place where architecture appears as structure rather than as isolated facts. Different law families read different projections of the same finite atom surface.

Proof idea. A relation law can ignore a semantic atom, an effect law can ignore a static import edge, and an authority law can ignore a cache state. The surface remains the same, but the selected law chooses which atom families it evaluates. This is why AAT keeps atoms separated before lawfulness is tested.

Source reading

Atomizing a source is not the same as copying syntax into a graph. A source fragment may expose declarations, imports, calls, state updates, effects, permissions, type contracts, semantic annotations, and runtime protections. The atom vocabulary tells AAT which primitive fact family each of those source facts contributes to.

These atoms do different work. The import and call relations feed dependency and layering laws. The state and write atoms feed transition laws. The effect and emit atoms feed effect ordering and idempotence laws. The authority atom feeds access laws. The contract and semantic atoms feed substitution and meaning-preservation laws.

Proposition 4.1. Under a fixed source and fixed atom vocabulary, a canonical atom surface is read before any lawfulness judgement. Coarser reports are projections of that surface, not competing atom truths.

Proof idea. A coarse projection may forget coordinates: for example it may retain component(Checkout) and relation_imports(Checkout, PaymentGateway) while forgetting the timeout, authority, state update, and semantic atoms. The lost coordinates limit which laws can be evaluated. They do not make the original atom surface ambiguous.

Observation

Observation is how a tool, reader, or workflow encounters atom candidates. It is not the source of atom truth. A raw candidate may point at a predicate-shaped fact with evidence. An observed atom observes an already-existing atom. A missing observation records a gap in the selected observation layer, not a theorem that the atom is absent.

This is why AAT separates Atom from RawAtomCandidate, ObservedAtom, AtomObservationGap, and AtomPresentation. The separation lets tools report evidence without turning every report into a mathematical root fact.

Proposition 5.1. Observation does not create atoms. A validated observation may justify a report reading, but pure AAT still reads the observed atom as already supplied by the atom axiom system.

Proof. ObservedAtom.observation_does_not_create_atoms returns system.noObservationBoundaryCreatesAtoms. ObservedAtom.atom_primitive returns system.Primitive observed.atom. Thus observation can carry evidence while the atom's primitive status remains a property of the root axiom system.

Non-conclusion 5.2. A missing observation is not a measured zero atom and not an absence theorem. It only says that the selected observation layer did not see or validate the atom.

Propositions

Proposition 6.1. Atom axes are coordinates for reading primitive facts. They are not scores, observation boundaries, theorem discharges, or extractor completeness claims.

Proof idea. In Lean, Axis is an inductive coordinate family with constructors such as static, semantic, specification, runtime, boundary, dataflow, governance, evolution, and policy. Nothing in the axis constructor proves that the axis has been measured or that a law has passed. Measurement and lawfulness appear later.

Proposition 6.2. Role and pattern names are not primitive atom kinds. They are molecule interpretations over finite atom configurations.

Proof idea. A repository, adapter, saga participant, or policy boundary is usually read from several atoms: components, capabilities, contracts, state, effects, relations, and authority facts. Treating the role name as the primitive fact would hide the atoms that laws need to evaluate.

Proposition 6.3. A tool output can be an evidence channel for atom observation, but it is not itself an atom generator inside pure AAT.

Proof. The theorem AtomAxiomSystem.tool_output_does_not_create_atoms exposes noToolOutputCreatesAtoms. Observation-side records can still carry evidence, validation status, and coverage information; those belong to the observation layer.

Worked example

A checkout feature can be read as a small atom surface. The important move is to keep dependency, state, effect, authority, contract, and semantic facts separate even when they come from the same code path. 

component(CheckoutService)
component(PaymentGateway)
relation_calls(CheckoutService, PaymentGateway)
capability(PaymentGateway, capture_payment)
state(OrderStore, order_status)
effect(capture_payment)
contract(PaymentGateway, idempotent_capture)
authority(CheckoutService, capture, PaymentGateway)
runtime_interaction(CheckoutService, PaymentGateway, retry)
semantic(capture_payment, charges_customer_once)

At this stage AAT has not yet judged the design lawful or unlawful. It has only named the primitive facts that later molecules, laws, obstruction circuits, and signatures can read.

  • Dependency law input relation_calls(CheckoutService, PaymentGateway) can be read by layering or reachability laws.
  • Effect law input effect(capture_payment) can be read by idempotence, compensation, or ordering laws.
  • Authority law input authority(CheckoutService, capture, PaymentGateway) can be read by access and policy laws.
  • Semantic law input semantic(capture_payment, charges_customer_once) can be read by meaning-preservation laws.

Counterexamples and non-conclusions

Counterexample 8.1. A static dependency graph can be acyclic while an effect atom still violates idempotence or compensation. Therefore a graph zero reading is not an all-axis atom reading.

Counterexample 8.2. A tool may observe relation_calls(A, B) and miss authority(A, capture, B). The missing authority observation is not proof that the authority atom is absent; it only marks the observation boundary.

Counterexample 8.3. Two semantic atoms can share the same subject. For example, semantic(sendEmail, twoFactorAuthentication) and semantic(sendEmail, newsletterDelivery) may both be present. A coarse observation such as notification has forgotten semantic coordinates; it has not shown that the canonical semantic surface is non-unique.

Non-conclusion 8.4. Atom status alone does not prove lawfulness, flatness, decomposability, runtime safety, semantic correctness, or SFT forecast correctness. Those readings require later structures: molecules, laws, finite universes, theorem packages, signatures, and operational evidence.

Formal reading

The Lean root is AtomAxiomSystem. It supplies Atom, AtomKind, Axis, and a predicate alignment between each atom and its typed predicate. Accessor theorems record that root atoms are single facts, preserve their predicate reading, and are independent of observation boundaries and selected design laws.

Observation-side structures live under Observation. Their proved accessors keep raw candidates, observed atoms, missing observations, and ArchMap layers from being confused with pure AAT atom generation.

Formal status 9.1. The root structures are defined in Lean. The accessor theorems for primitive atoms, predicate alignment, single fact, boundary independence, law independence, observation non-generation, law non-generation, tool-output non-generation, and SFT-event non-generation are proved.

Formal status 9.2. Observation structures are defined in Lean. The accessors that keep raw candidates from atom truth, observed atoms from atom creation, and missing observations from atom absence are proved.

Chapter summary

The Atom chapter fixes the root of AAT. Atom is the primitive typed fact from which molecule, object, law, circuit, signature, operation, and representation readings are built. It is not a tool warning, not a metric coordinate, not a role name, and not a theorem boundary.

The practical payoff is separation. A review can say which atoms are being read, which observation layer saw them, which atoms are missing from the observation, and which later law is allowed to evaluate them. That separation is what lets the next chapter build molecules and architecture objects without smuggling in lawfulness or extractor completeness.

Terms

Atom
A primitive typed architectural fact used as a generator in pure AAT.
AtomKind
The family coordinate of an atom, such as existence, relation, capability, state, effect, authority, contract, semantic, or runtime interaction.
Axis
The reading coordinate of an atom, such as static, semantic, specification, runtime, boundary, dataflow, governance, evolution, or policy.
AtomAxiomSystem
The Lean root that supplies atoms and records their primitive, boundary-independent, law-independent reading.
RawAtomCandidate
An observation-side candidate with evidence; not atom truth by itself.
ObservedAtom
An observation of an already-existing atom.
AtomObservationGap
A missing or unvalidated observation; not an absence theorem.
Projection
A coarser reading that forgets some atom coordinates while preserving selected ones.
