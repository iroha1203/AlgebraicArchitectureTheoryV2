Architecture Signature

An architecture signature reads an architecture across selected diagnostic axes. It keeps required zero-law axes, measured diagnostic axes, unavailable axes, and non-conclusions separate, so a report can say exactly what has been shown.

Research question

How can the condition of an architecture be summarized across many concerns without hiding uncertainty or reducing it to one score?

AAT answers by treating each diagnostic coordinate as a typed reading. The coordinate names an axis, a value when one is available, the evidence boundary, the claim level, and the non-conclusions. The signature is therefore not a verdict by itself. It is the structured surface on which lawfulness, obstruction absence, diagnostics, runtime readings, and empirical readings can be compared without confusing their status.

Signature axis

A signature axis is a diagnostic coordinate selected from a law, invariant, obstruction family, metric, or representation. It is not the same thing as the primitive axis field of an Atom. An Atom axis classifies an indivisible architectural fact: structural, behavioral, semantic, operational, organizational, or another primitive dimension. A signature axis is a later report coordinate built from configurations of such atoms.

Definition 1.1. A signature axis is a selected coordinate for reading an architecture object through a law, invariant, obstruction family, measurement rule, or analytic representation.

Proposition 1.2. Atom axes are source-level primitive classifications; signature axes are object-level diagnostic readings. A signature axis may depend on many atoms and many atom families.

Proof idea. A dependency-cycle axis is not a single dependency atom. It is computed from a relation family, a selected finite universe, and a reachability rule. Likewise, a substitution axis reads component, relation, contract, semantic, and observation atoms together.

Signature function

The mathematical shape of a signature is a partial vector of coordinates. For a selected architecture object \(A\), a signature reads each selected axis into a value or an unavailable status. The values may be Booleans, natural-number counts, ordered risks, proof-backed zero claims, runtime measurements, or empirical measurements, depending on the axis.

Definition 2.1. An ArchitectureSignature is a family of typed coordinates indexed by selected signature axes. Each coordinate is read with its value, measurement status, claim level, evidence universe, proof support when present, and non-conclusions.

The finite Lean v0 schema uses natural-number risk coordinates: hasCycle, sccMaxSize, maxDepth, fanoutRisk, boundaryViolationCount, and abstractionViolationCount. Signature v1 embeds v0 and adds core axes such as sccExcessSize, maxFanout, and reachableConeSize, plus optional axes for weighted SCC risk, projection soundness, LSP, nilpotency, runtime propagation, relation complexity, and empirical change cost.

Required zero axes

Some axes participate in the current required zero-law bridge. Those axes are not chosen because they are the only interesting diagnostics. They are chosen because the current theorem package connects their available-and-zero readings to the selected architecture lawfulness predicate.

Proposition 3.1. In the current Signature-integrated lawfulness bridge, lawfulness is equivalent to all selected required signature axes being available and zero.

Proof. Lean defines ArchitectureLawful as the conjunction of walk acyclicity, projection soundness, LSP compatibility, boundary policy soundness, and abstraction policy soundness. It proves architectureLawful_iff_requiredSignatureAxesZero by composing the exactness theorem for each required axis.

Non-conclusion 3.2. Runtime propagation, matrix nilpotency, relation complexity, empirical change cost, and other diagnostic axes are not required zero-law axes merely because they appear in a signature.

Measurement status

A signature must preserve the difference between measured zero and missing evidence. The optional axis value none can mean that a policy-dependent or empirical coordinate was not measured. It is not a zero theorem and it is not absence of obstruction.

Proposition 4.1. A missing signature axis value is not available-and-zero evidence.

Proof. Lean proves not_axisAvailableAndZero_of_axisValue_none. The strong predicate requires an available value equal to 0. A missing value cannot satisfy that predicate.

Measured zero
The selected axis was measured and the value is zero under the recorded evidence boundary.
Measured nonzero
The selected axis was measured and carries a nonzero witness, count, risk value, or obstruction reading.
Unmeasured
The axis belongs to the selected vocabulary, but the evidence needed to read it is absent.
Out of scope
The selected law package or report intentionally does not claim that axis.

AATCore bridge

The signature bridge is downstream of the Atom-generated AAT core. It consumes a ZeroCurvaturePackage and an ArchitectureLawModel. It derives selected signature lawfulness from pure AAT zero curvature; it does not define AAT, create atoms, or turn validation artifacts into Lean theorem discharge.

Proposition 5.1. The bridge derives required signature axes zero from pure AAT zero curvature through architecture lawfulness.

Proof. AATCoreSignatureLawfulnessBridge.architectureLawful applies the package's lawfulWithinMoleculeConfiguration. AATCoreSignatureLawfulnessBridge.requiredSignatureAxesZero then applies architectureLawful_iff_requiredSignatureAxesZero.

Proposition 5.2. The signature bridge preserves Atom priority: analysis output does not create primitive architectural facts.

Proof. The bridge carries archSigDoesNotCreateAtomsEvidence and exposes it through archsig_does_not_create_atoms. It also exposes archsig_does_not_define_aat and unknown_rejected_unmeasured_separated.

ArchSig boundary

ArchSig is the analysis layer that reads a source-grounded ArchMap together with one selected interpretation profile. The profile selects laws, witness rules, signature axes, coverage requirements, exactness assumptions, and measurement profiles. ArchSig can compute bounded analysis packets and human-readable summaries. It is not the AAT core and not a Lean proof engine.

Definition 6.1. An ArchSig signature reading is a bounded, policy-relative analysis result produced from supplied observation evidence and a selected interpretation profile.

Non-conclusion 6.2. ArchSig validation does not prove extractor completeness, global architecture truth, semantic correctness, universal quality, zero curvature, merge safety, or Lean theorem discharge.

This boundary is what makes signatures useful in practice. A report can place a nonzero obstruction axis, an unmeasured runtime axis, a proved static axis, and an empirical cost axis next to one another while still preserving the distinct claim level of each coordinate.

Worked example

Consider a payment capture architecture with services Checkout, PaymentGateway, Ledger, and ReceiptEmail. The object is represented by component, relation, effect, contract, semantic, and runtime-interaction atoms. A selected law package asks whether payment capture is acyclic, respects dependency boundaries, preserves substitution through the payment interface, and avoids replaying the charge effect.

The signature does not say "payment architecture score: 83." It says that the static cycle axis is zero, the boundary axis has one measured witness, the substitution and projection axes are zero under the selected evidence, the runtime propagation axis was not measured, and the semantic charge-once axis has a nonzero defect. Those readings support different actions: remove the forbidden boundary edge, add idempotency or a transaction guard, and measure runtime retry propagation before making runtime flatness claims.

Counterexamples and non-conclusions

Counterexample 8.1. All measured required axes can be zero while a non-required runtime axis remains unmeasured. The signature supports the selected static lawfulness claim, not runtime safety.

Counterexample 8.2. A clean ArchSig packet can be valid as a packet and still fail to discharge a theorem if the selected Lean preconditions are not supplied.

Counterexample 8.3. A dependency graph may have no cycle while a semantic diagram does not commute. The static signature axis and semantic signature axis are different coordinates.

Counterexample 8.4. A single aggregate quality score can improve while a required axis becomes nonzero. AAT therefore keeps the axis-local witness instead of replacing the signature by a scalar.

Formal reading

The main Lean files for this chapter are Formal/Arch/Signature/Signature.lean, Formal/Arch/Signature/SignatureLawfulness.lean, and Formal/Arch/Signature/AATCoreBridge.lean. They define finite signature schemas, axis values, required-axis selection, lawfulness equivalences, and the bridge from Atom-AAT zero curvature to signature lawfulness.

Formal status 9.1. The finite Signature v0 and v1 schemas, selected required axes, optional-axis availability predicates, and finite graph metrics are defined in Lean.

Formal status 9.2. The selected required-axis bridge is proved: architecture lawfulness is equivalent to selected required signature axes being available and zero.

Formal status 9.3. The AATCore bridge is defined and proves that signature lawfulness is downstream of Atom-AAT zero curvature, while ArchSig analysis does not define AAT or create atoms.

Chapter summary

An architecture signature is a multi-axis diagnostic object. It does not replace atoms, laws, obstruction circuits, or theorem packages. It collects their readings into a coordinate system that preserves axis, value, evidence boundary, claim level, and non-conclusion.

The current required zero-law bridge is intentionally narrow: selected static lawfulness is tied to selected available-and-zero axes. Other axes may be diagnostically important, but they remain runtime, semantic, matrix, tooling, or empirical readings until a bridge theorem or measurement boundary says otherwise.

Terms

Signature axis
A selected diagnostic coordinate over an architecture object.
Required zero axis
An axis whose available-and-zero reading participates in the selected lawfulness theorem package.
Available-and-zero
The strong zero predicate requiring an available value equal to zero.
Unmeasured axis
An axis for which the selected report has no usable evidence; it is not measured zero.
AATCore bridge
The downstream bridge from Atom-AAT zero curvature to signature lawfulness.
ArchSig boundary
The analysis-layer boundary for reading supplied ArchMap evidence through a selected interpretation profile.
