Part I. Atoms, Objects, and Laws

Molecules and Architecture Objects

Atoms become architectural only when they are arranged. A molecule is a finite configuration of already-existing atoms, and an architecture object is the structured object read from such configurations through selected maps, quantities, and evidence boundaries.

Research question

What makes a collection of architectural facts into a coherent object that can be analyzed and changed?

AAT answers this in two steps. First, atoms are assembled into finite configurations called molecules. Second, selected structure maps and quantities read those configurations as architecture objects. This is where the theory moves from individual facts to the architectural objects that later laws evaluate.

From atoms to configurations

An atom is a primitive fact, but a software architecture is not a bag of unrelated facts. The first construction after Atom is an atom configuration: a selected atom family together with the relations, identifications, and restrictions that make those atoms readable as one structure.

The relation component is not limited to dependency edges. It may record reference, realization, state update, emission, substitution, contract attachment, semantic interpretation, or runtime interaction. The identification component records the selected abstractions under which two presentations are read as the same architectural material.

Definition 1.1. An atom configuration is a selected atom family equipped with the relation and identification data needed to read it as architecture, not merely as a list of facts.

Proposition 1.2. Configuration is the first structure after Atom. It does not create atoms; it arranges already-existing atoms so later molecules, objects, laws, and signatures have something structured to read.

Proof idea. Atom existence is supplied by the atom axiom system. A configuration selects atoms and adds relation or identification data among them. Selection and arrangement can change the architectural reading, but they do not change the primitive status of the atoms being arranged.

Molecules

A molecule is a finite atom configuration that can be handled as one architectural unit. It is the first place where a concern, feature slice, boundary, interaction, state transition, or contract surface becomes a finite object of study.

Definition 2.1. A molecule is a finite configuration of atoms from a fixed AtomAxiomSystem. It is not an observation artifact, a role name, a design pattern, or a new atom family.

Proposition 2.2. Every atom contained in a pure AAT molecule remains a primitive atom of the root atom axiom system.

Proof. Lean defines AAT.Molecule.Contains M atom as membership in the molecule's atom predicate. The theorem AAT.Molecule.atom_primitive returns system.Primitive atom for any contained atom by calling system.primitive atom. Membership in a molecule therefore preserves atom origin rather than replacing it.

Proposition 2.3. Molecule inclusion supports local reasoning. If every atom in molecule M is also in molecule N, then M is a subconfiguration of N for the atom surface being read.

Proof. Lean spells molecule inclusion as AAT.MoleculeSubset M N, a predicate saying that every atom selected by M is selected by N. Theorems MoleculeSubset.refl and MoleculeSubset.trans provide the reflexive and transitive structure needed to reason about nested molecule surfaces.

Roles and patterns

A role or design pattern is not a primitive atom. It is an interpretation placed on a molecule. The same finite atom surface may be read as a repository, adapter, saga participant, event handler, policy boundary, projection, or integration point depending on which role vocabulary is selected.

Proposition 3.1. Role and pattern interpretation does not create atoms. It names a reading of a molecule.

Proof. Lean provides RoleInterpretation.toMolecule and PatternInterpretation.toMolecule, which forget the interpretation and return the underlying molecule. Theorems RoleInterpretation.does_not_create_atoms and PatternInterpretation.does_not_create_atoms expose the root system's non-generation boundary.

Non-conclusion 3.2. Naming a molecule "repository", "adapter", "bounded context", or "saga" does not prove that it satisfies the laws usually associated with that role. Lawfulness is evaluated later by selected design laws.

Architecture objects

An architecture object is generated from an atom configuration when selected structure maps and selected quantities are added. The structure maps may read the configuration as a graph, category, algebra, diagram, state-transition system, semantic surface, or runtime interaction model. The selected quantities may include invariants, measures, signature axes, curvature values, or witness requirements.

Definition 4.1. An architecture object is a structured reading of an atom configuration. It is not the atom family alone, and it is not the selected presentation alone; it is the atom-generated configuration together with the maps and quantities selected for the claim.

Proposition 4.2. Architecture objects have atom origin. AAT's objects do not appear independently of atoms; they are generated from atom families through configurations and selected structure.

Proof idea. The mathematical theory states the generated-object principle as F => A: an atom family F generates a configuration C, and C together with selected structure maps and quantities is read as A. The current Lean refoundation records the root side of this principle through AAT.PureTheory and AAT.AATCore: selected molecules in the theory contain only primitive atoms from the root system.

Proposition 4.3. Object equivalence is structure-relative. Two presentations may share the same atom family while differing as architecture objects because they choose different structure maps or selected quantities.

Proof idea. A dependency graph forgets path counts; a matrix representation keeps adjacency powers; a semantic diagram keeps commutativity conditions; a state-transition reading keeps operation order. These are different structure maps on related atom surfaces. Object equivalence must therefore specify which maps and quantities are preserved.

Selected presentation

A selected presentation is the bounded object-presentation layer: the chosen carrier, policy, observation context, and measurement universe through which an atom-generated object is read.

Definition 5.1. A selected presentation is the bounded reading context for an architecture object. It says which carrier, policies, observations, semantic diagrams, and measurements are in scope for a claim.

Proposition 5.2. Object claims are presentation-indexed claims. Changing the selected carrier, policy, observation context, or measurement universe changes the proposition being evaluated.

Proof. In the compatibility surface inherited from Foundations, AATJudgement P phi is defined as phi P, and aatJudgement_iff records that equivalence. For proof-carrying cores, ArchitectureCore.selectedPresentationJudgement_iff applies the same bridge to X.selectedPresentation. Thus the claim is licensed at the selected presentation, not at an unbounded repository.

Finite universe, coverage, and exactness

A finite universe is a measurement scope. It records which components, pairs, edges, diagrams, or witnesses have been selected for a bounded theorem package. It is what lets a finite computation or selected observation become a precise claim about the object being read.

Definition 6.1. Coverage says that the selected evidence contains the components or witnesses required by the claim. Exactness says that the measured value is the intended in-scope relation, witness, or axis reading.

Proposition 6.2. A finite universe supports selected completeness, not ambient extractor completeness. Complete extraction for an ambient relation is an explicit extra premise.

Proof. The bounded presentation surface proves that an ArchitectureCore supplies selected relation and graph completeness for its own selected presentation. It does not conclude arbitrary ambient repository completeness. Lean's CompleteExtractionCounterexample.selectedRestriction_same_but_completeExtraction_differs constructs two ambient relations with the same selected restriction while only one satisfies complete extraction.

Non-conclusion 6.3. ComponentUniverse is not an extractor correctness certificate. A zero result inside a selected universe is a selected zero reading, not proof that no out-of-scope component, edge, runtime interaction, semantic diagram, or effect exists.

Worked example

Take the checkout atom surface from the Atom chapter. A molecule can select the payment concern rather than the entire checkout application.

This molecule can be read through several object structures. A static graph reading uses the component and call relation. An effect reading uses capture and event atoms. An authority reading uses the permission surface. A semantic reading uses the "charges customer once" atom. A runtime reading uses retry and timeout facts.

  • As a molecule The surface is a finite selected configuration of root atoms.
  • As a role The same molecule may be interpreted as a payment adapter or external gateway boundary.
  • As an architecture object Selected maps turn the molecule into graph, semantic, effect, or runtime structure.
  • As a presentation Coverage and exactness say which selected evidence supports the claim.

Counterexamples and non-conclusions

Counterexample 8.1. A selected presentation may contain component A while the ambient repository also contains component C. The empty ambient relation and the relation containing only C -> A have the same selected restriction through the presentation that sees only A, but they differ on ambient complete extraction.

Counterexample 8.2. A molecule interpreted as "repository" may contain a state atom and a persistence capability but lack the contract atoms needed for a selected repository law. The role interpretation alone does not prove the pattern law.

Counterexample 8.3. Two architecture objects may be generated from overlapping atom families but choose different structure maps. A graph object may forget path count; a matrix object may keep adjacency power; a semantic diagram may keep commutativity. They should not be identified unless the selected equivalence preserves the maps and quantities in question.

Non-conclusion 8.4. Molecule membership, role interpretation, selected presentation, and finite universe coverage do not by themselves prove lawfulness, obstruction absence, zero curvature, or global extraction completeness.

Formal reading

The current Lean refoundation represents the pure object layer through AAT.Molecule, molecule inclusion, role/pattern interpretation, and AAT.PureTheory / AAT.AATCore. The bounded presentation surface is the selected-presentation and finite-universe reading of architecture objects, not the root of AAT.

Formal status 9.1. Molecule, inclusion, proper submolecule, role interpretation, pattern interpretation, pure theory, and AATCore are defined in Lean. Accessors proving that selected molecule atoms remain primitive and that pure AAT remains observation-independent are proved.

Formal status 9.2. The generated-object schema A = (C, S, Q) is the mathematical theory reading. The current Lean surface has the atom-rooted molecule/core API and the bounded presentation API; the full object equivalence calculus remains a later formalization target.

Chapter summary

The architecture object chapter moves AAT from primitive facts to structured objects. Atoms are arranged into configurations; finite configurations become molecules; roles and patterns interpret molecules; selected maps and quantities read those configurations as architecture objects.

The central boundary is now clear. Atom is the root. Selected presentation is the bounded reading context for an object. Finite universe is the measurement scope for a claim. None of these later layers should be mistaken for extractor completeness or global lawfulness.

Terms

Molecule
A finite configuration of already-existing atoms.
MoleculeSubset
Inclusion between molecule atom surfaces.
RoleInterpretation
A role reading attached to a molecule, not a primitive atom kind.
PatternInterpretation
A design pattern reading attached to a molecule configuration.
ArchitectureObject
An atom-generated configuration read with selected structure maps and selected quantities.
Selected presentation
The bounded carrier, policy, observation, and measurement context in which a claim is evaluated.
Coverage
The premise that selected evidence contains the witnesses required by the claim.
Exactness
The premise that measured evidence can be read as the intended in-scope relation or quantity.
