Reading template

A canonical example is a small, inspectable model that makes a reading discipline visible. Each example in this chapter says which atoms exist, which molecule is being tested, which law marks it bad, which circuit is minimal, which signature axis was measured, and where the theorem package stops.

Definition 1.1. A canonical AAT example is a finite presentation whose object, law, witness universe, signature axes, theorem package, and non-conclusions can all be named.

Principle 1.2. AAT examples are read axis by axis. A static witness licenses a static conclusion; a semantic witness licenses a semantic conclusion; a runtime witness licenses a runtime conclusion. The signature may place them side by side while keeping their proof boundaries separate.

Atom to circuit

The smallest worked example starts with one primitive API atom, forms a singleton molecule around it, chooses a law that marks exactly that molecule as bad, and proves that the molecule is a minimal obstruction circuit for that law.

Example 2.1. A singleton API molecule can be law-relative minimal bad even when every atom inside it is primitive and the law creates no atoms.

Proof. apiMolecule_atom_primitive reads the contained atom through the root atom axiom system. singletonBadLaw_does_not_create_atoms records law non-generation. singletonBadLaw_obstruction proves that the selected molecule is bad and that no proper submolecule is bad.

Counterexample 2.2. Minimal bad is law-relative. The atom remains primitive, while badness belongs to the chosen law and molecule surface.

Coupon extension

The coupon extension is the main multi-axis example. It adds a coupon feature to a checkout architecture. The static axis checks whether feature interaction goes through the declared interface. The semantic axis checks whether selected price-calculation paths produce the same observation. The runtime axis may remain unmeasured.

Example 4.1. In the bad coupon snapshot, the selected static hidden-interaction axis is measured nonzero and bridges to an actual selected static extension witness.

Proof. bad_staticHiddenInteraction_bridge combines bad.staticHiddenInteraction.SupportsSelectedAnalyticObstruction with StaticExtensionWitnessExists for the bad extension and declared interface.

Example 4.2. In the repaired coupon snapshot, the selected static hidden-interaction axis is measured zero and the selected static witness is absent.

Proof. repaired_staticHiddenInteraction_bridge pairs measured-zero evidence with absence of the selected StaticExtensionWitnessExists witness. This is a static conclusion, not a semantic conclusion.

Counterexample 4.3. Static repair can succeed while semantic curvature remains positive. repairedWithMeasuredSemanticCurvature_roundingOrderValuation_positive records the selected rounding-order residual after static repair.

Non-conclusion 4.4. Runtime exposure and semantic curvature values of none are unmeasured. Zero-reflecting claims require available-zero evidence.

Static flat, semantic obstruction

The repaired coupon graph supplies the canonical separation between static and semantic flatness. The static split can be restored, the graph can be walk-acyclic, and the static flatness package can hold, while a selected semantic diagram still fails to commute.

Theorem 5.1. Static flatness within the selected finite model and semantic flatness are separate law-package conclusions.

Proof. canonical_staticFlatWithin proves the selected static package for the repaired coupon graph. selectedCouponDiscount_not_commutes proves the selected semantic diagram fails. Therefore staticFlat_with_semanticObstruction holds: static flatness is present and semantic flatness is absent.

Counterexample 5.2. A dependency graph can be clean while the architecture is not flat under a semantic law package. This is not a caveat to static analysis; it is why AAT separates law axes.

Repair transfer

Repair examples are read as transfer, not as unconditional improvement. Removing a cycle or hidden dependency may lower a selected static measure while creating or exposing a semantic, runtime, or effect obstruction on another axis.

Principle 6.1. A repair operation licenses the measure decrease named by its selected obstruction universe. All-axis monotonicity requires an additional theorem package.

Counterexample 6.2. A static repair can transform a visible dependency problem into a semantic contract problem. The architecture has not become worse or better in a scalar sense; its signature support moved.

SOLID counterexamples

AAT can read design principles as law packages, but the examples prevent slogan-level inference. A design principle is not a magic predicate. It becomes mathematical only after projection, observation, lawfulness, and obstruction conditions are named.

Principle-style reading AAT interpretation Counterexample
Single Responsibility Selected reasons for change, state transitions, effects, and contracts should not be confused. A module with multiple effects becomes obstructed when the selected law exposes entangled change, state, effect, or contract support.
Open / Closed Extension needs a feature-extension theorem package. A new feature can preserve static split while breaking a semantic diagram.
Liskov Substitution Substitution requires selected observation and contract preservation. Subtype shape needs selected effect and semantic preservation evidence.
Dependency Inversion Projection, representative stability, and abstraction graph laws must be checked. DIP/LSP compatibility can hold while an abstract-layer cycle blocks decomposition.

Theorem 7.1. In the strong abstract-cycle example, DIP compatibility, exact projection plus representative stability, and LSP-style observation factoring can hold while the graph remains not decomposable.

Proof. StrongAbstractCycleComponent.lspCompatible, projectionExact, representativeStable, dipCompatible, and strongDIPCompatible establish the compatibility side. not_decomposable uses the abstract cycle orderPort -> paymentPort -> orderPort to refute a strict layering.

Counterexample 7.2. "Depends on abstractions" and "decomposable" name different law packages. The abstraction layer itself may contain the obstruction.

Reading rules

The examples enforce a small set of reading rules that should be used throughout the rest of AAT.

When you see Read it as Do not read it as
some 0 on a selected axis Measured zero for that axis. Unconditional architecture flatness.
none on an optional axis Unmeasured. Zero evidence.
Static split repaired Selected static witness removed. Semantic or runtime safety.
Design-pattern compatibility Compatibility under the named law package. Global decomposability or flatness.

Principle 8.1. AAT examples are successful when the reader can name the exact conclusion scope carried by the theorem package.

Formal reading

The Lean surface for this chapter is concentrated in canonical examples, static/semantic counterexamples, SOLID-style counterexamples, and the Chapter 11 analytic snapshot package.

Formal status 9.1. The atom/molecule/law/circuit example, coupon analytic snapshot, static-flat semantic obstruction counterexample, and SOLID-style dependency inversion counterexamples are defined in Lean.

Formal status 9.2. The key claims are proved for their selected premises: law non-generation, static hidden-interaction bridge, repaired static witness absence, unmeasured-axis non-certification, static-flat but semantic-obstructed separation, and DIP-compatible but non-decomposable examples.

Chapter summary

The examples make the theory operational. They show how to move from atoms to molecules, from molecules to law-relative circuits, from feature extension to multi-axis signatures, and from design slogans to theorem-shaped law packages.

The counterexamples are the main teaching device. They keep static flatness, semantic flatness, repair improvement, optional-axis measurement, and design-pattern compatibility in their own theorem packages.

Terms

Canonical example
A finite presentation with named object, law, witness universe, signature axes, theorem package, and non-conclusions.
Static / semantic separation
The fact that selected static flatness can hold while selected semantic flatness fails.
Repair transfer
Movement of obstruction support from one selected axis to another under a repair operation.
SOLID counterexample
A finite model showing that principle-style compatibility is weaker than global decomposability or flatness.
