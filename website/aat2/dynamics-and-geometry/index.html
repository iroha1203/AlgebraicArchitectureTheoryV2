Part II. Flatness, Calculus, and Geometry

Dynamics and Geometry

AAT does not treat architecture as a frozen diagram. It treats architecture change as finite paths, compares paths by selected homotopies, reads non-commuting squares as curvature, and keeps trajectory claims local to the observations that were actually measured.

Research question

When do different paths of architecture change lead to the same result, and when does their mismatch reveal a structural problem?

Static structure answers only one part of architecture. Real systems move: a feature is introduced, a policy is tightened, a contract changes, an obstruction is repaired, a runtime topology drifts. AAT gives these moves a finite path calculus. The calculus asks which paths have the same endpoints, which observations are preserved along them, which path pairs can be filled by allowed homotopies, and which path differences are selected evidence of non-fillability.

Architecture path

An architecture path is an endpoint-indexed finite sequence of primitive architecture steps. The type remembers both the source and the target. This matters because a path is not merely a list of edits; it is a composable proof object whose intermediate states line up.

Definition 1.1. An ArchitecturePath from \(X\) to \(Y\) is a finite composable chain of architecture steps whose type-level endpoints are \(X\) and \(Y\).

Proposition 1.2. Architecture paths form a finite concatenation calculus: appending with an empty path is neutral, append is associative, and path length is additive under append.

Proof. Lean proves append_nil, nil_append, append_assoc, and length_append by induction on the endpoint-indexed path. No semantic claim is needed for these structural facts.

Theorem 1.3. If every primitive step in a finite path preserves a selected invariant, then the whole path preserves that invariant.

Proof idea. pathPreservesInvariant performs induction over the path. The first step transports the invariant from the source to the next state; the induction hypothesis transports it through the remaining path.

Signature trajectory

A signature trajectory observes each architecture state visited by a finite evolution. If the path has \(n\) transitions, the trajectory has \(n + 1\) signature samples: source, intermediate states, and target. The delta sequence has \(n\) per-step deltas.

Definition 2.1. A SignatureTrajectory is the finite list of selected signature observations induced by an architecture evolution path.

Proposition 2.2. The trajectory length is one more than the path length, while the delta sequence length is exactly the path length.

Proof. signatureTrajectory_length and signatureDeltaSequence_length follow by structural recursion on the finite evolution path.

Theorem 2.3. Under an explicit additive signature-delta law, the sum of per-step selected deltas telescopes to the endpoint selected delta.

Proof idea. netSignatureDelta_telescopes requires AdditiveSignatureDeltaLaw. The theorem is about the selected additive delta, not about unmeasured axes, cost, incidents, review quality, or future behavior.

Theorem 2.4. If the initial observation is inside a selected safe region and every step preserves that region, then every observation in the trajectory remains inside the region.

Proof. trajectory_preserves_safeRegion again uses path induction. The theorem is deliberately path-local: it does not say that CI, review, or policy automatically supplies preservation evidence.

Path homotopy

Two architecture paths with the same endpoints may represent the same selected change. AAT does not identify them by wishful equivalence. It introduces a generated homotopy relation whose generators say which local rewrites are allowed.

Generator Meaning Claim boundary
Reflexivity, symmetry, transitivity Basic equivalence closure. Gives a relation, not semantic equality by itself.
Independent square swap Two adjacent steps commute under a selected independence witness. Only for the square whose independence evidence is supplied.
Same external contract Replace a step by another step with the same selected outside behavior. Contract equality is selected, not all semantic equivalence.
Repair fill Use a selected repair filling between two endpoint-matched paths. Only the repair relation admitted by the package is used.
Context closure Use the same homotopy under prefix or suffix context. Preservation must still be proved for the selected observation.

Definition 3.1. A PathHomotopy is the generated relation between endpoint-matched paths built from selected local generators and their congruence rules.

Theorem 3.2. If every homotopy generator preserves a selected path observation, then homotopic paths have equal selected observations.

Proof idea. PathHomotopy.observation_eq performs induction over the generated homotopy. The assumptions handle independent squares, same-contract replacement, repair filling, and context closure. Without those assumptions, homotopy is only a syntactic relation.

Diagram filling

A diagram in this chapter is a pair of endpoint-matched paths: one path says "do \(p\), then \(q\)"; another says "do \(r\), then \(s\)." A filler is selected evidence that these paths are homotopic under the allowed generators.

Definition 4.1. A DiagramFiller is selected path-homotopy evidence for a pair of endpoint-matched architecture paths.

Theorem 4.2. A filled diagram preserves selected observations, provided the selected homotopy generators preserve those observations.

Proof. diagramFiller_observation_eq delegates to the observation invariance theorem for PathHomotopy.

Theorem 4.3. If a selected observation distinguishes the left and right paths, then the proposed diagram filler is refuted.

Proof idea. observationDifference_refutesDiagramFiller argues by contradiction: a filler would imply selected observation equality, contradicting the observed difference. observationDifference_nonFillabilityWitnessFor packages the same difference as a selected non-fillability witness.

Non-conclusion 4.4. A selected non-fillability witness is not a proof that no architecture change could ever connect the endpoints. It refutes the filler relation, witness universe, and observation boundary named by the package.

Monodromy and continuation

Monodromy is the path-sensitive residue left after continuing an observation around a measured square or loop. In an architecture setting, it is the difference between "make this change by route A" and "make the endpoint-compatible change by route B" under a selected continuation function.

Definition 5.1. A selected monodromy defect is a finite measured defect attached to one selected axis and one measured path square.

Theorem 5.2. A nonzero selected-axis defect refutes selected path homotopy whenever the homotopy API proves that homotopy preserves the same observation.

Proof. nonzero_refutes_selectedPathHomotopy combines AxisDefect.observationDiff_of_nonzero with PathHomotopy.observation_eq.

Theorem 5.3. If a finite weighted aggregate of selected defects is zero, then every positive-weight entry in that measured family has local zero defect and equal selected observations.

Proof idea. localZero_of_weightedAggregate_zero and observationEq_of_weightedAggregate_zero use nonnegative natural weights. Zero aggregate cannot hide a positive-weight nonzero defect, but it says nothing about unmeasured squares or zero-weight entries.

Theorem 5.4. Under a supplied finite measured filling, nonzero boundary holonomy forces nonzero measured local-curvature mass.

Proof. localCurvatureAggregate_nonzero_of_boundaryHolonomy_nonzero uses the bounded Stokes-style inequality stored in the measured filling. exists_nonzero_localCurvature_of_boundaryHolonomy_nonzero then extracts a positive-weight nonzero local curvature cell.

Non-conclusion 5.5. An unfilled loop is an architectural hole, not an automatic violation. unfilledLoop_not_violationConclusion records that missing filler evidence must not be promoted into a violation conclusion.

Repair trajectory

A repair plan is not just a final target state. It is a path through intermediate architecture states. A selected measure may decrease, a selected obstruction may be cleared, and yet the trajectory may temporarily leave another safe region. Dynamics makes those intermediate states first-class.

Proposition 6.1. A finite evolution path preserves selected flatness when every transition in the path preserves the selected flatness predicate.

Proof. evolutionPathPreservesFlatness is the evolution-specific bridge to pathPreservesInvariant.

Proposition 6.2. A finite accepted evolution preserves a selected signature invariant when every accepted primitive transition is known to preserve that invariant.

Proof idea. acceptedEvolution_preserves_selectedInvariant uses a DampingControlSchema. The controller may represent review, CI, type checking, policy, or another gate, but the theorem is only about steps explicitly classified as accepted.

Worked example

Consider a payment architecture that adds coupon support. Two endpoint-compatible paths are available.

On a static dependency axis, both paths may look equivalent: coupon logic enters through the declared payment interface and no forbidden internal dependency is introduced. On a semantic axis, the paths may differ because discount-before-tax and tax-before-discount can produce different rounded amounts. The endpoint type is the same; the selected observation is not.

AAT reads this as geometry. If the selected path homotopy includes only static independence generators, it cannot justify the semantic swap. A nonzero receipt or ledger defect refutes the proposed filler for that selected square. A later repair may add a rounding-normalization step, but that repair is another path with its own trajectory and proof obligations.

Counterexamples and non-conclusions

Counterexample 8.1. Same endpoints do not imply same path. Two endpoint-matched paths require a selected homotopy or a selected filler before their observations may be identified.

Counterexample 8.2. Endpoint delta zero and safe endpoints do not imply path safety. Lean's endpointSafe_and_zeroDelta_but_not_pathSafe visits an unsafe intermediate signature even though the endpoint delta is zero.

Counterexample 8.3. Axis-wise safety does not imply global safety when the global predicate contains cross-axis relations.

Counterexample 8.4. An unfilled loop is not a violation conclusion. It records missing filler evidence, which may be operationally serious, but the theorem package refuses to turn absence of evidence into a semantic violation.

Counterexample 8.5. A zero aggregate over a measured family says nothing about unmeasured loops, unselected axes, zero-weight entries, extractor completeness, empirical calibration, or future behavior.

Formal reading

The Lean surface for this chapter is concentrated in evolution paths, signature dynamics, diagram fillers, selected monodromy measurement, and homotopy / holonomy guardrails.

Formal status 9.1. Finite architecture paths, append laws, path length laws, selected invariant preservation along paths, architecture evolutions, signature trajectories, additive selected deltas, safe-region preservation, path homotopy, diagram filling, selected non-fillability, selected-axis monodromy, and bounded holonomy / Stokes guardrails are defined in Lean.

Formal status 9.2. The key theorems are proved for their bounded premises: homotopy-invariant observation equality, diagram-filler observation equality, observation-difference refutation, aggregate-zero local equality, nonzero selected defect non-fillability, selected delta telescoping, safe-region preservation, and finite counterexamples to endpoint-only and axis-wise overreadings.

Chapter summary

Dynamics turns AAT from a static algebra of architecture objects into a geometry of architecture change. Paths compose, trajectories record what is observed along the way, homotopies say when two changes can be identified, fillers certify commuting diagrams, and selected defects measure where a square refuses to commute.

The discipline is again local. A filler is relative to selected generators. A monodromy reading is relative to selected axes. A safe trajectory requires per-step preservation. Non-fillability is a bounded witness, not a universal impossibility theorem.

Terms

ArchitecturePath
An endpoint-indexed finite sequence of primitive architecture steps.
SignatureTrajectory
The selected observations of every state visited by a finite architecture evolution.
PathHomotopy
A generated relation between endpoint-matched paths using selected local rewrite generators.
DiagramFiller
Selected homotopy evidence that two endpoint-matched paths fill a diagram.
Selected monodromy defect
A finite measured path-sensitive defect on one selected axis and one measured square.
Repair trajectory
A repair plan read as a finite path whose intermediate states and selected observations remain visible.
